## Music catalog (epic #115, Phase 0 — issue #120)

`music_catalog.py` (the store) + `music_index.py` (the read-only indexer) are the foundation every later music phase reads. **`music.db` lives on pearl**, next to its writer — `weather_archive.py:22-29`'s rule, unchanged: `merle.db` sits on bluejay where daemon-down is the steady state, and music must play when bluejay is asleep. It is **never on the NAS**: SQLite's locking is unreliable on SMB/CIFS, which is a corruption risk rather than a performance note. Config is `MERLE_MUSIC_DB` / `MERLE_MUSIC_ROOT` (`get(X,"").strip() or DEFAULT`); any future MCC-side reader takes an **absolute path with no default** (`mcc/app/weather/history/route.ts:20-25` explains why). Timestamps are **unix epoch seconds**, siding with `weather.db` over `merle.db`'s ISO-8601: ratings and history get ranged far more than read by a human.

**The audio files are an immutable input — we never write tags back to them.** Tags are read once at index time and every derived field lands in the catalog, which makes it rebuildable and disposable and lets metadata iterate against a 612.7 GB library without risking it. The share is mounted read-only (`//hummingbird/music on /mnt/music type cifs (ro,...)`), so the principle is enforced by the mount rather than by discipline.

**Identity is a hash of the audio stream, not of the file, and that was measured rather than argued.** A path is a location — files get renamed. But hashing whole file *bytes* has a subtler failure: this library is iTunes-managed (the share carries `.itl` and 2,241 `.itc2` files), so tag rewrites are live, not hypothetical. Retagging one m4a and one mp3 changed the whole-file hash both times and left the audio-stream hash **byte-identical** — a whole-file hash would mint a new identity on every tag edit and orphan that track's ratings and history, the exact failure the content hash exists to prevent. *(Rejected: whole file bytes — trivial, and wrong for this library.)* Spans are located with stdlib parsing, no dependency: `mdat` for m4a/mp4, ID3v2-syncsafe-to-ID3v1/APEv2 for mp3, post-metadata-blocks for flac, the `data` chunk for wav — probed at 32/32 across the real formats. FLAC's STREAMINFO carries an **MD5 of the decoded audio** (present on 8/8 samples), which survives even a re-compression, so it wins where non-zero. The hash is **blake2b-128** (stdlib): content-addressing for identity has no adversary, so SHA-256 buys nothing, and at GB/s the hash is free against a wire that delivers ~61 MB/s. A file whose span won't parse is **not dropped** — it lands with `needs_attention` set (Phase 1's bucket), a queryable number rather than a mystery.

**`tracks` is keyed by hash; locations live in `track_files`, one-to-many.** A 26k-track library rips the same recording twice (an album and a greatest-hits), and the real 400-file smoke pass found **369 tracks in 400 files** — 31 duplicates on the first try. Collapsing them is correct: rating a track rates the recording, not whichever copy played. Folding path into `tracks` would make a duplicate either a spurious second identity or a row whose path flip-flops every pass. A rename is therefore a re-hash that re-links to the same id, leaving a stale row that **`--prune` reclaims** — never automatically, because the indexer cannot distinguish "the files moved" from "the share isn't mounted", so `prune_is_safe()` refuses below a 50% floor. `forget_paths()` only ever drops locations: a track we can't find is not a track that never existed.

**The analysis columns are populated since #136** — `bpm`, `replaygain_db`, and `dynamic_range_db` went from <9% (tags) to 92.3% (measured), and `needs_attention` holds its first real entries: 15 files that hash fine but won't decode. See the audio-analysis section below.

**Both feedback tables are live**: `play_history` since #129 (a row per play, the moment anything sounds) and `ratings` since #135 (the daemon's `POST /rate`). Explicit and implicit feedback now accumulate together, which is what Phase 3 reads as rules and Phase 4 as evidence. Implicit feedback cannot be backfilled or bought, which is why the table predates its writer and why Phase 2a shipped Denon-first rather than waiting for the browser path. The tables are also what makes **schema evolution** non-deferrable: `CREATE TABLE IF NOT EXISTS` does not alter an existing file, and these tables accumulate for months before Phase 3/4 read them — exactly when a schema change lands on irreplaceable data. Hence `PRAGMA user_version` + an ordered, append-only `MIGRATIONS` list, built before it's needed. **The catalog is rebuildable from the NAS; ratings and play-history are not** — they are to music what `weather.db` is to the station.

**The walk skips the share's `#recycle` bin** (`EXCLUDED_DIRS`, issue #129). The first pass indexed it — 3,096 locations of *deleted* tracks, one shelf-render away from the GUI once real rows arrived — and the fix is two-part by design: the exclusion stops new bin entries at the walk, and a `--prune` pass reclaimed the ones already cataloged (23,494 locations remain; the pass saw 88% of known locations, comfortably over `prune_is_safe()`'s floor). Locations only, as always — any track that also exists outside the bin kept its identity, ratings, and history.

**Measured, replacing epic #115's estimates:** the library is **612.7 GB / 26,590 audio files** (m4a 62.3% · mp3 32.1% · flac 4.2% · wav 1.4%), not ~500–600 GB / ~27k. The indexer runs at **~61 MB/s hashing, ~56 MB/s with tag reads → a ~3.0 h first pass**; a raw sequential read of large files clocks 99.8 MB/s, so the real walk runs at ~60% of best-case streaming — per-file open/seek overhead across 26k mostly-12 MB files, not the hash and not the chunk size (1/4/8 MB read chunks landed within 1%, since the mount's `rsize=65536` governs). The **second pass is seconds**: the hash cache is keyed on `(path, size, mtime)` and re-reads nothing. Both fields are load-bearing — the measured retag left file **size unchanged** and moved only mtime, so a size-only cache would skip that file forever. Two consequences: **`vers=2.0` costs nothing** (the wire is the limit, so there's no throughput to win by revisiting it — #115's open question, closed), and **pearl reads at wire speed**, so D2's transcode cache stays on pearl and hummingbird's escalation triggers remain unmet.

**The pass runs on pearl, not bluejay, despite bluejay's faster CPU** — because it is wire-limited, and that was measured: pearl reads the share at 99.8 MB/s to bluejay's 90.2, both at gigabit saturation, while blake2b hashes at GB/s. pearl also already has the mount and already owns `music.db`, so it writes rows directly with no export/import hop. *(Rejected: bluejay + JSONL + `INSERT OR IGNORE`. That is the right shape for **Phase 1**, where beat tracking is genuinely CPU-bound and bluejay is several times pearl — but for Phase 0 it buys a slower pass and an extra moving part.)*

## Music audio analysis (epic #115 Phase 1a — issue #136)

`jukebox/music_analyze.py` (bluejay, offline) → JSONL → `jukebox/music_import.py` (pearl). BPM, ReplayGain, and dynamic range for every track, computed from the audio because the tags don't carry them (all three were under 9%). **Coverage went 0 → 23,440 of 25,382 tracks (92.3%)** in a single 1:32:43 pass; the missing 7.6% is not a shortfall but the 1,927 tracks whose only copy lived in the `#recycle` bin and lost its location to #129's prune — **of tracks that have audio, coverage is 99.94%**.

- **One ffmpeg invocation does both jobs.** The `ebur128` filter measures loudness *and* passes the audio through, so the same decode feeds beat tracking on stdout (`-f f32le -`; nothing is ever handed a path to write). Reading twice — once for loudness, once for beats — would double the wire cost for nothing. *(Rejected: `librosa.load()` for the decode, which hands non-WAV/FLAC to audioread, which shells out to ffmpeg anyway — the same read with less control and no loudness.)* ffmpeg is not optional: the library is 62% ALAC, which libsndfile cannot decode.
- **The epic's "CPU-bound" premise is false at scale, and this is the number that matters.** Measured on bluejay's 16-core i9-12900KF: 8 workers → 2.64 tr/s, **16 → 3.18**, 24 → 3.29 (+3% for 50% more workers). The full run sustained **3.99 tr/s ≈ 92 MB/s**, which is bluejay's measured wire ceiling (90.2 MB/s, Phase 0) — so past ~16 workers **the wire is the limit again**, not the CPU, and more workers buy contention. D2's "put the CPU work on the fast CPU" reasoning still holds; its stated reason stops applying at the knee.
- **The JSONL is the resume state, deliberately — not the catalog.** A bluejay-side snapshot won't know a track is analyzed until the import runs on pearl, possibly days later. Records are flushed per track under a lock, so a `taskkill /F` mid-run loses only the ~16 in flight (demonstrated: killed at 875 tracks, restarted, resumed at exactly 891 with the 16 in-flight redone). A torn final line is skipped, not fatal.
- **`ex.map` + `with ThreadPoolExecutor` is a trap for an interruptible pass**: `map` submits every future up front and the context manager's `shutdown(wait=True)` drains the whole queue, so Ctrl-C raises in the main thread and then waits out the entire run. `shutdown(cancel_futures=True)` plus a `stop` flag that no-ops dequeued work is what makes an interrupt interrupt.
- **The importer UPDATEs, it does not INSERT.** The epic said `INSERT OR IGNORE`, but Phase 0 put these on `tracks` as nullable columns — same idempotency, different verb (proved on the real 25,382-row table: a second import leaves the fingerprint byte-identical). It updates *unconditionally* so a future better tempo algorithm can land without a migration; the *analyzer* is what skips finished work (`--force` re-does it). An id the catalog doesn't know is counted and skipped, never inserted — `tracks` rows are the indexer's to create.
- **Two silent failures the catalog could not have caught.** Phase 0 reported **0 `needs_attention`** because hashing only *locates* the audio span; it never decodes. This pass decodes every byte and found **15 genuinely broken files** (0.064%) — 13 that ffmpeg refuses outright (three consecutive Genesis tracks on *The Lamb Lies Down on Broadway*, a bad rip) and 2 truncated stubs whose tags lie about their length (The Prodigy's "Claustrophobic Sting" claims 434 s in 0.3 MB, where ALAC would need ~40 MB). A valid container with corrupt guts hashes perfectly and looks healthy.
- **Path remapping, not a second catalog.** `music.db` stores locations as *pearl* sees them (`/mnt/music/...`); the analyzer runs elsewhere, so `remap_path()` swaps the prefix for `MERLE_MUSIC_ROOT` (`\\hummingbird\music`) rather than teaching the store about a second machine's filesystem. A path outside the catalog root returns None and is skipped loudly — better than building a nonsense path and blaming the NAS.
- **Measured, for whoever tunes this next**: BPM median **117.5** (p10 89, p90 152; only 4 tracks over 220, so no double-time epidemic), ReplayGain median **−7.7 dB**, dynamic range median **5.4 LU**. **The loudness war is visible in our own library**: average gain by release decade runs −2.5 dB (pre-1990) → −5.2 (90s) → −7.6 (2000s) → **−8.4 (2010+)**, i.e. masters got ~6 dB hotter across four decades.
- **Nothing downstream depends on the analyzer existing** — verified with the analyzer stopped: pearl serves `/state` and the app on :3001 unchanged. After the backfill bluejay can be powered off forever.

## Music catalog normalization (issues #163 genre, #152 artist)

One pass (`jukebox/music_genre.py`), one rules file, two normalized columns. The library's **178 feral iTunes genre strings** (`Electronica\Trip-Hop` in three separator spellings, `FLAC` as a genre, `Industrial Jungle Pussy Punk`) normalize into a **22-tag canonical vocabulary** in `tracks.genre_norm`, and its **26 case-split artist identities** (`Panic! at the Disco` / `Panic! At the Disco`, measured across 748 identities) collapse into a canonical display name in `tracks.artist_norm`. Raw `tracks.genre` / `artist` / `album_artist` are never touched — they are provenance, the indexer keeps writing them verbatim, and every decision stays reversible because the input is still there. **Every surface reads only the normalized columns** (pills, browse filter, playlist affinity; artist grouping, ids, art keys), which is the enforcement: a weird new tag has no UI existence until deliberately mapped, and a new casing cannot split an artist.

- **The artist stage (#152) is the #163 materialization move applied to identity**: fold each track's artist identity (`COALESCE(album_artist, artist)`, trimmed + `str.lower()` — deliberately *not* `casefold()`, because simple lowercase is what a TS twin could implement identically and casefold's `ß`→`ss` mappings couldn't be), tally casing frequency, write the winner. Tie-break, deterministic (the catalog has a live 12–12 tie, Run DMC): most tracks → a casing seen in `album_artist` beats one only seen in track `artist` → lexicographic. The rejected alternative was fold-key resolution at read time — three lowercase implementations (Python/TS/SQL) that must never drift, one of which (SQLite `lower()`) is ASCII-only and wrong by construction; materializing means **no folding exists downstream at all**. `artist_display` in the rules file is the taste veto (`GWAR`, not the library's majority `Gwar`) and the pin against a future import flipping a winner; an override must fold to the same identity — it picks a casing, never renames.
- **Identity flows through one SQL fragment, twinned**: `music_catalog.ALBUM_ARTIST_SQL` / `db.ts`'s `albumArtistCol` put `artist_norm` first with the raw COALESCE as fallback, so a pre-pass snapshot degrades to case-split grouping — never to a crash or broken art. `ALBUM_KEY_SQL` (art keys) and `albumIdOf`/`artistIdOf` (URLs) therefore mint canonical, case-collapsed identities; art rows written under a dead casing simply re-enter `albums_missing_art` / `artists_missing_art` and the next art-pass run re-extracts them (self-healing by worklist — run it once after the first normalize on a populated catalog). Track rows keep the raw per-track performer credit; `Track.albumArtist` carries the identity for surfaces (search grouping) that need the canonical name off a track alone.
- **The stages feed forward**: genre's artist-majority inheritance and `artist_overrides` key on the folded identity, so a majority split across `Gwar`/`GWAR` tallies as one artist and override keys match case-insensitively.

- **The rules are data, not code** (`jukebox/genre_rules.yaml`, the owner requirement): `vocabulary` + `map` (canonical → raw strings; a raw maps once) + `artist_overrides` (judgment calls no string can express; also where the future curation of unresolved artists lands) + `clusters` (the playlist engine's genre families — a custom vocabulary needs custom families, so they travel together) + `tuning` (the 0.6 inherit threshold and the `triage` junk-drawer list). The repo file is the owner's live ruleset, so **rule changes are commits**; a second deployment (the metal-heavy friend library) points `MERLE_MUSIC_GENRE_RULES`/`--rules` at its own file, never at code. YAML over JSON specifically for comments — `FLAC: # that's Burial` is half the file's value. Validation fails loudly before any write: targets in vocabulary, no duplicate raws, no unknown sections, mapped-or-triaged-never-both. `test_music_genre.py` pins the shipped file as valid (the typo canary — a bad edit fails CI, not pearl mid-pass).
- **The pass** (`jukebox/music_genre.py`, the #153 reusability rule whole): syncs the file's map into the `genre_map` table one-directionally (`source='file'` rows upserted, stale ones pruned; `external` rows — the future bulk-metadata pass's alias lane — never touched by the sync), then recomputes every track's expected pair and writes only diffs. Same run twice = zero writes; post-ingestion run touches exactly the new rows; **a rules edit re-run IS the "reprocess under my new rules" job** — there is no separate backfill tool.
- **Precedence**: `owner` (hand-set, protected in the UPDATE's WHERE like the art store — no automated pass can clobber it) > `artist_overrides` > string map > `external` (a per-track backfill value yields only to an explicit rule) > **inheritance** (a track whose raw maps nowhere takes its artist's genre when ≥60% of the artist's mapped tracks agree; the artist key is the album-artist COALESCE so compilation guests don't fragment the majority) > honest NULL. Provenance lives in `genre_norm_source` (`mapped|inherited|external|owner`); both columns ride migrations v5/v6 (one ALTER per step — the focal_y lesson) and sit outside `TRACK_COLUMNS`, so the indexer's tag-refresh upsert can't wipe them.
- **The deferred set is named, not lost**: NULL-genre tracks plus the five `triage` strings (`Pop\Rock` is AllMusic's catch-all — Violent Femmes/Skinny Puppy/Fear Factory/Prince side by side — plus `Pop/Rock`/`Styles`/`Popular`/`Other`) stay `genre_norm` NULL for the bulk-metadata-import pass (bios + genre via MusicBrainz/Last.fm, routed through the same `genre_map` as `external` rows). Triage strings stay off the unmapped report; anything else unmapped prints at the end of the pass with counts — the fix is one rules line + a re-run.

## Music playback daemon (epic #115 Phase 2a — issue #129)

`jukebox/music_daemon.py`: a pearl-resident FastAPI service (**port 8090**, the `music-daemon` unit — `Servers/Pearl.md`) that streams catalog tracks over HTTP and drives the **Denon AVR-X4000** over UPnP/DLNA. The epic's spike inverted the design's build order — the Denon plays ALAC natively and untranscoded (verified, audible, parses the container's own duration), while Chrome refuses ALAC outright (`SRC_NOT_SUPPORTED`) — so 2a shipped the Denon first (starting the unback-fillable `play_history` clock), and the browser became its own phase with a transcode path (2b, next section).

- **DLNA means the renderer pulls.** `POST /play` hands the Denon a URL (`SetAVTransportURI` + `Play`); the Denon then fetches `GET /stream/{track_id}` itself. The daemon is two halves meeting at a URL: a Range-capable file server and a small SOAP client. **The Denon receives the file's bytes untouched** — nothing sits between `open("rb")` and the socket, which is the point of that output. `MERLE_MUSIC_STREAM_BASE` (required, no default) is the daemon's own LAN-visible URL, because guessing your own address is how you stream to a renderer that can't reach you.
- **Range is hand-rolled and non-negotiable** (`parse_range`, exhaustively tested): `http.server` doesn't implement it, no Range means no seeking, and the Denon really does open with a probe GET then a Range request into the audio. Suffix ranges are real (trailing-index reads); multi-range is answered with the whole file (RFC 7233 allows ignoring Range; no real client sends it).
- **Capability is a table we own** (`OUTPUT_FORMATS`), not a negotiation — the Denon answers `GetProtocolInfo` with HTTP 500, so asking is not an option. The answer to "does this need transcoding?" is the catalog's `format` column, read before the stream starts. Nothing sniffs bytes at runtime.
- **Discovery, never hardcoding**: SSDP `M-SEARCH` at startup (retried on first `/play` if the renderer was off), control URLs read from each device's description XML — the epic's warning about UUID-embedded paths, and this AVR has already changed address once between probes.
- **One history row per play, adjudicated by the watcher thread**: `completed` when the transport stops with the last position observed *while playing* near the end (the Denon zeroes `RelTime` the instant it stops — reading position after the stop would call every completion a skip-at-zero), `skipped` when the listener moved on (`/play` over a playing track, `/stop` mid-song, or a stop at the AVR's own remote). Unknown duration or position is always a skip: implicit feedback errs toward not crediting a listen that didn't happen.
- **Presence on the bus for free**: `bus.EventPublisher("music", status_topic=MUSIC_STATUS_TOPIC)` — retained `online`/`offline`, Last Will, own `music/` namespace (a jukebox is not a narrator; the Field Journal's wildcard must not light up). `systemctl stop` flips it within seconds with zero signal-handling code — verified with a real SIGTERM.
- **Two flags are load-bearing on the uvicorn line**: `--no-access-log` (the GUI polls `/state`; issue #125's flood, preempted) and `--timeout-graceful-shutdown 3` — the **renderer** holds `/stream` open for the whole song, so an unbounded graceful drain turns SIGTERM into a zombie that outlives its replacement. Observed live on pearl mid-playback; the vision daemon's flag, same lesson, different client.
- **Hardware findings, recorded so nobody re-discovers them**: the Denon refuses AVTransport `Seek` (HTTP 500) on external streams — the GUI's restart-track gesture re-issues play instead of seeking to 0, and a scrub snaps back to the truth on the next poll. Real browser seek arrives with the 2b browser output, where Range actually serves it.
- **The daemon is the catalog's only writer** (issue #135). `POST /rate` `{track_id, value}` records a thumb; the music app's own handle is `readOnly: true` by construction, so a rating comes here rather than minting a second writer. Validation lives in `music_catalog.rate()` next to `RATING_VALUES` — one copy of the legal set, or the two drift. Two traps it closes: **`bool` subclasses `int`**, so a JSON `true` satisfies `in RATING_VALUES` and would file itself as a thumbs-up (rejected explicitly); and **`value: 0` is legal to send but illegal to store** — it's the control's third click, so it dispatches to `unrate()` and deletes the row. An unrated track is the *absence* of a row, never a stored zero: Phase 3's filters read what exists, and a 0 would be a third state every one of them would have to know to ignore. The timestamp is the daemon's, never the client's — a browser clock is a guess, and this is the table we can't rebuild.
- The id allowlist (`music_catalog.valid_track_id`, `[A-Za-z0-9_:-]+`) runs before any catalog query — `frame_archiver.py`'s traversal-guard genre, wire ids never meet the filesystem. The share stays `ro`; every open is `"rb"`.

## Music browser output (epic #115 Phase 2b — issue #149)

The third output is the listener's own browser: the music app's hidden `<audio>` element streams `GET /stream/{id}?output=browser` straight from the daemon (not through the Next proxy — piping audio through the app server would re-buffer it and break Range). The daemon drives nothing; the tab is the transport.

**The lossless rule is absolute: no lossy transcode anywhere in the music path, ever** (owner requirement, recorded in #149's do-not-change list). The library was ripped lossless on purpose, so the browser's per-output policy (`music_cache.needs_flac`, decided from catalog columns, never by sniffing) is:

| Catalog says | Browser gets | Why |
|---|---|---|
| mp3 / flac / wav | **raw bytes**, the Denon's exact serving code | decoded natively |
| m4a/mp4 + `codec='aac'` | **raw bytes** | already lossy; repacking to FLAC inflates without gaining a bit |
| m4a/mp4 + `codec='alac'`, `NULL`, or anything else | **FLAC repack** | Chrome refuses ALAC; ALAC→FLAC is lossless-to-lossless (identical PCM, proven by `ffmpeg -f md5` decoded-PCM equality). Unknowns take this path because its worst case is wasted bytes, never lost ones |

- **The `codec` column** exists because `format` is just the extension and `.m4a` holds both ALAC (the majority) and lossy purchase-AAC. The indexer probes the mp4 `stsd` sample-entry fourcc (`music_index.mp4_codec` — a real atom walk, not a byte scan; `b"alac"` appears in magic cookies and cover art) at index time for new files; `python3 -m jukebox.music_index --codecs` backfills rows indexed before the column existed (header reads, minutes over SMB). It rode in as the migration seam's **first real migration** (v2) — which surfaced the seam's latent fresh-file trap: `connect()` now stamps a brand-new file straight to `SCHEMA_VERSION` instead of replaying ALTERs against a schema that already mirrors them, and `migrate()` tolerates "duplicate column" for the DDL-landed-stamp-didn't crash window. Only m4a/mp4 rows carry a codec; elsewhere the extension *is* the codec and a second column would be drift risk.
- **The cache** (`jukebox/music_cache.py`) lives at `MERLE_MUSIC_CACHE` (`/srv/media-cache/music` on pearl — music's subdirectory of the **shared** media-cache LV; tenants get sibling dirs and their own retention, see `Servers/Pearl.md`). **The cache key is the track id** — the audio-stream content hash — so staleness is structurally impossible: a re-rip is a new hash and a new entry; a retag changes nothing; the old entry becomes an orphan and is swept. There is no invalidation logic to get wrong.
- **Cached files, not live pipes**, because a pipe can't serve Range and no Range means no seeking (the epic's hard UX floor). The cold click takes the middle path: ffmpeg writes `<id>.flac.part`, the response **tails the growing file** (`iter_growing` — playback starts within the first frames, chunked, Range ignored per RFC 7233), and success renames to `<id>.flac` — rename-then-signal, so presence of the final name *is* the completeness contract. Degradation on that first cold serve: ffmpeg's placeholder STREAMINFO means no duration/scrub until the file completes (seconds); every later serve — and the MD5 acceptance check — reads the patched file. `POST /precache {track_ids}` warms the next queue tracks (gated to 2 concurrent so warming never starves a cold click, which starts immediately), so queue advances are cache hits and the transcode delay is unobservable in playlist listening.
- **Eviction is a pure plan** (`plan_evictions`, tested): stale `.part`s (crashed transcodes), then **orphans before any live entry** (hash no longer in the catalog — the re-rip leftover), then oldest-mtime LRU down to `MERLE_MUSIC_CACHE_CAP_GB` (default 40). mtime is the recency clock on purpose — the daemon `touch()`es on every serve because the mount is `noatime`. Swept after every transcode and at startup. A full or missing cache degrades (log, fall back), never fails playback.
- **Unset `MERLE_MUSIC_CACHE` is the kill switch** (the `MERLE_OLLAMA` convention): the daemon offers no browser output at all — `/state`'s picker row says `available: false` honestly rather than shipping an output that's broken for the ALAC 62%. Set-but-missing dies loudly at startup (the LV didn't mount).
- **Play-history parity**: a browser tab can't be polled, so the provider **reports** each session's end — `POST /report {track_id, position_s}` on `ended`, on skip-away, and on output switch. The daemon adjudicates with the *same* `outcome_for` the Denon watcher uses and stamps its own clock: one arbiter, two transports, rows land as `output='browser'`.
- **Client side** (`music/components/PlayerProvider.tsx`): one provider, two transports. Browser mode plays the hidden `<audio>` (position from `timeupdate`, advance from `ended` — the 2s daemon poll stays off; both transports share one `advanceAfterEnd` so repeat behavior can't drift), seeks for real, warms the next two queue tracks on every playhead move, and builds srcs from the daemon's `stream_base` (served by `/state`; server config never bakes into client bundles). Switching outputs mid-track **stops rather than migrates** — the transports don't share a position. The Denon path is byte-identical to 2a: `?output=` defaults to `denon`, so every URL the daemon ever handed a renderer means what it did before.
- **The chosen output is sticky per browser** (issue #169): the id is written to `localStorage` (`music.outputId`) on every real switch and read back in the provider's state initializer, falling back to `denon` when nothing is stored or storage throws. Per-browser rather than a server-side preference because "this browser" is itself a per-browser output — one remembered answer for all devices would be wrong the moment a second one opened the app. Nothing rendered depends on `outputId` (the picker mounts on click), so the initializer read is hydration-safe and there's no window where the provider believes it's on the Denon. The stored id is **not** validated against discovery — outputs are only enumerated when the picker opens, the hardcoded default never got that check either, and a stale id self-heals on the next pick.

## Music cover art (epic #115 GUI track — issue #153)

Real artwork on every surface, from the library's own files — no network fetch anywhere in the path.

- **Sources and coverage (measured)**: embedded art on ~83% of tracks, WMP-era `Folder.jpg` in 38% of album dirs, union ~90% of the 1,872 albums. The extractor (`jukebox/music_art.py`, runs on pearl, minutes) takes the **largest embedded picture** across an album's tracks, falls back to `Folder.jpg`/`Cover.jpg`, and writes nothing for the remainder — those keep the generated SVG and stay on the worklist for the next run.
- **Focal analysis** (issue #159): every stored cover gets `focal_y` — the vertical interest centroid (FIND_EDGES over a 64×64 grayscale downscale, 1px border cropped so the frame doesn't drag everything to center; clamped [0.2, 0.8], uniform covers land at exactly 0.5), computed **inline at extraction** since the image is already open, pytest-covered. Migration v3/v4 added the column to `album_art`/`artist_art` (two steps, one ALTER each — `migrate()`'s duplicate-column tolerance would strand the second ALTER of a combined script); `--focal` backfills pre-#159 rows from the `.large.webp` variants (measured: ~2,500 rows in under half a minute, local reads) and is a no-op forever after — a fresh install never runs it. Analysis is not a clobber: the focal-only setters skip the owner guard on purpose, since *where the interest sits* is orthogonal to *which image the owner chose*. The album page anchors its backdrop band with `object-position` from `Album.artFocalY` (rides the index entry through `getAlbum`; null = center = pre-focal behavior); square renders (grids, the anchor cover, the lightbox) stay dead-center.
- **The reusability rule** (owner requirement): every enrichment pass — index, `--codecs`, analysis, art, blurbs, bios — is worklist-driven, idempotent, and standalone. Art's worklist is "albums with no `album_art` row", so the future ingestion pipeline re-runs the same pass and touches exactly the albums it just added. Nothing assumes "full library".
- **The store** is `/srv/media-cache/music-art/` — a **sibling** tenant on the media-cache LV, deliberately not inside the FLAC cache's dir (its sweep deletes files it doesn't recognize). Content-addressed: `<blake2b128>.orig` (untouched bytes) + `.thumb.webp` (~160px) + `.large.webp` (~600px), generated at extraction — the serve path does zero image work. Lossy sizes are fine; the lossless rule covers audio, not pixels. All derived and rebuildable, no backup — **except owner-override images**, which are irreplaceable and live with the `music.db` backup story.
- **Provenance is structural** (`album_art` / `artist_art`, `source ∈ embedded|folder|derived|owner`): the upsert's `WHERE source != 'owner'` means no automated pass can ever clobber the listener's own pick — the rule lives in SQL, not caller discipline. The tables ride `SCHEMA` with no version bump (`CREATE TABLE IF NOT EXISTS` self-applies; the migration seam is for ALTERs only).
- **Artist images**: zero exist in the library, so v1 **promotes** the artist's most-rated album cover (summed thumbs, tie-break lowest album key — byte-stable across runs), recorded as `source='derived'`. The owner's own photo is a one-row `owner` update when the picker UI lands; external photo fetch is deferred to the Phase 1b bio-fetcher.
- **The album key is a cross-language contract**: `music_catalog.ALBUM_KEY_SQL` and `music/lib/db.ts` both derive `COALESCE'd artist + U+241F + COALESCE'd album` (with `artist_norm` leading the artist half since #152) — the exact string `albumIdOf` base64urls into URL ids. Paired fixture tests (`test_music_catalog.py` / `catalog-rows.test.ts`) pin identical fixtures on both sides.
- **Serving** (`/api/art/[hash]/[size]` in the music app): hash allowlisted (32 hex) before any path is built, plain file read from `MERLE_MUSIC_ART` (no default — unset means SVGs everywhere, the kill-switch convention), `Cache-Control: immutable, max-age=1y`. Content-addressed URLs change exactly when pixels change, so browsers cache every cover forever and a repeat page load makes zero art requests.
- **Rendering**: `art_hash` rides the existing row mappers onto `Track`/`Album`/`Artist` (optional field, so pre-art shapes — fixtures, the daemon's `/queue` payload — type-check unchanged), and `CoverArt` renders the `<img>` absolutely over the generated SVG, which remains the loading state (art fades in) and the fallback (missing art, 404s via `onError`). **The fade settles cached images via a mount-time ref** (`el.complete` check) — an immutable-cached image finishes before hydration attaches `onLoad`, and waiting on the event left loaded covers invisible on hard reload (issue #155). Zero layout shift by construction. `lib/db.ts` guards every art query on the tables existing, so a pre-art snapshot degrades to "no art, full library" — never to an empty catalog.
- **The album page anchors on its cover** (256px, 384px at sm+ — the `large` asset earning its size), and clicking it opens a hand-rolled **lightbox serving `/orig`** — the untouched original bytes' one surface. OutputPicker's backdrop idiom: click-away and Escape close; no dependency.

## Music artist bios (epic #115 Phase 1b — issue #170)

`jukebox/music_bio.py`: MusicBrainz for identity, Wikipedia for prose, Last.fm as fallback. Run on pearl by hand; ~720 canonical identities, ~2 requests each at 1 req/s, so roughly half an hour once.

- **Live coverage** (full pass on pearl, 2026-07-18): **520 of 719 identities (72%)** — Wikipedia 489, Last.fm 31, 15 resolved-but-articleless, 182 unresolved to the review queue, 2 rules-skipped, 0 errors, 44.5 minutes. The 182 is larger than this issue's "clears in one sitting" estimate, and most are systematic iTunes naming (`Ali, Azam` inverted, `10.000 Maniacs` with periods, `Afrojack - Roland Clark`) rather than genuine ambiguity — a normalization rule would clear more of them than 182 hand-pins would. MusicBrainz returned 52 rate-limit 503s at the 1.3s interval, every one recovered by the retry; the interval wants raising to ~1.5s.
- **A transient network failure is recorded as a miss.** `fetched_at` cannot distinguish "no article exists" from "the fetch failed", so an artist whose Wikipedia call blips after identity resolution is skipped by every subsequent normal run. `--retry-missing` is the escape hatch and is worth one pass after any full run.
- **The epic's premise was measured false and this issue corrected it.** The `comment` tag was assumed to hold Last.fm artist bios needing a merge/dedupe rule; it holds iTunes/Amazon *album* copy (see the album-descriptions section), genuine artist bios exist for ~3.4% of artists, and there is nothing to merge with. Fetched bios simply win.
- **IDENTITY RESOLUTION IS THE WHOLE PROBLEM, and the issue's own accept rule did not survive contact with the API.** Measured 2026-07-18: searching for the library's band "We" returns "We Are Scientists" at **score 100** — MusicBrainz's score is Lucene relevance, not confidence, so a fuzzy match ties a perfect one. "Exact-fold name + score threshold" would have published a New York indie band's bio on that page.
- **So the gate is corroboration**: an exact case-folded name match must also share an album title with the MusicBrainz artist's release groups. The library already knows what it holds, and the evidence is free — release groups ride the lookup the pass makes anyway for the Wikidata link. Where MusicBrainz lists *no* release groups, corroboration is impossible rather than failed and the pass falls back to name-and-score; refusing every sparsely-documented artist would empty the page for exactly the obscure bands a bio helps most. Two artists genuinely sharing a name is resolved by whichever one the shelf corroborates; two corroborating matches is a real ambiguity and goes to the queue.
- **Two corroboration bugs found by running it, not by tests.** (1) A lookup's `inc=release-groups` subquery is **capped at 25 and cannot be paginated** — "A Tribe Called Quest" and "Andrew Manze" both came back "none of its 25 release groups match" with the matching album outside the window. Exactly 25 now means *probably truncated*, and a failed corroboration at the cap triggers a paginated `browse` follow-up before the artist is condemned. (2) MusicBrainz house style is **typographic punctuation** where iTunes writes ASCII: the one ATCQ album here is `We Got It from Here... Thank You 4 Your Service` against their U+2026 ellipsis. `fold_title` folds curly quotes, dashes and ellipses to ASCII and drops a leading article — every step an encoding difference, never a loosening; the words still must agree exactly.
- **Nothing downstream of the MBID is guessed**: MusicBrainz → the Wikidata URL relation → the enwiki sitelink → the article's lead extract. Name→MBID is the only fuzzy step in the chain.
- **Wikipedia via the Action API's `prop=extracts`, deliberately not `rest_v1/page/summary`** — the REST path entered gradual deprecation in July 2026 with replacement routes unannounced until H2. `extracts` is the long-stable route and takes `maxlag`, which is what a non-interactive job is supposed to send. Both APIs require a real contact User-Agent; an anonymous agent gets the stricter bucket or a block.
- **The review queue is a report plus a rules file** (`jukebox/bio_rules.yaml`, `skip` / `pin` / `tuning`, `MERLE_MUSIC_BIO_RULES` to override). Unresolved artists print with their candidate MBIDs and disambiguation text; paste the right one into `pin`, re-run, and exactly those artists change — the genre pass's UNMAPPED loop. A pin skips resolution entirely: it is the owner overruling the resolver, so it is not re-corroborated. Real queue residents are iTunes naming variants (`10.000 Maniacs` with periods, `Ali, Azam` inverted, `Afrojack - Roland Clark`) and classical, where album titles genuinely disagree.
- **`fetched_at` marks ATTEMPTED, not succeeded**: an artist resolved with no article anywhere still gets a row, so re-runs skip known misses; `--retry-missing` is the deliberate way back, and it never reopens an `owner` row. `bio_src ∈ wikipedia|lastfm|owner` with the same `WHERE bio_src IS NOT 'owner'` upsert guard as the art and note tables. `mbid` and `bio_url` arrived as migrations v8/v9 (one ALTER each — the focal_y lesson).
- **Last.fm is gated on `MERLE_LASTFM_KEY`** (kill-switch convention: unset means a complete Wikipedia-only pass, not a degraded one) and is queried **by MBID, never by name** — re-introducing a name lookup at the fallback would re-introduce the ambiguity the resolver exists to avoid. Their prose arrives as HTML with a "Read more on Last.fm" licence tail; the tail is stripped *before* the tags, or its anchor becomes bare words that read like content.
- **UI.** The artist page's bio slot and Read-more have existed since the fixture era — filling the table lights them up. Both surfaces gained a `Source · Wikipedia` stamp (`SourceStamp`, shared across bios and album descriptions) linking `bio_url`; Wikipedia's CC BY-SA makes attribution part of using the prose properly. The album page gained an **About {artist}** panel below the tracklist (the TIDAL/Apple Music idiom), clamped to three lines with the artist name linking through — deliberately no second Read-more, since the full prose is one click away and an expand-in-place control would grow the page under the reader. It is **server-rendered** via `getArtistBio(name)` in the album page's server half, so the panel is present or absent on first paint: **no pop-in, no layout shift**, and an artist without a bio renders the page with exactly the sections it had before (verified: 2 sections, no panel, no horizontal overflow, phone and desktop). `artistBioFor` is a point read rather than a reuse of `getArtist` — hydrating an artist's whole discography to render one paragraph would be a query storm for text. `hasBioSrc` guards the new columns the way `hasFocal` guards `focal_y`: SELECTing a missing column throws, and `withDb` would turn that into an empty catalog.

## Music album descriptions (epic #115 — issue #171)

`jukebox/music_blurb.py`: the `comment` tag's store copy, ingested as album-level descriptions and rendered in an **About this album** panel on the album page.

- **Live coverage** (full pass on pearl, 2026-07-18): **383 of 1,872 albums (20.5%)** carry a description — against the #115 measurement's independently-derived 20.6% prose-blurb figure. 43 more were dropped as truncated past saving, 1,446 had no usable comment, 0 errors, 12 minutes. **313 of the 383 (82%) needed sentence-trimming**, which is how load-bearing that policy turned out to be.
- **The pass reads ~86 GB off the NAS** to extract text. mutagen parses containers rather than peeking at headers, so it saturates the link for its 12 minutes — worth knowing before scheduling it alongside anything else that wants the wire.
- **What the tag actually holds** (measured on #115, 2026-07-16, n≈1,333 albums): iTunes/Amazon *album* editorial copy, not the Last.fm artist bios the epic assumed — 61.7% nothing, **20.6% prose blurb**, 12.8% scrap (`USA`, `amrc`), 3.5% one-liners, 1.6% EAC/encoder rip junk. Genuine artist bios are ~3.4% of artists, which is why bios are #170's fetch problem and this is a separate album-level field. FLAC is 0% commented: coverage tracks *how a record was acquired*, not anything about the music.
- **Truncation is the whole reason this was hard.** 49% of prose blurbs are exactly 255 chars, cut mid-word, and the lost text is not recoverable from the files. Owner decision (2026-07-18): **trim back to the last complete sentence and show what survives; drop the blurb entirely when nothing complete remains** — a page ending "an outstanding level of musicianship and a so" is worse than no panel. A survivor under the 80-char prose floor is dropped too: one opening clause stripped of what it was setting up reads as a bug.
- **Detection is by how the text ends, not how long it is** — and that correction came from running the pass, not from the measurement. A length-based detector (`255 chars AND no terminal punctuation`, straight off #115) passed real blurbs cut mid-word at 105, 112 and 114 chars straight through raw. The 255 wall is where truncation happens *most*, not where it happens. Text ending without terminal punctuation (closing quotes and brackets allowed after it) was cut, at any length; a 255-char blurb ending on a period was not, and trimming it would eat a real final sentence.
- **`album_notes` stores raw alongside description** (`album_key`, `description`, `raw`, `source`, `truncated`, `updated_at`). Keeping the untouched tag text means a future policy change — or a diff against a fetched source — is a re-run over one small table rather than another walk of 26k files. Rides `SCHEMA` with no version bump, the `album_art` precedent (`CREATE TABLE IF NOT EXISTS` self-applies; the migration seam is ALTERs only).
- **Provenance a third time** (`source ∈ comment-tag|external|owner`), same `WHERE source != 'owner'` upsert guard. `external` is reserved for a richer album-level source (a Wikipedia article via #170's MBIDs) that would supersede the store copy — album entity resolution is materially harder than artist-level, which is why it isn't simply part of #170.
- **Classification is pure and tested** (`classify_comment`), and **order is load-bearing**: rip junk is tested first because an EAC log passes every prose-length test beneath it. Markers match on word boundaries — a bare substring test eats "peace" via `eac` and "prologue" via `log`, both plausible in real album copy.
- **The pass is a thin loop over a per-album function.** `propose_for_album(paths)` takes one album's files and returns a proposal dict — always a dict, never `None`; its `status` is `ok` / `none` (no usable comment in the files) / `dropped` (there *was* prose, the truncation policy salvaged nothing whole). Those are different answers to "why does this album have no description", so the pass tallies them apart and the refresh button can say which happened. It touches no database; `main()` writes what it returns. That order is required, not stylistic: the planned **refresh-this-album/artist button** (owner requirement — find bad data, click, get suggested corrections) calls the same function in-process on one album and writes only on approval, so the button and the bulk pass cannot drift into different answers. `--album <key>` runs exactly that path from the CLI, and `album_paths()` is the single-album twin of the worklist query so both read the same files through the same key derivation.
- **Reading comments needs raw mutagen.** `music_index.read_tags()` uses `easy=True`, whose key maps exclude comments entirely — which is why the catalog never ingested any of this. `read_comments()` opens the raw tag object and branches per container (`©cmt` / FLAC `comment` / ID3 `COMM`), `embedded_pictures`'s shape. It returns **every** value, not the first: a file can carry a rip log *and* the store's blurb, and returning whichever frame came first would let the junk hide the prose. Ranking is `pick_comment`'s job — best bucket first, then longest (the least-truncated copy of the same text), then earliest for a stable tie-break. Values are **HTML-unescaped at read time** (`clean_comment`): the store copy carries entities verbatim (`Demos,&amp; Two Live Tracks`) and React escapes on render, so leaving them in prints a literal `&amp;` on the page. Unescaping here rather than at render means `raw` in the catalog is already the text a human would read.
- **iTunes' own bookkeeping is junk that reads as prose.** `iTunNORM`/`iTunSMPB` gain and gapless data are skipped by description prefix on ID3 — but on m4a they ride the `©cmt` atom itself with no frame name to filter on, arriving as ~89 chars of whitespace-separated hex that clears every prose test. Classified as junk by an anchored all-tokens-hex match (anchored so it can't swallow a description that merely opens with a hex-ish word).
- **Albums with no usable comment write no row**, so ~62% of the catalog re-reads its headers every run (minutes, not hours — header reads, the #153/#136 timings). Inherited from the art pass deliberately: the alternative is a "checked, found nothing" row whose staleness nobody would ever invalidate. `--dry-run` classifies and reports without writing.
- **UI**: an **About this album** panel below the About-artist panel (the order #171 specifies; album-first would arguably read better and is a one-line swap). Clamped to three lines with a Read more past 260 chars — the artist hero's threshold, and user-initiated so it does not violate the nothing-moves-on-its-own rule (verified: expanding moves nothing above it). Absent entirely for the ~80% with no description: no empty shell, no reserved gap. The `SourceStamp` component serves bios and descriptions alike — `Source · the album's tags` here, named for where the text actually came from, since claiming a publisher we cannot verify would be worse than being vague.
- **`albumKeyOf` is now exported from `catalog-rows.ts`** and is the single TS-side derivation of the raw album key. `albumNoteFor` needs the key itself rather than its base64 id, and a second `artist + "␟" + album` spelled out at a call site is exactly how the two language sides drift — `albumIdOf` builds on it too, with a test pinning that they agree.

## Music playlist engine (epic #115 Phase 3 — issue #139)

`jukebox/music_playlist.py`: *"play me stuff like this"* as pure arithmetic over the analysis axes — **no LLM anywhere in this phase**, the epic's bar verbatim (Phase 4 makes it smarter, not possible). The engine is pure end to end: candidates, clock, and RNG are injected, no SQL inside scoring, no `time.time()` inside ordering. The daemon exposes it as one endpoint, **`POST /queue` `{seed, n, exclude}` → an ordered list of track shapes** — it generates lists only, starts no playback, holds no state; the daemon stays one-track-at-a-time on the transport verbs and the GUI owns the queue it fetches.

- **Two measured facts shape the scoring** (prototyped live against the real catalog, 2026-07-16, epic #115 comments). (1) **BPM is lattice-quantized at ~4–5% resolution** — librosa reports tempo on the lattice `60·22050/(512·lag)`, so exact BPM ties across genres are the *normal* case, tempo distance under one bin (~0.03 in log2) is clamped to zero, and the GUI must never render "112.3" as precise. (2) **Because tempo ties constantly, the tiebreak carries the ranking and must know genre *families*** — the first draft's exact-tag bonus put Queensrÿche at #2 for a Stereo MC's seed. Affinity works on the head token plus a cluster table, never string equality and never as a hard filter. Since #163 the daemon feeds the engine `genre_norm AS genre` (canonical tags, NULL neutral) and injects the families from `genre_rules.yaml`'s `clusters` section the way candidates and RNG are already injected — the module's `GENRE_CLUSTERS` raw-tag set survives only as the no-rules fallback.
- **The recipe, with provenance** (weights as live-tuned; retuning is a constant change plus a test update): octave-folded log2 tempo distance ×22 (half/double time is the same groove; 10% off ≈ 3.0) · |ΔDR| ×0.55 · |Δgain| ×0.30 · |Δyear|/18 capped at 30 · genre family −2.5 same head / −1.8 same cluster. Era and loudness deliberately double-count a little (#136 measured the loudness war by decade — known, accepted). **The regression baseline is the Stereo MC's "Lost In Music" benchmark** on the epic: early-90s electronic/hip-hop nearly wall to wall.
- **The seed is a flexible intent** — `resolve_seed(seed, candidates) -> (target, exclude)`. A track seed is that track's features (and excludes itself); an artist seed is the artist's **median** centroid (one 287-BPM outlier must not drag it) plus modal genre head; `{"target": {...}}` passes a prebuilt feature point through — the mood/weather seam Phase 5 fills (`narrator.py`'s roster-of-one move: the shape exists now so later seeds plug in without a rewrite). An unknown artist is an empty queue, never an error.
- **Ratings are rules here, evidence in Phase 4.** Strong-down (−2) is a hard filter **in the daemon's candidate `WHERE` clause**, never the scorer — the ban is enforced by the query, not entrusted to arithmetic, and a test proves absence over many seeded runs. Down/up/strong-up are fixed score adjustments (unmeasured starting values; the table had ~0 rows at ship). An empty ratings table is indistinguishable from "no opinions yet" — the quality floor is the analysis axes alone. Skip-weighting from `outcome` is named and deferred to Phase 4.
- **Anti-repetition**: the candidate query excludes anything in `play_history` within `COOLDOWN_HOURS` (24; the query derives its cutoff from the engine's constant so the two can't drift, and the pure rule `too_soon()` is the same policy under an injected clock), no artist twice within `ARTIST_SPACING` (6) consecutive slots, and the caller's `exclude` list keeps a refill from repeating what the GUI's queue already holds. A library too small for the spacing runs dry honestly — a short queue, never a repeat.
- **Determinism with variety** (`lib/queue.ts`'s injected-entropy precedent): each slot samples from the top-K (40) still-eligible candidates weighted `exp(−Δscore/1.5)`, ranking ties broken on track id. Same inputs + same RNG seed → byte-identical queue (tested); production passes a fresh `random.Random()`.
- **Radio in the GUI** (`PlayerProvider.startRadio`): one affordance — Radio from a track row or the artist header — seeds the engine, replaces the client queue, plays. Continuous play is a **refill loop, not a server queue**: at ≤3 upcoming tracks the provider fetches the next window of 25 with everything it holds excluded. Responses come back in the `TrackRow` wire shape and feed through the same tested `trackFromRow` mapper as every other surface; an empty or failed fill leaves the current queue playing out. Any explicit `playTracks` exits radio mode.

## Music app (epic #115, GUI track — issue #116)

The music player UI is a **second top-level Next app, `music/`, a peer of `mcc/`** — epic #115's D1, decided in #116. #110 reserved it a launchpad tile as an independent app, MCC's shell lives inside the 4,142-line `Dashboard.tsx`, and "Merle Control Center" doesn't stretch to cover albums; the rejected alternative (a `/music` route inside MCC) would have started with a shell-extraction refactor that has nothing to do with music. Same stack and conventions as MCC (Next 16 App Router, Tailwind 4, pnpm, Vitest on `lib/` pure helpers, components untested by policy), with tokens **copied, not abstracted** — a shared package is premature until two apps demonstrably need the same change twice.

- **The data seam is real now** (issue #129; the fixture era's design paid off exactly as intended). **`lib/api.ts` is server-only and async**: per-request `node:sqlite` reads of `music.db` (the `mcc/app/weather/history/route.ts` pattern whole — `MERLE_MUSIC_DB` absolute with **no default**, read-only, unset-or-missing degrades to an empty catalog, never a 500), shaped by pure mappers in **`lib/catalog-rows.ts`** (Vitest), ordered by the *same* tested `browse.ts`/`shelf.ts`/`search.ts` helpers the fixtures used — the album/artist index (~2k rows) loads whole and sorts in JS rather than porting `sortKey`'s leading-article rules into SQL; only the visible window hydrates tracks. **Artist/album ids are base64url of the names** (an artist *is* a name in this catalog — the *canonical* name since #152, `artist_norm`'s case-collapsed identity; the pages build raw `/album/${id}` hrefs and this library really contains "AC/DC"); track ids are the indexer's content hashes, shared verbatim with the daemon's `/stream/{id}`. **The m4a→alac/aac split believes the catalog's `codec` column** (#149's stsd atom walk, wired through the seam in #157 — the same provenance the daemon's `needs_flac` policy trusts); the bitrate heuristic (>500 kbps — ALAC runs ~700–1100, store AAC ≤320) survives only as the fallback for null-codec rows, where its worst case is a mislabeled badge, never a playback decision. The raw container rides along as `Track.container` for the album header's format pill (`lib/format-badge.ts`, tested: `ALAC · M4A` where codec and wrapper differ, a single token where the extension is the codec, majority label with stable tie-break at album level). Client components can't import `api.ts` (node:sqlite fails their build, loudly): search debounces into `/api/search`, the `[id]` pages split into server wrappers + client views, the seed queue arrives as a layout prop, and the player talks through **`/api/player/[verb]`** — an allowlisted verb proxy to `MERLE_MUSIC_DAEMON` (`state` on GET; `play`/`pause`/`stop`/`seek`/`rate`/`queue` on POST), because the daemon's address is server config and a second origin would drag CORS into a two-box LAN app. `lib/fixtures.ts` stays on disk feeding the pure-helper tests, nothing else.
- **`PlayerProvider` plays for real.** The queue stays client-side on purpose (one-track-at-a-time is the daemon's whole contract; since #139 Phase 3's engine *generates* sequences and this provider consumes them — radio mode's refill loop, see the playlist-engine section) but time and truth come from pearl: a 2s poll of `/api/player/state` supplies the position, and "the track ended" is *observed* — the daemon's watcher adjudicated the history row and cleared its slate — then the queue advances and posts the next play. An unreachable daemon freezes the bar at the last known position rather than inventing progress. Restart-this-track re-issues play (the Denon refuses Seek — see the daemon section); a scrub snaps back to the truth within a poll.
- **Surfaces**: persistent bottom player bar (now-playing cluster, transport + seek, queue toggle / output picker / volume / quality badge), queue slide-over (history · playing-from · next-up, with per-track remove — user-driven edits are in scope; *agent*-driven queue mutation stays the epic's non-goal), sound-output popover (Phase 2's three confirmed targets verbatim), everything-search in the chrome (grouped artists/albums/tracks overlay), artist pages (hero, read-more bio, top tracks, discography), album pages (#157/#159: recognizable own-cover backdrop — lightly blurred, double-scrimmed so the bottom-left text zone stays legible while the top stays scrim-free as the art's showcase, and **focal-anchored**: `object-position` follows the extraction pass's interest centroid so the band features the cover's subject, not dead center; TIDAL-scale title with the artist as the clear second line, text bottom-aligned *on purpose* — the backdrop owns the space above; quality + format pills sharing one border weight; and the retired heavy-blur wash demoted to a top-anchored fade behind the tracklist, quieter than the hero ever ran it, focal-anchored the same way).
- **The front door scales by curation, not enumeration** (issue #118). Home is genre pills + three bounded shelves — recently played (**real `play_history`** since #129; the shelf is simply absent until listening exists), **Today's dig** (the rediscovery shelf: a date-seeded deterministic shuffle over everything *not* recently played — anti-recency is a hard filter, and the seed is computed in a `force-dynamic` **server component** so the client never runs Date-in-render), and recently added (**by newest file mtime** since #129 — the honest "added" proxy until the catalog has ingest timestamps). The genre pills render the **canonical 22-tag vocabulary** since #163 (`lib/db.ts` reads `genre_norm`; the feral-taxonomy era — `ELECTRONICA\DUBSTEP`, `FLAC`-as-a-genre — ended when the normalization pass landed). Each shelf is **exactly two rows at every width** (`.shelf-grid`: 7×2 desktop down to 3×2 phone, extras `display:none`). The clamp is CSS, not JS, for two reasons: measuring width in JS would desync the server and client renders (hydration), and a reflowing shelf shifts the page under a thumb. A horizontal scrollbar was the first draft and hid half the shelf behind a gesture. First-paint card count is bounded (~50) regardless of catalog size.
- **The full catalog lives at `/albums` and `/artists`** — the browse pages TIDAL refuses to build. Both **load a window at a time as you scroll**: the page server-renders the first 60 and `components/Browser.tsx` appends each next window from **`/api/albums` / `/api/artists`**. The client holds what's been scrolled to, never the catalog — at 27k tracks, shipping the library to the browser to slice it there is the exact failure these pages exist to prevent. The feed contract is **offset+limit in, `{items, total, nextOffset}` out** — `LIMIT ? OFFSET ?` wearing a URL, so Phase 0 swaps the fixture call for a query against `music.db` inside the same handler (`mcc/app/weather/history/route.ts` is the precedent) and nothing upstream changes. Three load-bearing details: **`nextOffset === null` is the only end signal** (inferring "done" from a short window breaks when the last window is exactly full); **there is a real Load-more button, not just a scroll sentinel** (auto-load is invisible to keyboard users, so the button is the honest control and the observer the convenience — both call one re-entrancy-guarded loader); and a fetch failure **degrades** (the list keeps what it has, the button offers Retry) rather than throwing.
- **Filter/sort/letter stay URL state**, all `<Link>`s, re-rendered on the server — only the scroll-append is client-side. Changing a filter must start a new list, not append to the old one, which is what the `key` on `<Browser>` enforces. `sortKey` drops a leading "The " (record-store filing, tested). The rail renders only letters present, and a letter jump sets **where the window starts** (`?letter=S`), not a page — with infinite scroll there are no pages, and an offset in the URL would rot on re-index. It hides under newest-sort, where a letter jump is meaningless. **A jumped list that reaches the end says "end of the list", never "that's all of them"** — the entries before the jump were never loaded and the footer must not claim otherwise. Genre is one-per-album at the UI seam — the dominant `genre_norm` across the album's tracks (canonical since #163; a pre-normalization snapshot degrades to Uncategorized-heavy, never to raw tags leaking back — `hasGenreNorm()` guards the column like `hasFocal()` does); an **artist** filter means "has any album in this genre", because artists span genres in a real library. Pure mechanics live in `lib/shelf.ts` + `lib/browse.ts` under Vitest.
- **The palette is "same station, different room."** Base tokens are MCC's verbatim; the species accents and `--rain` are deliberately **absent** (they name detections and measurements; nothing here does either). `--led` keeps its one job — green means LIVE — worn by the now-playing row and its equalizer glyph. The one new accent is **`--hires` `#e0b25c`** (OKLCH ~L 0.78 C 0.11 h 82): it names a *quality the catalog indexed* (>16-bit lossless) the way `--rain` names a measurement, colors only the quality badge, and the badge always carries its label in text so no meaning rides on hue. Tiering (`lib/quality.ts`): hi-res gold > 16-bit lossless quiet > lossy faint; an album wears its best track's badge.
- **Four-level rating** (`lib/rating.ts`): split thumbs; click sets ±1, same-thumb again escalates to ±2, third click clears, opposite thumb replaces. The transition table is exhaustively tested because a strong-down becomes Phase 3's hard `WHERE` clause — the control mints contract data. **It persists since #135**, and the shape is worth knowing: **the catalog's answer rides on every `Track`** (a scalar subquery in `TRACK_COLS`, so all three track queries get it including `topTracks`' `GROUP BY`, where a joined column would have to be grouped), and `PlayerProvider`'s map holds **only this session's edits** — `ratingFor(track)` is `edits[id] ?? track.rating`. Hence it takes the *track*, not an id: an id alone can't answer for an album-page row that was never in the queue, which is what a queue-only map got wrong. The write is optimistic like `startTrack`, and **rolls back only if the click it fired for is still the one showing** — the control is built to be clicked in a burst (set → escalate → clear), so a slow failed POST must not stomp a newer click.
- **Generated cover art** (`lib/cover.ts` + `CoverArt`): fixtures have no artwork, and gray squares would let layout bugs hide — each album hashes its id into two hues + a pattern (rings/beams/blocks/waves), deterministic so an album looks the same everywhere including its own blurred backdrop. Real art replaces the component's internals, not its callers.
- **No-layout-shift discipline** (the house #1 rule): the idle player bar renders the playing state's exact skeleton; toggle-button active dots and output-picker checks are present-but-`invisible` when off; rating controls are always mounted and only fade on hover; search results are an absolutely-positioned overlay.
- **Dev**: **port 3001, pinned in the `dev`/`start` scripts themselves** (`next dev -p 3001`) rather than only at the launcher — a bare `pnpm dev` in `music/` would otherwise default to 3000 and collide with MCC. The app owns its port however it's started. Dev env (`music/.env.local`, not in git): `MERLE_MUSIC_DB` pointing at a **backup-API snapshot** of pearl's catalog (never `cp` a live WAL file), `MERLE_MUSIC_DAEMON=http://192.168.1.64:8090` — browse reads the snapshot, play drives the real daemon, and history rows land on pearl's copy, so a stale dev snapshot only means a stale recently-played shelf. **Ratings make that split sharper (#135), and it's worth knowing before it fools you**: a thumb clicked in dev writes to *pearl's* catalog and is read back from the *snapshot*, so it renders correctly until you reload and then appears to vanish — which looks exactly like the bug #135 fixed. The data is on pearl; the snapshot is stale. Re-snapshot to see it, and don't debug the reload in dev without checking pearl's table first.
