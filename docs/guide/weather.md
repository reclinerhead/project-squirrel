### The weather post (`weather.py`)

Conditions at the seed pile (issue #25, re-instrumented in #51): a pearl-resident service (needs neither GPU nor camera, just the bus, the gateway, and internet) with **two sources and one truth**. The driveway's own station — an **Ecowitt GW2000B gateway + WH90 sensor array** on the LAN — is the system of truth for everything it can measure: polled every 60 s over its unauthenticated local HTTP JSON API (`/get_livedata_info`), it supplies temp, feels-like, dew point, VPD, humidity, wind speed/gust/max-daily-gust/direction, solar, UV, the piezo rain suite (raining-now bit, rate, event/day/week/month/year totals), barometric pressure (abs + rel, from the gateway's internal WH25), indoor temp/humidity, and the WH90's battery/voltage (radio signal comes from `/get_sensors_info`, fetched on the slower cadence). **OpenWeather's classic free APIs keep exactly two jobs the station can't do**: the 5-day/3-hour forecast (hourly) and the sky garnish — condition text, sunrise/sunset, and the station's **lat/lon** (issue #111: read from the response's `coord`, so a `MERLE_WEATHER_LOC` given as a bare zip still resolves to real degrees; the dashboard needs them to compute sunrise/sunset for any day but today, for the deep-pan night bands) — from a demoted 10-minute `/weather` call whose measured fields are discarded. Measured fields **never** fall back to OpenWeather: a gateway outage is a skipped poll and a chart gap, never a silent source switch. Publishes the retained `weather/*` topics; consumers are the dashboard's Weather Post panel + station view and the narrator's prompt context (issue #26, `weather/current` only).

- **Config** (env, the MERLE_MQTT conventions): `MERLE_ECOWITT` (gateway `host` or `host:port`, e.g. `192.168.1.210`) and `MERLE_OWM_KEY` are both **required, no default** — a service that can't name its station (or would poll 401s while looking healthy) fails at startup instead. `MERLE_WEATHER_LOC` takes `zip`, `zip,CC`, or `lat,lon` (default `49001,US` — the station's home turf; lat/lon is recognized because both halves parse as floats, which a country code never does). `MERLE_WEATHER_HISTORY` overrides the history-file path.
- **Gateway parsing** (`parse_station`): the GW2000B keys live data by opaque ids (`0x02` temp, `0x0E` rain rate, `"3"` feels-like…) and wraps every number in a unit-suffixed string (`"29.24 inHg"`, `"46%"`); `STATION_COMMON_IDS`/`STATION_RAIN_IDS` are the Rosetta stone (verified against the real device) and `station_num()` strips the units. A response where nothing parses is treated as a failed poll, not a report of Nones. The `weather/current` payload is a **superset of the pre-#51 shape** — every old field name survives with its old meaning, so the narrator and any open dashboard tab ride through a deploy; `ts` is our fetch time (the station has no observation clock).
- **Timestamps on the weather topics are unix epoch seconds**, not the ISO strings used elsewhere: the dashboard formats them in the viewer's locale and the narrator compares against `time.time()` for staleness — nobody wants to parse.
- **Forecast points carry precipitation** (issue #56, un-summed in #65): besides `pop` (probability 0..1), each `weather/forecast` point has `rain_rate_inhr` — the step's `rain["3h"]` mm converted to an **average in/hr over the 3-hour window ending at the point's `ts`**, sharing the observed piezo trail's unit — and `snow_3h_in`, the step's `snow["3h"]` mm as plain inches (its own strip charts it per step; no station instrument shares a scale with snow, the piezo is snow-blind). OWM omits both objects entirely on dry steps; the fields default to 0, not null: a step the forecast left dry is a real forecast of zero, unlike a station gap (which stays an honest hole in the observed trail).
- **The 48h rolling window** is the live state the service owns: polls run every 60 s but the trail keeps **5-minute resolution** (`should_record` gates appends; ~576 points of the 12 charted fields, tens of KB), pruned by age *measured from the newest point* (deterministic for tests), republished whole to `weather/history` on every append, and persisted to a JSON file on pearl so restarts don't blank the dashboard's observed trail. Old history files load fine — pre-#51 points simply lack the new fields, which chart as gaps that heal within 48h. Deliberately a flat file, not SQLite — bounded, rewritten whole, refetchable.
- **The seasonal archive** (`weather_archive.py`, issue #105) is the permanent record beside that window, and the two are not alternatives: every point that passes `should_record()` is appended to the window **and** recorded to a SQLite `observations` table that never prunes. The window stays the retained-bus payload (a fresh tab rehydrates its trail from the broker with no fetch — the contract that keeps weather off HTTP); the archive is a second consumer of the same tick, read over HTTP only when someone asks for the deep past. A failed archive write is logged and skipped — one missing point is a gap the chart already draws honestly, and never worth killing a service whose job is the bus. Startup backfills the window into the archive (idempotent on every restart, not just the first, so a fresh archive inherits up to 48 h of real readings). Config is `MERLE_WEATHER_DB` (default `weather.db` under `WorkingDirectory`), and an unopenable path fails at launch rather than publishing weather for a week while recording none of it. Four decisions worth the ink:
  - **On pearl, not in `merle.db`.** `merle.db` lives on bluejay next to the daemon, and daemon-down is the *steady state* for the 24/7 dashboard — archiving through the daemon's HTTP surface would punch holes in the record exactly when bluejay naps, which is most of the time. `weather.py` runs on pearl 24/7, so the rows live where the writer lives. This is `frame_archiver.py`'s reasoning applied to a different kind of byte: pearl-resident writer, pearl-local store, an MCC route reading off local disk, never a `/daemon/*` endpoint. `storage.py` and `merle.db` are untouched.
  - **5-minute rows, not an hourly rollup.** Issue #72 proposed hourly because that's all its forecast scorer needs, but 5-minute data is a strict superset — hourly means, min/max, and vector-averaged wind all derive from it at read time — and it's what the station already records and the charts already draw. Hourly would throw away resolution we have and cap any pressure curve at a day's zoom. ~105 k rows/year, which SQLite does not notice. **#72's hourly rollup is therefore a query, not a store**, and these rows are the `observations` table its scorer joins forecasts against.
  - **Epoch seconds, not ISO-8601 — a deliberate divergence from `storage.py`.** Every other table in this project stores ISO strings; this one stores integers. The weather bus is epoch end to end, the points arrive as epoch, and the chart consumes epoch, so converting at the store boundary and back at the route boundary would be two lies for zero benefit. (#72's forecast rows keep ISO because they juggle multi-source issuance times; a single-source observation stream has no such problem.) `ts` is the PRIMARY KEY, which SQLite aliases to the rowid — so the range scan is already indexed, and re-recording a point the archive has is a no-op rather than a conflict: `roll_history`'s dedupe rule enforced by the store. Append-only, no UPDATE path. Columns are exactly `HISTORY_FIELDS`; a test asserts the two can never drift apart.
  - **The archive is the one irreplaceable file the stack owns.** Every other piece of state here is a cache of something re-fetchable — `weather_history.json` refills in 48 h, frames re-archive, `merle.db` re-derives. A deleted observation is a moment of driveway weather no API sells back. `Servers/Pearl.md` says so where it can be acted on.
- **Resilience**: any gateway or OpenWeather failure means a skipped report (logged, the next 60 s pass retries — garnish and forecast timers stay put on failure rather than waiting out their full cadence), never a dead service; OWM errors are logged without the URL, which carries the API key. The bus side inherits `EventPublisher`'s run-with-or-without-broker contract.
- **Deployment**: runs on pearl as `willard-weather.service` (see Services on pearl under Quick start in [the hub](../../TechnicalGuide.md)). The unit carries `MERLE_MQTT=localhost:1883`, `MERLE_ECOWITT=192.168.1.210`, `MERLE_OWM_KEY`, and — for the on-air segment — `MERLE_OLLAMA=192.168.1.79:11434` (bluejay's GPU, same as the narrator unit); `WorkingDirectory` is the repo checkout, which is where the default-pathed `weather_history.json` lives. One-time hardware note: the gateway's altitude/relative-pressure calibration is set in its own web UI — until then `rel` equals `abs` and "sea-level pressure" is a polite fiction.
- **Willard's on-air segment** (issue #45): every `REPORT_INTERVAL_S` (30 min, first one on the first loop pass), the service narrates the conditions + next-24h outlook through the narrator's LLM tier — same `MERLE_OLLAMA`/`MERLE_OLLAMA_MODEL` env vars, same kill switch (unset = no `weather/report` topic at all, everything else unchanged), and the same client code: `weather.py` imports `narrator.Ollama` and calls its generic `complete()` rather than growing a second copy that could drift. The persona is an exaggerated **Willard Scott** homage (booming folksy showmanship; the squirrels and turkeys are the viewership), a constant in `weather.py` rather than a `personas/*.yaml` — that system carries narrator-only machinery (`mqtt_id`, pacing, TTS), and Willard is a reporter, not a narrator. The prompt follows the narrator's desk-tested labeled-paragraph shape: a dry `current_facts()` paragraph (since #51 including the station's dew point, rain-falling-now + rate, rainfall today, UV when ≥1, and the barometer — each only when measured, Willard never narrates a hole), a dry `outlook_facts()` paragraph from `forecast_digest()` (next-24h high/low, strongest wind or gust, peak precipitation chance, expected rainfall in inches — each step's #56 rate × its 3 hours, summed — expected snow in inches on its own ledger (#65: Willard says "two inches of snow", never folds it into the rain number), and conditions in order of appearance), a dry `extended_facts()` paragraph from `extended_digest()` (issue #60: one terse weekday-labeled line per local calendar day after today, all ~5 days the API publishes; tomorrow deliberately overlaps the rolling 24h outlook — today in detail, the week at a glance, the way a broadcast desk talks), then the cue; all flavor lives in the persona, and the rules tell Willard to close with a sentence or two picking out the days that matter, never reciting the list (desk-tested live: he singles out the soaker and the nice one). Output is `sanitize_report()`-cleaned (collapses whitespace within paragraphs but *keeps* the paragraph break, unlike the narrator's one-line `sanitize_line`) and published retained as `{ts, text, model}` (`ts` epoch seconds, the weather-namespace convention). Failure = a skipped broadcast retried on the next 10-minute pass (the forecast-retry pattern); a current report older than `REPORT_MAX_AGE_S` (30 min) skips the broadcast rather than narrating yesterday's rain. The blocking Ollama call is safe here — it runs on the main loop's own thread (not paho's; `EventPublisher` has its own network thread) with 10 minutes of slack per pass. — `EventPublisher` grew an optional `status_topic` (Last Will `offline` + retained `online` on every (re)connect, offline published on `close()`), so any future bus process gets presence for one constructor arg. The Last Will fires on any socket death without an MQTT DISCONNECT — a crash, but also systemd's SIGTERM — so `systemctl stop` flips the dashboard within seconds with zero signal handling; the `KeyboardInterrupt` handler covers graceful manual runs. Freshness (`STALE_AFTER_S`) remains the *online-but-not-reporting* signal and the whole judgement when no retained status exists (a pre-#31 service).
- Pure logic (config/location parsing, gateway live-data + sensor-roster parsing, OWM response → bus payload mapping, the station/garnish merge, window roll/prune/thin/persist, segment prompt assembly/digest/sanitation) is covered by `test_weather.py` — the Ecowitt fixture is lifted verbatim from the real GW2000B; the fetch loop, MQTT plumbing, and live Ollama call are desk-tested against the real services (`weather/current` payload round-trips a fresh subscriber intact).
