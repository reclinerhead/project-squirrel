### The narrator (`narrator.py`)

The scene narrator: **one voice per process, LLM prose with template fallback, real pacing** — and since issue #80 the show is a two-hander: **Marlin** (the studio host, on pearl next to the broker) and **Jim** (the field correspondent, on merle — the Pi 5 that will be the rover's brain, positioned there on purpose). Each narrator is its own `narrator.py` process subscribing to `driveway/events` and publishing spoken-style lines to `narration/lines` — it never plays audio itself; a consumer (the dashboard's TTS) speaks. A narrator needs neither the GPU nor the camera, just the bus (the LLM runs remotely on bluejay's GPU). Design decisions:

- **Persona vs bible**: a persona YAML (`personas/marlin.yaml`, `personas/jim.yaml` — name, `mqtt_id`, `tts_voice` hint, `personality_prompt`, pacing knobs) is one voice; `character_bible.yaml` is shared world canon (seed-pile location, Big Chonk lore). Kept separate from day one so multiple narrators never need untangling — Jim reads the same bible and already knows where the seed pile is. Since issue #88 the knobs encode the *roles*: Jim carries announcer levels (0.9 chattiness / 0.4 threshold / 20s cooldown — the field man is the first voice on any new development) while Marlin defers the raw play-by-play entirely (see Announcer roles below). Each process still carries its own Editor min-interval, so the ceiling stays one line per narrator per 30s.
- **One pacing gate** (`worth_speaking()`): cooldown first, then per-kind interest scaled by the persona's `chattiness` against its `interest_threshold`. Silence is most of the show — with the default knobs, most events pass unremarked. The gate is identical across prose tiers.
- **The editor's desk** (`Editor`, issue #74): the narrator-side defense against a noisy bus. Tracker id churn on a steady feeding scene floods `driveway/events` with phantom departure+arrival pairs, and v1 handed each one to the talent — a nonstop Ollama-call storm on bluejay's GPU. The Editor sits between the wire and the roster (scene-level state, one desk for the whole show — personas keep their own cooldown/interest knobs downstream, unchanged) with two defenses. **Hysteresis**: an arrival/departure becomes narratable only once the species' reported count has held at its new value for `MERLE_NARRATE_STABLE_S` (default 20s); a departure undone by an arrival of the same species settles back to the old count and cancels outright — the id-churn signature's explicit outcome is silence, never a hedged line. **Rate limit + burst collapse**: the desk hands the talent at most one event per `MERLE_NARRATE_MIN_INTERVAL_S` (default 30s — a hard ≤2 generations/min ceiling regardless of upstream behavior; deliberately independent of the persona cooldown, which tunes character while this caps cost). Whatever stabilized or happened during a cooldown collapses into ONE `scene_update` event summarizing where the scene ended up — a full `{counts}` snapshot including unchanged species (the scene is the story, not the diff), with quiet-pavement phrasing when empty — which then narrates through the normal gate and prose tiers (its own `INTEREST` entry, templates, and `event_summary` branch; a lone ripened change still narrates as the real arrival/departure, `duration_s` and all). Non-presence moments (`crowd_snapshot`, `clip_recorded`, unknown kinds) skip hysteresis — the daemon already debounces them — but ride the same slot, keeping only the latest per kind. Dropped play-by-play is fine: SQLite has the record; the bus is live transport. The plumbing changed with it: `on_message` now only parses and enqueues (stamped with arrival time, so hysteresis judges when a change was reported, not when the loop read it), and the main thread runs a ~1s pacing loop (`EDITOR_TICK_S`) that drains the queue through the Editor and owns the blocking Ollama call — which therefore no longer runs on paho's network thread, retiring the old "timeout must stay under the MQTT keepalive" constraint. Editor logic is pure (injected clock, no I/O) and covered in `test_narrator.py`.
- **Two prose tiers, one swap point** (`generate()`, issues #9/#20). **Tier 2 (LLM)**: when `MERLE_OLLAMA` (`host` or `host:port`, port defaults to 11434) is set, each line is synthesized by an Ollama server — `MERLE_OLLAMA_MODEL` picks the model (default: `OLLAMA_DEFAULT_MODEL` in `narrator.py`, currently `gemma3:12b`); model choice is deployment config, not persona, because it tracks what's pulled on the Ollama host. The system prompt is the persona's `personality_prompt` (Marlin's is an exaggerated Marlin Perkins / Wild Kingdom homage) + bible canon + output rules that live in code (`LINE_RULES`) so persona files stay pure character; the user prompt (`build_user_prompt()`) is an optional recent-lines memory block over a dry factual `event_summary()` — the standing hook for extra bus context, which now carries the weather sentence. **Tier 1 (templates)**: Mad-Libs fill-ins from the event + bible, and the fallback whenever the LLM is unset, unreachable, times out, or returns nothing — the show never goes silent. `MERLE_OLLAMA` unset is the kill switch and keeps a bare checkout working.
- **Weather context** (issue #26): the narrator subscribes to the retained `weather/current` on its existing MQTT connection and caches the latest payload; when a fresh report is in hand the user prompt gains a labeled current-conditions paragraph — one deliberately dry sentence ("It is 78F with a light breeze under overcast clouds.") plus a one-line usage nudge (`WEATHER_GUIDANCE`); all flavor stays in the persona prompt. The paragraph-plus-nudge shape is desk-tested necessity, not decoration: gemma3:12b ignored the same sentence appended bare to the event summary 4 generations out of 4, and wove it in naturally 4 out of 4 once labeled and nudged. A report older than `WEATHER_STALE_S` (30 min = 3 missed 10-minute polls) is ignored, so a dead weather service degrades to exactly the no-weather prompt — never to narrating yesterday's rain. Weather rides the LLM prompt only (templates stay v1) and is context, never a speaking trigger — the pacing gate doesn't know it exists.
- **Recent-lines memory** (issue #28): each `Narrator` keeps an in-process deque of its last 10 spoken lines (`MEMORY_LINES`; ~400 extra prompt tokens, no knob needed) as `(ts, event_kind, text)`, captured in `speak()` right where the bus payload is built so memory and the bus can never disagree — template fallback lines count (they went on air), a failed LLM call leaves no trace. The prompt gains a "recently on air" block above the event summary, ages rendered with the `human_duration()` vocabulary; the variety/continuity guidance travels in the block header (not `LINE_RULES`), so an empty memory — including every restart, which deliberately starts blank — degrades the prompt to exactly the pre-memory shape. The deque is deliberately separate from the journal window below: the journal is the show's record for the dashboard, the deque is prompt seasoning, and seeding one from the other would couple what the audience sees to what the model is told.
- **Mention triggers** (issue #80): the opt-in persona knob `answers_to` (a list of names; absent/empty = the machinery is entirely off and the process is byte-identical to the one-voice era — the #26/#28 degradation convention). When set, the narrator additionally subscribes to `narration/lines`: a line whose `mqtt_id` differs from its own and whose text matches any `answers_to` name (case-insensitive, word-boundary — "Jimmy" is not a mention of Jim; `mentions_name()`/`colleague_mention()`, pure and tested) becomes a synthetic `kind: "colleague_mention"` event with `details: {narrator, mqtt_id, text}`. `INTEREST["colleague_mention"]` is 0.95 — the marquee trigger clears any sane threshold — but the event rides the existing Editor as a non-presence *moment*, so it respects the narration rate limit like everything else; if a busy burst collapses it into a `scene_update`, that's showbiz. `event_summary()` quotes the colleague's line and cues the follow-up; a Tier-1 `colleague_mention` template pair keeps the acknowledgment airing when the LLM is down. **Never a direct reply channel** — everything rides the bus. Loop safety with both narrators listening (issue #88 cashed in #80's "revisit" note): a narrator ignores lines carrying its own `mqtt_id`, and a line whose `event_kind` is itself `colleague_mention` never triggers — **never reply to a reply** — so a chain is bounded at announcement → one follow-up per listener. A standing instruction still reaches the colleague through the heard-lines memory (Jim carries out Marlin's assignment in his *next* report); it just can't trigger.
- **Announcer roles** (issue #88): with both narrators live, both gates cleared the same event — a double announcement seconds apart. The fix is a deterministic role split, not coordination: the opt-in persona knob `defer_events_to` (a list of colleague `mqtt_id`s; Marlin carries `[jim]`) subscribes the narrator to the existing retained `narrators/+/status` presence (parsed Python-side by `bus.narrator_status_id()` and handled *before* the JSON path — status payloads are raw strings), and `wants()` refuses raw events while any listed colleague reads `online` (`defers_event()`, pure): the field has the announcements. Mention follow-ups are never deferred — reacting to the field report *is* the studio's beat (Marlin also carries `answers_to: [Marlin]`, and Jim's prose tosses to the studio by name). When every listed colleague is dark (`covering_field()`), announcements resume through the normal gate math and the LLM prompt gains one dry `COVERING_NOTE` paragraph — the persona hangs the coffee-break ribbing on it — so the show keeps running when merle/the rover is off, with the Last Will flipping the roles within seconds in both directions. Colleague presence lands in `Narrator.colleagues` from paho's network thread (per-key dict assignment, the `latest_weather` atomicity pattern).
- **Broadcast-context memory** (issue #80): alongside the own-lines deque, a mention-listening narrator keeps a small bounded record (`HEARD_LINES`, 5) of recent lines *heard* from other narrators — `Narrator.hear()`, a plain tuple rebuilt per line so paho's network thread can write while the pacing loop reads (atomic assignment, the `latest_weather` pattern, no lock). The prompt gains a "recently on the broadcast, other voices" block (`heard_block()`) between the own-lines memory and the event summary, with don't-repeat/build-on-it guidance riding the block header — this is what makes a mention follow-up context-aware and non-redundant against what Marlin already said. Empty block (every persona without `answers_to`, and Jim before anyone speaks) ⇒ prompt byte-identical to today, same as the memory/weather blocks.
- **The field journal window** (issue #58, per-narrator since #80): the last `JOURNAL_LINES` (50, matching the dashboard's `JOURNAL_LIMIT`) published lines, oldest first, published retained to `narration/journal/<mqtt_id>` (`bus.narration_journal_topic()`) as `{lines: [...]}` and republished whole on every new line — the weather/history pattern, so a fresh dashboard tab rehydrates the Field Journal straight from the broker with no HTTP path. Persisted to a flat JSON file (`MERLE_NARRATION_JOURNAL`, default `narration_journal.json` in the unit's `WorkingDirectory` on pearl — the `MERLE_WEATHER_HISTORY` convention) so a narrator restart doesn't blank the journal either; missing and corrupt files both mean "start over" (the journal is the show's recent record, not a record of record — SQLite keeps the events). The window is also republished on every (re)connect: the file, not the broker, is the source of truth, which heals the retained copy after a broker restart (Mosquitto only keeps retained state across restarts with persistence configured). File first, then the retained publish, so a crash between the two loses a broadcast, never the record. Deliberately a flat file, not SQLite — bounded, rewritten whole, refetchable (the `weather_history.json` argument). `narration/lines` itself is unchanged and stays non-retained: it carries moments (TTS speaks each once); the journal carries state. Continuity is the model's job, not code's: the block gives timestamps and event kinds and lets Marlin *speculate* about returning individuals in character — real identity stitching across the bus is issue #22's problem. Tier 1 gets only the cheap version: remember the last template index per event kind and re-roll once on an immediate repeat.
- **The Ollama call is blocking and non-streaming** (`stream: false`, one `POST /api/generate`): the bus contract is one JSON object per spoken line and the dashboard TTS speaks whole lines, so nothing downstream could use a token stream. Since issue #74 it runs on the main pacing loop, not paho's network thread, so `OLLAMA_TIMEOUT_S` (30s; desk-tested ~8s warm, ~15s cold-load) no longer has to stay under the MQTT keepalive; events arriving mid-generation wait in the queue and mostly collapse at the Editor afterwards, which is the desired pacing anyway. `keep_alive: 30m` keeps the model resident between sparse events. Rejected alternative: streaming over MQTT would have needed a new topic contract plus reassembly in the dashboard for no audible benefit. The `Ollama` class splits into a generic `complete(system, prompt, num_predict, temperature)` core and the narrator-shaped `narrate()` wrapper — `complete()` is shared with the weather post's Willard segment (issue #45), so there is exactly one Ollama client in the codebase.
- **Embedded producer**: the producer/orchestrator lives inside `narrator.py` as `Producer`, deliberately shaped around a *roster* (a set of voices) even though the roster is one — `cast(event)` picks who speaks, so solo-beat/banter-beat casting slots in later without a rewrite. What narrators do when a standalone producer is absent is a known future question, deferred to the promotion issue.
- **Still-shot pass-through** (issue #90): a spoken line caused by an event whose `details` carry a `frame_id` copies the id onto the `narration/lines` payload — and therefore into the journal window — so the Field Journal can show what he was narrating (the event-time frame is the honest moment: by speak time, 20–50s of hysteresis and rate limit later, the scene has changed). The Editor's burst collapse carries the **newest ripened change's** `frame_id` onto the `scene_update` (best-effort; a moments-only collapse carries none); a lone ripened change or moment passes its own through untouched. `colleague_mention` lines never carry one — their details are the colleague's line, not a moment on the pavement. Absent `frame_id` ⇒ payloads byte-identical to today (the #26/#28 degradation convention): old journal files, stale retained windows, and template-tier lines all keep parsing.
- Pure logic (gate, scoring, templates, persona loading, LLM config/prompt-building/sanitation/fallback) is covered by `test_narrator.py`; the MQTT plumbing and the live Ollama call are desk-tested against the real services.
- **Deployment**: Marlin runs on pearl as `narrator-marlin` (see `Servers/Pearl.md`), Jim on merle as `narrator-jim` (see `Servers/Merle.md` — the full bare-Pi-to-enabled-unit runbook; the Pi installs only the narrator subset, `paho-mqtt` + `pyyaml`, never the vision stack). Both units set `MERLE_OLLAMA=192.168.1.79:11434` (bluejay). Ollama on bluejay binds the LAN address; if Windows Firewall blocks inbound 11434, a narrator silently degrades to templates — check the log's startup "narration tier" line and per-call `[ollama]` fallback messages when lines sound suspiciously Mad-Libs.
- **Rehearsal**: `replay_events.py` republishes archived SQLite events onto the bus with original relative timing (`--speed`, `--kinds`, long silences clamped by `--max-gap`) — the narrator can't tell the difference, which is the point of the bus.
