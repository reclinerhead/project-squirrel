# The MCC dashboard

> Spoke of the [Merle Technical Guide](../../TechnicalGuide.md) — read the hub first for the machine roster, quick start, and cross-cutting conventions.
>
> **Covers:** `mcc/` — the dashboard proper: panels, the daemon proxy, Field Journal / lightbox / broadcast view, the weather views, design language
> **Runs on:** dev on bluejay (`:3001`), production on pearl (`mcc-dashboard`, `:3000`)
> **Related:** epic #1; design issues #113 #185

### The MCC dashboard (`mcc/`)

The daemon's face: a Next.js App Router app, one page, one client component (`components/Dashboard.tsx`) that polls `/daemon/state` every second (10s while the daemon is asleep) and renders the live MJPEG stream plus the instrument rail (current counts, run census, controls, event log, and coming-soon placeholders for future panels).

- **Production runs on pearl** as `mcc-dashboard.service` (`next start`, port 3000) and deploys via `Servers/deploy-mcc.sh` — the script enforces pull → install → build → restart, because `next start` serves the compiled `.next/`, not source: pull + restart re-serves the old code, and building after the restart rewrites `.next/` under the live server. A `fast-stop.conf` drop-in sets `TimeoutStopSec=5` (Next's graceful shutdown waits on browser keep-alive connections, up to systemd's 90s default — the MCC is a stateless proxy with nothing worth waiting for). The unit's `MERLE_DAEMON_URL` points at bluejay, which is why the daemon there runs with `--host 0.0.0.0` (see Quick start in [the hub](../../TechnicalGuide.md); the Asleep panel's wake command prints exactly that). Ops detail lives in `Servers/Pearl.md`.
- **The Live Watch source toggle** (issue #236, `SourceToggle` in `Dashboard.tsx`): an "eyes" nameplate in the Live Watch masthead — the EnhanceToggle idiom, one pill per source in `/state`'s `sources` roster, rendered only when the roster has 2+ entries (the synthetic world and older daemons keep exactly the old masthead; the roster is stable per session, so nothing appears or resizes after first paint). Clicking an inactive pill fires `set_source` through the `/daemon` proxy (`setSource()` in `lib/daemon.ts`) and the row disables until `/state` reports the swap, with an 8s failsafe — a refused swap never flips `/state`'s `source`, so the lit pill simply never moves (the snap-back is the daemon's honesty, not client state). The swap gap rides the existing reconnecting veil, and the stream `<img>` is deliberately **not** remounted: `/daemon/stream` is the same endpoint either way, so frames swap server-side under the open MJPEG connection (verified: the `?v=` key never bumps while `naturalWidth` flips 3840→640). The row dims and disables while the daemon is asleep — the roster from the last honest `/state` keeps the masthead geometry, but there is nobody to command.
- **The rail panels have fixed species slots.** On the Pavement and Run Census render one row per species in the daemon's `/state` roster (stable roster order via `rosterCounts()` in `lib/daemon.ts`), zero or not — a zero-count row dims instead of disappearing. Rows never insert, remove, or reorder as counts change, so a species blinking in (or a one-second misidentification) lights its gauge rather than shoving everything below it down and back up. Species counted but absent from the roster (older daemon, future classes) are appended so nothing is hidden.

- **All daemon traffic goes through an explicit proxy route** (`app/daemon/[...path]/route.ts` → `MERLE_DAEMON_URL`, default `127.0.0.1:8000`; on pearl it points at bluejay). The browser stays same-origin — no CORS in the daemon — and a phone on the LAN reaching the MCC also reaches the daemon through it; the infinite MJPEG stream passes through un-buffered as a piped body. It was a `next.config.ts` rewrite until issue #35: Next's internal proxy logs every failed upstream attempt, and with the MCC running 24/7 on pearl while the daemon only runs during bluejay test sessions, daemon-down is the *normal* state — at a 1s poll that was ~86k journal lines/day. The route instead answers a quiet `503` (any non-OK response is already the client's "asleep" path) and journals only the **transitions**: one line when the daemon drops, one when it returns. A client-side abort (closed tab) is neither logged nor treated as a transition. Upstream connects are bounded by a **3s headers timeout** (cleared once headers land, so it can never cut the infinite stream body): a dead daemon on localhost refuses instantly, but cross-machine Windows Firewall silently *drops* packets to a listenerless port and the connect would otherwise hang for minutes — the dashboard's first symptom was a permanently "loading" page on pearl. **Bus traffic is the one exception**: HTTP proxying can't carry WebSockets, so the browser connects to Mosquitto on pearl directly (`NEXT_PUBLIC_MERLE_MQTT_WS` in `mcc/.env.local`; `lib/bus.ts`).
- **Field Journal** (in `Dashboard.tsx`): the narration panel. Subscribes over mqtt.js to the retained per-narrator journal windows via the `narration/journal/+` wildcard (issue #58, namespaced in #80) — the entries' source of truth, every narrator's window replayed by the broker the moment the tab subscribes, so a reload rehydrates the whole show the way the Weather Post rehydrates its state. Windows live in a ref keyed by the topic-derived `mqtt_id` (`journalTopicId()`) and merge chronologically (`mergeJournals()` — stable sort on the ISO `ts`, capped at `JOURNAL_LIMIT` keeping the newest), so one narrator restarting or republishing can never blank the other's history; the retired bare `narration/journal` topic parses to null, so a stale pre-#80 retained blob is ignored. Also subscribes `narration/lines` and `narrators/+/status` (presence lamps, one per narrator: a dot + the name, no on/off-air text (issue #84) — the pulsing green `lamp` dot reads as "live" on its own, so a roster of two fits the panel label; `online` is the pulsing led dot + led name, `offline` a static faint dot + faint name, and any other retained payload shows `name · status` verbatim with the breathe/turkey dot — a future narrator can be "on coffee break"). The live lines topic keeps two jobs: TTS (speaking is a *moment* — a replayed window must never re-speak) and an instant prepend ahead of the window republish, which also keeps a pre-#58 narrator working through a deploy. Entry keys are derived from line content (`toJournalEntries`), not minted on arrival, so a republished window keeps React's DOM nodes and the filed-entry animation plays only for genuinely new lines (entries render in the display face — the narrator's *voice* against the mono telemetry). Each empty state says exactly which command to run (bus down vs no narrator hired), and a failed connection surfaces its reason inline (`busError`). A default-muted speaker toggle TTS-speaks new lines via `speechSynthesis`, matching the persona's `tts_voice` hint against installed voices by substring (`pickVoice`). Pure parsing/shaping (`parseLine`, `parseJournal`, `toJournalEntries`, `statusTopicId`, `busUrl`, `pickVoice`) lives in `lib/bus.ts` with Vitest coverage.
- **Journal still shots** (issue #90, presentation reworked in #96): any entry whose line carries a `frame_id` renders a lazy-loaded thumbnail **floated left in its text flow** — the prose wraps down its right side magazine-style (`flow-root` on the entry body contains the float, so a one-line entry still holds open the print's full height; the print sits left in both the panel and the broadcast view since issue #107) — served by `app/frames/[id]/route.ts` from `MERLE_FRAMES_DIR` (the folder `frame_archiver.py` fills on pearl — `?thumb=1` is the ~320px copy the panel shows; bare `/frames/<id>` the full-size variant the lightbox consumes). This is the one image path that does **not** ride the `/daemon` proxy: the whole point of archiving on pearl is that thumbnails survive bluejay's nap, so a fresh tab rehydrating the journal from retained topics gets its images too — the browser never reaches bluejay for journal images. The route answers quietly — 404 for missing/pruned files, unset dir, or an unsafe id (`frameFilename()` in `lib/frames.ts` mirrors the archiver's `[A-Za-z0-9_-]` allowlist; Vitest-covered) — and marks hits `immutable` (a frame_id's bytes never change, so window republishes re-render without re-fetching). Layout is the no-layout-shift rule made concrete: an entry's image presence is known at arrival (`frame_id` on the parsed line — optional in `lib/bus.ts`, key absent when the wire had none), so entries with images reserve a fixed 144×81 slot (`FrameThumb`) from first paint and entries without never gain one; a missing image fills its reserved slot with a quiet "faded" stamp, never a broken-image icon. The error state lives in the component, which rides the entry's stable content-derived key, so a window republish keeps it.
- **The journal lightbox** (issue #96, `yet-another-react-lightbox`): a live print is a button (`cursor-zoom-in`, hairline hover brightening, keyboard-focusable) opening the full-size `/frames/<id>` in a lightbox themed to the station (`.yarl__root` overrides in `globals.css`: near-opaque pine-black backdrop, ink-scale chrome, the entry's byline — narrator + broadcast clock — as a stamp-lettered caption, the print in a panel hairline frame). **One instance serves both surfaces** (#102): it lives in `FieldJournal`, the broadcast view is its sibling and opens it through a callback — a second lightbox would mean a second slide list to keep honest. The slide list is **every still in the current window**, opened at the clicked entry, so ‹/› page through the morning's photos; the carousel is finite (the window has real ends — a disabled arrow says so honestly). **Slide order follows the reading order of the surface that opened it** — newest-first from the panel, oldest-first from the broadcast view (`broadcastView` names the live surface, since the view covers the panel and only one is ever clickable) — otherwise the arrows run backwards through the column the viewer just clicked in. The open slide is tracked by the entry's stable content-derived **key, not an index**, re-derived each render via the controlled `index`/`on.view` pair: neither that order flip, nor entries prepending, nor a window republish can swap the photo under the viewer, and an entry aging out of the window closes the lightbox rather than showing a neighbor. Escape, backdrop click, and the close button all exit; opening and closing shift nothing underneath (the overlay is fixed-position, the no-layout-shift rule). A "faded" print is inert — the archive prunes both sizes together, so there is nothing to open. **Escape belongs to the topmost layer**: the broadcast view's own Escape handler is on `window`, which the lightbox's Escape bubbles up to, so the view stands down (`stillOpen`) while a print is up — one keypress closes the photo, the next the reading room. Its scroll lock is a separate effect with its own deps for the same reason: unlocking/relocking around a still opening would race the lightbox's own `body.overflow` restore.
- **The broadcast view** (issue #89, `FieldJournalView` in `Dashboard.tsx`): an expand control in the Field Journal masthead opens the station-view treatment for the narration — a full-screen CSS overlay (the `WeatherStationView` mechanics: `fixed inset-0 z-50` over opaque `--bg`, body scroll locked, Escape or the close button exits), live from the same entries state the panel holds (one bus client, no new subscriptions), with the presence chips and TTS toggle in its masthead (`PresenceChips` is shared by both mastheads). It reads like a conversation where the panel reads like a log: a centered `max-w-[1500px]` column (issue #107 — the station view's measure, itself the main dashboard wrapper's, so the app's two full-screen overlays agree on the page's width; the room it buys goes to the prints), **oldest-first and pinned to the newest line at the bottom** — scrolling up to reread unpins, returning to the bottom re-pins — and **threading is data-driven**: entries whose `event_kind` is `colleague_mention` are the follow-ups, indented under the report they answer with a `↳` + "follow-up" stamp, no narrator hardcoding. **The flash is for news, not history** (issue #107): `journal-filed` bundles `journal-in` + `journal-flash` and both run *on mount*, which the panel gets right for free (its list stays mounted, so only a genuinely new entry animates) but this view did not — it unmounts on close, so reopening re-ran every entry's mount animation and lit the whole thread up as new. It now snapshots the keys it opened with (`openedWith`, a `useRef` captured at first render); those render with plain `journal-in` and only lines arriving while the room is open get the full `journal-filed` flare. Stable content-derived keys mean a republished window never remounts and so can never re-flash, and reopening re-snapshots — which is what makes read history stay quiet. **Voice colors** (both views): each narrator wears a stable accent on its name stamp and entry rail — hue carries identity, intensity carries recency (newest rail full strength, the rest dimmed to 45%), body text stays ink so readability never pays for attribution. The named cast is art-directed (`voiceColor()` in `lib/bus.ts`, Vitest-covered: Marlin squirrel-orange, Jim turkey-khaki); guest voices get a deterministic palette pick, led last since it moonlights as the live signal. **Its stills are the panel's magazine wrap writ large** (issue #102): the same `FrameThumb` at `size="broadcast"` — ~40% of the reading column, ~580px at the full-width measure (the float drops on narrow screens, where a 40% print would leave a two-words-per-line ribbon beside it), pulling the **full-size** archived variant rather than the panel's thumb, which is only ~320px wide and would land far under its display size, soft on any screen. Lazy loading means only the prints in view fetch, and those are the exact bytes the lightbox opens next, so it comes up warm-cache instant. The reserved fixed 16:9 box is load-bearing twice over: the no-layout-shift rule, and the pinned scroll — the pin sets `scrollTop = scrollHeight` in an effect, so an unreserved image loading afterwards would grow the thread and strand the pin mid-column. Deliberately a reading room, not an instrument wall — more lands on this page later.
- **Weather Post** (in `Dashboard.tsx`, next to the Field Journal): current conditions + a hand-rolled SVG trend chart — observed temps trail solid to the left of a dashed "now" line (from `weather/history`), the forecast extends dashed to the right (from `weather/forecast`, bridged at the last observed point so the lines meet), wind on its own scale underneath. 24h back + 48h forward puts "now" at the 1/3 mark. **The forecast's temperature is station-calibrated at render time** (issue #71, `blendForecast` in `lib/weather.ts`): the offset between the last observed temperature and the forecast's opinion of that same moment (linearly interpolated) is added to every forecast temp, decaying linearly to zero across `BLEND_HORIZON_S` (12 h) — MOS-style bias correction anchored to the instrument of record, not cosmetic smoothing, closing the ~5 °F cliff the OWM grid cell's bias against the driveway's microclimate used to draw at the seam. Temperature only (an additive offset on the station's sheltered wind would go negative — a multiplicative wind blend is a possible follow-up); it runs inside `trendSeries`, so the panel chart and the station view both get it for free, and the hover chip reads the calibrated values because the chart and its readout must never disagree. Render-time rather than publish-time on purpose: the seam is drawn against a station point minutes old, so a blend baked in at the hourly forecast publish would re-gap whenever temperature moves fast — and the bus payload stays OWM's actual forecast (the never-silently-switch-sources ethos; Willard narrates the raw numbers). Since #51 the conditions block reads the station: five fixed right-column rows (wind, humidity + dew point, barometer + trend glyph + UV, rain today + rate-when-falling, sun times), each rendered with em-dash placeholders so data landing never shifts anything. **Outdoor owns the stage; indoor is one quiet strip** ("indoors · at the gateway", WH25 temp + RH) below the conditions — stacked, not tabs: ~15 outdoor readings vs 2 indoor ones would make a tab switch land on a nearly-empty pane and kill the at-a-glance comparison. Barometric pressure groups with outdoor/atmospheric despite being measured indoors — pressure is sky, not room. The barometer's tendency (`pressureTrend`, `lib/weather.ts`) compares the newest observed reading against the one nearest 3h earlier — real points, never interpolation — with ±0.02 inHg as meteorological "steady" and no opinion on a trail shorter than half the span; since #67 the core is the series-agnostic `seriesTrend(history, now, value, eps)`, and the station view runs it over temperature (±1.5°F), dew point (±1°F), and humidity (±3 points) too — temp/humidity arrows mostly ride the sun, honest but diurnal, while a moving dew point means the air mass itself is changing. Dew point and humidity wear theirs in their WxStat sub-lines, the hero temperature on its feels-like line (stacked vertically beside the number since #78, making room for the icon); only the barometer's appears in the small panel. **A condition icon rides left of the temperature in both views** (issue #78): eight hand-drawn SVG glyphs (`ConditionGlyph` in `Dashboard.tsx` — sunny, mostly sunny, partly cloudy, cloudy, stormy, raining, snowing, super windy) in the house line-work style — structure strokes inherit `currentColor` so stale/off-air dims them with the text, the subject of each sky (sun disc, rain, lightning) wears its accent (`--turkey`/`--led`) only while the report is fresh, and clouds fill with `--panel` so overlaps occlude. Which sky wins is `conditionIcon()` in `lib/weather.ts` (Vitest-covered), precedence drama-first: Thunderstorm > Snow > Rain/Drizzle-or-piezo (`raining === 1` outranks OWM's word — the driveway instrument beats the grid cell) > windy (sustained ≥ 20 mph or gusts ≥ 30, plus Squall/Tornado) > the four cloud-cover states read from OWM's `condition`/`description` (the rest of the atmosphere group — Mist, Fog, Haze — reads as cloudy for now; night/moon variants are a deliberate follow-up). The icon slot (`ConditionBadge`, 44px panel / 58px station view) reserves its footprint whether or not a sky is known — no report means an empty reservation, never a collapsed one. Its own mqtt.js client (same shape as the Field Journal's — panels stay self-contained; Mosquitto shrugs at a second WebSocket) subscribes to the retained weather topics, so a fresh tab paints weather instantly with zero HTTP. Beneath the chart's time axis sits **Willard's on-air segment** (issue #45): the retained `weather/report` narration rendered in the display face with a "willard, on the air" stamp + broadcast clock — the Field Journal convention (voice = display type, telemetry = mono). A segment older than `REPORT_STALE_S` (90 min = three missed half-hour broadcasts) is history, not news, and yields to the between-broadcasts placeholder; the block reserves a fixed min-height either way (no-layout-shift rule) and lives inside the dimmable content, so it fades with the rest of the panel when Willard is off duty. `parseReport` lives in `lib/weather.ts` with Vitest coverage. The masthead bills the reporter Field Journal-style — green lamp + "willard with the weather" when fresh (Willard is the weather post's on-air name; purely a dashboard label for now, a real persona if he ever narrates), "on coffee break" the moment the retained `weather/status` reads `offline` (the veil then shows when he last checked in, clock + relative age — the chart's "now" line means nothing once he's off duty — plus the `systemctl start` command), a turkey-lamp stale warning when the report is older than 30 minutes with no offline announcement, "off the air" when there's no report at all — while the conditions text ("broken clouds") sits with the temperature in the data block. The panel reserves its full footprint before data arrives (the no-layout-shift rule); when off the air, the data skeleton dims to 30% and the what-to-do message overlays it (the Live Watch veil idea), with bus-unreachable and no-service-publishing kept distinct. The chart is interactive (issue #40): hovering (or touch-scrubbing — `touch-action: pan-y`, so vertical page scroll survives) shows a crosshair + readout chip that **snaps to the nearest real report/forecast point** (`nearestPoint`) — never interpolated values — with day + time, temp, wind/gusts, condition, and an observed/forecast stamp; the chip rides the roomier side of the crosshair and everything hover-related is an absolute overlay, so it can't shift layout. Because the viewBox is stretched (`preserveAspectRatio="none"`), pointer math runs against the container rect and the marker dots/labels are HTML overlays, not SVG text/circles (which would distort). The time axis carries 12h ticks (`timeTicks`) with faint gridlines, plus near-black night bands (`nightBands`) — one sunset→next-sunrise interval per day, each day's sun times **computed from the station's lat/lon** (issue #111, `sunTimes` — the pure NOAA solar-position algorithm) rather than repeated from today's, so the shading is right at any pan depth — so the 72h window reads as days at a glance. Scale whispers sit on the chart edges: temp min/max on the left (squirrel-tinted), wind ceiling on the right (the two series ride different scales, so a shared axis would lie). Pure parsing/shaping (`parseCurrent`, `parsePoints`, `trendSeries`, `blendForecast`, `tempRange`, `windCeil`, `seriesCeil`, `pressureRange`, `pressureTrend`, `linePath`, `nearestPoint`, `timeTicks`, `nightBands`, `sunTimes`, `compass`) lives in `lib/weather.ts` with Vitest coverage; temp axis pads to a 16°F minimum span so a flat day reads flat, wind axis floors at 10 mph so calm days hug the baseline, pressure holds a 0.3 inHg minimum span for the same reason.
- **The weather archive's read path** (issue #105, `app/weather/history/route.ts`): `GET /weather/history?from=<epoch>&to=<epoch>` → `{points: [...]}`, the deep past the retained bus window can't carry (a payload republished whole on every append has to stay bounded). Like `/frames/[id]` and unlike everything else the dashboard fetches, it does **not** ride the `/daemon` proxy — the writer is on pearl precisely so the record survives bluejay's nap, so the reader is a route off the same local disk. The body is byte-shaped like a `weather/history` bus payload, so `parsePoints()` reads both and **the chart cannot tell an archived point from a retained one**; the client half is `fetchArchive()` in `lib/weather.ts` — thin `fetch`, hand-mirrored types, throw on non-OK, `cache: "no-store"` (the `lib/history.ts` convention; deliberately *not* the frames route's `immutable`, which is honest only because a frame's bytes never change — a time range's contents grow). The range is **clamped to 90 days** at both ends of the wire (`parseRange` here, `clamp_range` in Python — the `/history` precedent: "a typo can't bucket ten years"), anchored at `to`, so `from=0` returns the newest 90 days rather than scanning the table; an inverted range is left inverted because it honestly selects nothing. **Unset `MERLE_WEATHER_DB`, a missing DB file, a malformed range, and an empty range all answer a quiet `{points: []}`, never an error** — on day one an empty archive is the normal state, and unlike a pruned frame (a 404), "no data here" is a chartable answer the gap-drawing chart already handles. The route has **no default path**, deliberately: `mcc-dashboard`'s `WorkingDirectory` is the `mcc/` subdirectory while `willard-weather`'s is the repo root, so a shared *relative* default would name two different files and serve an empty archive with no error anywhere — the unit carries an absolute path, the `MERLE_FRAMES_DIR` coupling exactly. No UI consumes it yet; #106 is the deep-history chart it was built for.
- **Gotcha — the SQLite driver is `node:sqlite`, the stdlib module** (issue #105's open question, resolved by checking rather than assuming). It needs **no dependency and no native build on pearl**, which is what settles it against `better-sqlite3` (proven and synchronous, but a native module wanting a build toolchain on the box). The catch is that it has been experimental and flag-gated: unflagged from Node **23.4**, and it stops emitting an `ExperimentalWarning` somewhere in 24.x (pearl's 24.18 is quiet; a 24.14 desk still warns on import — harmless, and the reason a stray warning in a build log isn't a bug). Pearl runs 24.x, so it just works; **if pearl's Node is ever downgraded below 23.4, this route is what breaks**. Two consequences already visible in the repo: `@types/node` had to go from `^20` to `^24` (the types ship with the runtime, and `next build` type-checks the route), and the route is untested in CI — the `web` job pins Node 22, where the import would need a flag. That's tolerable only because nothing in Vitest imports the route; its pure half (`parseRange`) lives in `lib/weather.ts` and is covered there. The database is opened **read-only, per request** rather than cached in a module: the MCC doesn't own the file, on a fresh pearl this route can serve traffic before `willard-weather` has created it, and a cached handle would pin that day-one miss until the next deploy — a local SQLite open is microseconds, and this is a hand-fetched panel, not the 1 s `/state` poll.
- **The station view** (issue #51, `WeatherStationView` in `Dashboard.tsx`): an expand control in the Weather Post masthead opens a full-screen CSS overlay (the Live Watch fallback pattern — `fixed inset-0 z-50` over an opaque `--bg`, body scroll locked, Escape or the close button exits; no `requestFullscreen`, it's a page of instruments that should scroll on phones). Left column: hero conditions (big temp + the WxStat grid: dew point, humidity, wind + gusts + max daily, barometer + tendency, UV + solar, rain today + falling-now lamp; rain ledgers and VPD in a fine-print line — the four less-self-explanatory metrics carry a **circled-i info affordance** (issue #63): a quiet icon after the stamp label whose hover or keyboard focus reveals a field-manual note in the readout chip's dress explaining how the station measures it and what the number means; pure CSS via `group-hover`/`group-focus-within`, an absolute overlay so revealing it never shifts the grid, `tabIndex` so touch taps and keyboards get it too) and the trend chart writ tall — its own **144h window** (issue #60: the panel's 24h trail + the full 5 days the free API publishes, "now" at the 1/6 mark; fixed span, never sized to the payload, so a short forecast leaves the far end honestly blank instead of reflowing the axis) plus **three instrument strips sharing its time axis** (rain-rate bars in weather blue, the barometer's wander, solar in turkey yellow with UV dashed on its own scale), so every timestamp lines up vertically across all four. At that span hour-offset ticks stop meaning anything, so the gridlines are the **viewer's local midnights** (`dayTicks` in `lib/weather.ts` — DST-safe 36h-then-refloor stepping, Vitest-covered) and the axis reads as a calendar: each short weekday label centered over its day's slice, with narrow edge partials and anything shadowing the "now" stamp suppressed (the hover chip names every point anyway); the panel chart keeps its 72h window and offset ticks — six days in ~400px would be a smear. The overlay wrapper matches the main dashboard's `max-w-[1500px]` (the right rail is fixed 320px, so the extra width all goes to the chart). The strips chart the observed trail — the forecast carries none of those series, and the honest blank future half of each strip is where the series label sits. Two exceptions. **The rain strip changes units at the now line** (issue #65, superseding #56's volume bars): the observed half is the piezo's measured in/hr (ceiling from observed points only), the forecast half is the **chance of precipitation** — `pop` as ghost bars on a fixed 0–100% scale (full strip = certain), the same `--rain` ink as the observed bars but shaded by chance and ceilinged below their full voice (issue #113), each spanning the 3-hour OWM window ending at its point, clipped at the now line. Chance is what a viewer plans around; the forecast's volumes read as slivers on the piezo's scale and live in the hover chip instead. **Snow is its own strip** below rain, forecast-only ghost bars of `snow_3h_in` in snow-white ink (`--ink`, never the rain's blue), shaded by depth to full white for the heaviest (issue #113), with its own ceiling (floor 1 in/3h, so a dusting hugs the baseline) — the observed half is honestly blank forever (the piezo cannot see snow), and the strip is **seasonal** (issue #69, the owner-sanctioned exception to the reserve-the-space rule): hidden April–October instead of sitting dead for seven months, shown November–March (`snowSeason`, `lib/weather.ts`), with a safety valve — a forecast actually carrying snow shows the strip in any month, so the calendar gate can only ever hide an empty row, never data. One crosshair spans the whole stack; the readout chip gains a station line (rain rate, baro, solar, UV) on observed points, and on forecast points the expected rain rate, snow inches, and chance (`pop`, shown from 5% up; rates that would print as zero stay quiet). Right rail: Willard's segment with display-face room to breathe, then "the station itself" — the indoor strip, absolute pressure, WH90 battery (5-segment meter, LED→turkey→chipmunk as it drains) + voltage, radio signal (4 segments), last-report age. The battery row prints its **voltage first and the meter last** (issue #103) so the two green meters share a right edge and read as one column — a meter placed before its voltage is the only thing in that box that can't line up with the row beneath it. Indoor readings and hardware health share that box deliberately: both are "the station", not "the weather".
- **The station chart pans through history** (issue #106): grab it and drag, mouse or finger, and the 144h window slides back over #105's archive — the barometer's wander over a week is the payoff the archive was built for. **The window is state, not a constant**: `null` means live (24h back, 120h ahead, "now" at the 1/6 mark — the view opens exactly as it always has), and once panned it holds an **absolute right edge**, so a window dragged to last Tuesday stays on last Tuesday instead of creeping as `now` ticks under it. That null doubles as the snap-back control's home. **Span is fixed; only position moves** — `clampWindow` (Vitest-covered) preserves the span on every clamp, and when the walls are closer together than 144h the *right* wall wins, which is not an edge case but the young-archive's normal state: the default window must stay exactly reachable on day one. `trendSeries` and `dayTicks` now take an **explicit `[ts0, ts1]`** rather than deriving one from `now`; `now` survives in `trendSeries` only as the seam that splits observed from forecast. The panel chart (`WeatherPost`) passes the same live window it always had and is **deliberately not pannable** — six days in ~400px is a smear, a legibility decision rather than an oversight.
  - **The right edge is pinned at `now + 120h` and that is not a bug.** Five days is *everything* OpenWeather's classic free tier publishes and `parse_forecast` caps nothing — the UI truncates no data, so there is no scrollable space to invent behind it. More forecast means a different tier or source (#72's territory).
  - **The left wall is discovered, never announced.** The route has no "oldest reading" endpoint (and #106 adds no backend), so the client simply clamps at the oldest point it holds and asks for one `ARCHIVE_CHUNK_S` (7 days) below it whenever a drag *reaches* for something older. The request keys off what the viewer **asked** for, not the clamped result — the clamp pins `ts0` at the wall, so waiting for the clamped window to cross it would mean the fetch never fires at all. An empty answer means the record stops there and sets `exhausted`, which also stops the asking: without it a drag held at the wall would refetch forever. A **7-day** chunk is what keeps a station outage from reading as the end of the record; a longer one still would. Fetch failure and unset `MERLE_WEATHER_DB` take the same path — the chart is exactly the chart it was before the archive existed and panning stops at the retained window's edge.
  - **Panning the first day costs no network.** The bus already publishes 48h and the view charted only 24h of it, so the second day was being thrown away; it now pans for free. The archive is merged **additively** onto the retained trail (`mergePoints`, deduped by ts — the two genuinely overlap, and the bus wins a tie), never replacing it: the trail still rehydrates from the retained topic with zero fetches. Ranges are fetched once and cached in component state, history being immutable once past. The dedupe that matters is a **ref, not state**: a chunk resolving doesn't re-render synchronously, so a `pointermove` landing between the fetch settling and React catching up still sees the previous render's `oldestKnown` and would re-request the same week — `inFlight` can't catch that, since it's already back to zero.
  - **The axes settle when the gesture does.** Every value scale is computed from the points in view, so a drag would rescale them every frame and the chart would breathe — the no-layout-shift rule violated inside the chart's own frame. They freeze at what's on screen on `pointerdown` and settle **once**, when the drag is over *and* any chunk it kicked off has landed: one controlled rescale when the viewer stops moving, never a jitter under the finger and never a jump the instant a fetch returns.
  - **Night bands are computed per day from the station's lat/lon** (issue #111, landed after #106). #106 shipped with a **7-day honesty horizon** because the browser was only ever told *today's* sunrise/sunset (repeated every 24h), and day length drifts ~2 min/day: a week out that's ~15 min ≈ 2px, invisible, but a month out it's an hour, visibly wrong — so rather than drift into fiction the bands simply stopped, and panned deep the chart had no shading. #111 removed the guesswork at the source: `weather/current` now carries the station's **lat/lon** (from OWM's `coord`), and `sunTimes` (the pure NOAA solar-position algorithm, `lib/weather.ts`, Vitest-checked against published almanac times within ±2 min) recomputes each day's sunset→next-sunrise for real. The horizon and its clamp are **gone** — `nightBands(lat, lon, ts0, ts1)` bands any window at any depth. **Honest absence stays the fallback**: a report without lat/lon (an older `weather.py` through a deploy) draws *no* bands rather than drifting ones — #106's ethos survives even though its horizon didn't.
  - **Interaction**: the first drag in the codebase, so it sets the pattern — pointer events throughout (one path for mouse and touch, as the hover scrub already did), `setPointerCapture` so a drag leaving the chart doesn't strand, and `touch-action: pan-y` so a horizontal drag pans while a vertical one still scrolls the overlay (the browser fires `pointercancel` when it claims the gesture, which ends the drag cleanly). Mouse hover scrubs as before and a press-drag pans, release handing hover back. On touch the two collide, so they split: **drag pans, tap places the crosshair** — a finger resting on glass isn't hovering, and today's touch-scrub is a happy accident of pointer events that a tap preserves rather than drops. A press travelling under `TAP_SLOP_PX` is a tap. The snap-back control is a `now` button in the chart's legend line, **always rendered and merely disabled while live** — a control that appeared on pan would shove the legend sideways (house rule #1) — wearing the masthead's chrome and the day-stepper's disabled idiom rather than a new vocabulary. Zoom/span presets are the acknowledged follow-up once the archive holds enough months for dragging 144h at a time to feel tedious.
- **`/weather` is the station view as a page** (issue #142, epic #110 Phase 2 — the launchpad's weather tile needed a plain URL; until then the view was only reachable by loading the dashboard and clicking the masthead). `app/weather/page.tsx` renders `components/WeatherStation.tsx`, a thin client wrapper around the same exported `WeatherStationView` the dashboard overlays — no parallel implementation, so every station-view change shows up in both automatically. The wrapper feeds it from `useWeatherFeed()`, the bus subscription + staleness/presence judgements extracted from `WeatherPost` when this page became its second caller (each caller still runs its own mqtt client — the panels' self-contained idiom). All weather topics are retained, so a fresh tab paints state straight off the broker with no HTTP and no spinner-then-jump; the view's em-dash placeholders reserve the footprint for the first paint. The overlay's close affordance (and Escape) navigates to `/` — standing alone there's nothing to dismiss, so "close" means "the full dashboard". The segment now holds a page *and* an API child (`/weather/history`) — fine in the App Router, deliberate rather than surprising.
- **The Aviary** (epic #182) — the Earl Birdwatch GUI at `/aviary`, in its own spoke: [aviary.md](aviary.md).
- **Gotcha — import the browser build of mqtt.js.** `Dashboard.tsx` imports `mqtt/dist/mqtt.esm`, not the default `mqtt`. The default entry is mqtt.js's **Node** build (Buffer + node streams); Turbopack bundles it for the browser, but it can't serialize packets there, so the CONNECT never sends and the client dies with `connack timeout` (~30s) — the connection opens but never completes the MQTT handshake. The `dist/mqtt.esm` build is self-contained for browsers. Types are borrowed for that subpath in `lib/mqtt.browser.d.ts`. An `error` handler on the client is mandatory: mqtt.js is an EventEmitter, and an unhandled `error` throws in the browser and wedges its own reconnect loop.
- **Design language: "Ranger Station, Night Watch"** — pine-black panels with topographic-contour background, Fraunces display type + Sometype Mono telemetry, and species accent colors that are the *actual box colors* the vision stack draws (squirrel `#FF7031`, chipmunk `#FF3838`, turkey `#CFD231`), so the UI and stream read as one instrument. Tokens live as CSS variables in `app/globals.css`. The two muted inks were raised in #222 — the original `--ink-faint` measured 3.0:1 against `--panel-2` and was unreadable on real hardware at the 9–10px mono sizes it mostly renders at; both now clear WCAG AA worst-case (faint 4.9:1, dim 7.6:1), same sage hue, hierarchy intact. Re-tune by measuring against both panel surfaces, never by eye.
- **`--wing` `#9db35c` is the palette's second non-species accent** (issue #185), and it extends `--rain`'s rule rather than bending it: species colors are the vision stack's literal box colors and mean *"this species"*, while this names a **module** — the Aviary, whose visit chart counts any bird. Per-species meaning is carried by the page itself (a profile chart is already about one bird), so the bars need module identity, not taxonomy. Not invented for the chart: it's the blaze the launchpad's Aviary placard has worn since the tile went live, so the module reads the same from the porch to its own instruments (the narrators' voice-color idiom). **Validated, not eyeballed**: chroma 0.13 clears the 0.10 floor, contrast against `--panel` clears 3:1, and CVD separation from `--led` — the nearest green, and the one it must never be confused with, since green means *live* — is deutan ΔE 13.2 against a target of 12. `--led` never appears as a bar, so the two never share a mark. (The dataviz validator's *lightness-band* check fails this palette wholesale, `--rain` and `--squirrel` included: that band is tuned to its own reference palette, while this one did its own lightness reasoning in #113 and its inks sit where pine-black needs them.)
- **The palette has two kinds of accent, and the distinction is the rule** (issue #113). The three species colors mean *"this species"* and are the vision stack's literal box colors — never pick them for decoration. **`--rain` `#6bb6e5` is the first accent that isn't a species**: it names something the station *measures* rather than something the model *detects*, which is the line that keeps the species rule intact. It also gives `--led` its one job back — green means **live** (presence lamps, on-air stamps, the falling-now dot), and it had been moonlighting as rain since #56. It was **chosen by the validator, not by eye**, and the finding is worth keeping: *any* blue light enough to match `--led` (OKLCH L 0.829) or `--turkey` (0.835) falls under the 0.10 chroma floor and reads **gray** — blue's gamut peaks darker than green and yellow, so "match the palette's brightness" is a trap. `#6bb6e5` (L 0.746, C 0.12) is the brightest blue that still reads blue, and it lands between `--squirrel` (0.711) and `--led` — inside the palette's own range. Its CVD separation from every other ink is ≥ 34 ΔE against a target of 12; the palette's genuinely tight pair remains the pre-existing chipmunk↔squirrel (ΔE 10.1 deutan), which is why `SPECIES_ORDER` keeps orange and red from ever touching.
- **Precipitation wears its intensity as shade** (issue #113): each bar's ink tracks **its own value** — rate for the observed piezo bars, chance for the forecast ghosts, inches for snow — so a 90% Friday reads as a tall vivid slab and a 20% Tuesday as a short ghost without hovering. Redundant with the bar's height *on purpose*: the shade makes no claim the height doesn't already make, so it cannot be misread and needs no legend of its own. (The tempting alternative — height = chance, shade = expected *volume*, restoring what #65 banished to the hover chip — is a real candidate that needs a legend to explain it, and so wants its own issue.) Three details are load-bearing: **the ramp mixes toward `--panel` rather than varying opacity**, because the observed bars are 0.7 viewBox units wide but 5-minute points sit ~0.56 apart — they *already overlap*, and stacked translucent bars compound into intensities the station never measured (the warning that has sat in the rain strip since #56); solid fills overlap by overwriting. **The faint end is floored** so "unlikely" never reads as "no data" — and the floors differ per ink because white and blue don't vanish at the same rate: `RAIN_SHADE_FLOOR` 0.4 and `SNOW_SHADE_FLOOR` 0.3 both land at ~2.1:1 against the panel (measured, not guessed). **The forecast's ceiling stays under the observed trail's full voice** (`FORECAST_SHADE_CEIL` 0.85): a prediction, however confident, must never shout as loud as a measurement — while snow reaches *full* white, because the piezo is snow-blind so its strip has no measured sibling to out-shout. Direction is "vivid = more": on pine-black, fading a colour blends it toward the near-black panel, so faint reads **dimmer and greyer**, not lighter — which is why "solid white for heavy snow" and "stronger blue for a sure thing" are the same ramp described from opposite ends. `precipShade`/`precipFill` are pure and Vitest-covered; the normalisation is what earns the test (hand it the wrong ceiling — `rainMax` vs a fixed 100% vs `snowMax` — and every shade is wrong while still looking plausible).
- **The strips' headings read over their own data** (issue #113). They sit in the top-right corner because #56 put them in the forecast half, on the premise that the station leaves the future blank — **a premise #65 destroyed** when it rescaled the ghosts to chance on a fixed 0–100% strip, where a confident Friday is a full-height bar growing straight into that corner. So they stop relying on it: 11px in `--ink-dim` (was 10px in `--ink-faint`, the palest ink on the board) wearing a `--panel/80` **scrim**. The scrim is the part that matters — measured against the loudest possible bar the heading went from **1.80:1 to 4.98:1**, clearing WCAG AA, and bigger-and-brighter alone wouldn't have: the old label was only 3.23:1 even over bare panel. Both the heading and the left-edge scale whisper stay absolute overlays, so none of it can shift the chart.
- **Each forecast peak and valley wears its temperature** (issue #113, `tempMarks` in `lib/weather.ts`): the question a viewer actually has — how hot Friday gets, how cold tonight — answered without hovering point to point. **They are turning points, not per-calendar-day min/max, and that distinction is the whole design.** Calendar bucketing is right for *speech*, which is why `weather.py`'s `extended_digest` does exactly that for Willard's "days ahead"; it is wrong for labelling a *curve*, and it fails precisely when the chart is most worth reading — on an evening cold front the temperature falls all night, so Friday's coldest sample is 23:00 (a bucket edge, still dropping) and Saturday's is 06:00: two labels seven hours apart on one continuous slide into one valley. Turning points give that valley the single label it deserves with no special-casing of midnight. **Willard and the chart therefore answer different questions** — "how cold does Friday get" versus "where does this curve bottom out" — and will visibly disagree on exactly that night, and *both are right*; that divergence is deliberate, not a bug to file. The labelled point is always a **real** forecast point: averaging the two lows would invent a temperature nobody forecast, which is the one thing this chart doesn't do (`nearestPoint` snaps, `linePath` splits on null rather than bridging). Endpoints are never marked — the first and last points aren't turning points but where the data ran out. **The observed-side bridge is dropped outright rather than merely left unlabelled** (issue #103): `coming[0]` is `trendSeries`' seam stitch, a 5-minute trail sample wearing the forecast series' clothes, and leaving it in the array to act as a *neighbour* is enough to do the damage — it is the one sample guaranteed to sit beside the first forecast step, so on any morning climbing toward its high the last reading before the seam lands above that step and manufactures a valley the weather never had, labelled at the now line where the trail is steepest and cuts straight through the glyphs. That is precisely the 5-minute noise this function is forecast-only to avoid, readmitted one point at a time. `trendSeries` therefore reports `bridged` (only the drawn line wants that point; nothing reading the forecast can tell by looking), and `tempMarks` slices it off **before** the null filter — the bridge is `coming[0]` whatever it measured, so filtering first would eat a real step instead and the phantom would simply move one point right. The cost is a genuine dawn valley losing its label whenever it falls on the first step: that step has no forecast predecessor to turn against, which makes it an endpoint like any other, and silence beats a turning point judged against a different series. A `TEMP_MARK_MIN_GAP_S` (10h) rule thins a shower's 2°F wiggle into the real afternoon; because extrema always *alternate*, the same-kind rival is two marks back, not one — comparing only against the previous mark would never catch the pair that matters. Labels read the **blended** temps (#71 — `trend.coming` is already calibrated) or they'd disagree with the line they sit on, and they're HTML overlays for the same reason as the snapped dots (the viewBox is stretched; SVG text would squash). Forecast only: the observed trail is 5-minute data where every passing cloud is a turning point. **Caveat worth knowing**: OWM's free `/forecast` is 3-hour steps, so the labelled peak is the *sampled* peak — the true daily high may sit between two samples and run a degree or two warmer.
