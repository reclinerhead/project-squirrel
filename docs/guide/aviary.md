# The Aviary

> Spoke of the [Merle Technical Guide](../../TechnicalGuide.md) — read the hub first for the machine roster, quick start, and cross-cutting conventions.
>
> **Covers:** Earl's GUI — `/aviary` + `components/Aviary.tsx` in `mcc/`, its read routes over `earl.db`, the visits chart, field notes, and the `listener/` enrichment passes behind them (`species_profile.py`, `species_analysis.py`, `enrichment_loop.py`)
> **Runs on:** pearl (MCC production + enrichment loop)

- **The Aviary** (epic #182 Phase 1, issue #183, `/aviary` + `components/Aviary.tsx`): the Earl Birdwatch GUI — every life-list species as a tile with its visit count, a live ticker of arrivals with inline clip playback, and a per-species profile page (`/aviary/[species]`, keyed by **scientific name** — the stable identifier BirdNET emits; common names are display). Read-only end to end — until #216 carved the one deliberate exception, species removal (its own bullet below); the reader routes still never write `earl.db`, and the listener needed zero changes. The launchpad's Earl tile went live pointing here (name "Aviary", presence lamp on `audio/status`).
  - **Read routes over `earl.db`** — the `/weather/history` shape verbatim: `/aviary/roster` (life list joined with per-species visit tallies + today's count) and `/aviary/recent` (newest sightings first; optional `species` filter — one sci name or a comma-separated list, exact-match parameterized — and an optional **inclusive** `before` ts cursor for the archive's pages (#211), limit clamped at both ends of the wire). `/aviary/recent` orders by `ts DESC, id DESC`, not bare insertion order: the cursor paginates on ts, and cursor and order must ride the same axis — bare `id DESC` matched ts order only by the coincidence of sequential insertion (measured breaking on a store with out-of-order timestamps); id still breaks same-second ties, keeping two species sharing a window stable. `node:sqlite`, read-only per-request open, quiet `{...: []}` for unset env / missing file / missing tables, **no default path** (`MERLE_EARL_DB` must be absolute in the `mcc-dashboard` unit — the two-WorkingDirectories lesson, third occurrence). `/aviary/recent`'s body is **byte-shaped like `audio/events` payloads** (`kind:"detection"`), so `audioEventFrom()` in `lib/bus.ts` parses hydration and the live bus with one function and the two can't drift.
  - **Visit counts are grouped at query time, never rewritten in the store**: `tallyVisits()`/`collapseVisits()` in `lib/aviary.ts` mirror `gate.VisitTracker` exactly (60s gap against the *last* detection, strictly-greater opens a new visit), so the pre-#175 per-window day (25 rows per singing cardinal) counts as one visit while post-#175 rows pass through unchanged. Grouping ignores source on purpose — two mics hearing one cardinal is one bird visiting. "Today" buckets a visit by its **opening** detection and the boundary is the **client's local midnight**, passed as `?today=` and clamped server-side to ±2 days (the server can't know the viewer's timezone; a typo can't relabel the archive).
  - **The clips route** (`/clips/[...path]` from `MERLE_EARL_CLIPS`) is `/frames/[id]` one sense over: `clipRelPath()` mirrors `gate.clip_relpath`'s allowlist (exactly `<source>/<name>.wav`, `[A-Za-z0-9_-]` only, no dots outside the extension), quiet 404 for missing/unsafe, and a pruned clip renders the "faded" stamp in the reserved player slot, never a broken control. **Served `no-store`, not `immutable`** — decided during #183: best-clip upgrades rewrite a visit's clip **in place** (#175), so an immutable-cached replay during an open visit would pin the opening take and hide the bird's best moment, the one thing best-clip exists to keep; clips are ~9 s LAN-local WAVs fetched by hand, so honesty costs nothing.
  - **Live mechanics are the Field Journal's**: hydrate over HTTP, then merge live `audio/events` over mqtt.js (browser build) with stable content-derived keys (`audioEventKey`: ts|source|kind|species — two species in one window differ, a republish dedupes), prepend-on-arrival with the filed-flash reserved for genuinely live rows (hydrated rows are already-read — the broadcast view's `openedWith` lesson). A live detection **is** a visit opening (#175 publishes only the opening window), so counts increment per event by the same rule the query-time grouping applies. `kind:"sound"` events (#174) render as quieter species-less rows and vanish on reload — bus-only by design, `sightings.py` doesn't persist them (accepted asymmetry, #182). The masthead wears an Earl presence lamp off retained `audio/status`, so the ticker's empty state can tell "listening, quiet yard" from "bus down".
  - **New Arrivals** (issue #224): the rail's middle panel, between the ticker that mentions a lifer once and the visitors rail that treats it like a regular. Species first heard inside the window (24 hours by default — tightened from 48 in #226 — a week on an EnhanceToggle-idiom `24h`/`7d` nameplate) render as **featured cards**: full-rail-width portrait (the 4:3 top-crop tile idiom; a not-yet-dressed lifer wears the pending glyph in the same reserved frame until #217's loop gets to it), display-type name, first-heard moment, lifer number, and a play slot for the first-contact recording — #220's affordance promoted to where the excitement is. **Zero new backend**: a pure filter (`newArrivals`) over the roster the grid already maintains, so a live lifer surfaces the moment the bus announces it (the ticker-prepend precedent — appearing is the panel's purpose; existing cards never reorder, and the window is measured from a mount-time `now`, so a card never silently ages out mid-session). The empty state holds the panel's footprint with the honest sentence for the active window — most days, and deliberately so: the quiet frame is what makes a card landing feel like an event.
  - **The archive hero** (issue #260): a four-tile band between the masthead and the two columns, speaking for the whole record where every other panel speaks for a slice — species on record (sub: the visit total), species this week, species today, and *listening since* (the earliest `first_ts`, with an inclusive days-on-the-air count). The tiles are the field desk's `StandingTile` idiom scaled up a step (`HeroTile` — deliberately its own shape, so the standings band keeps its #220 look untouched). **Everything derives from the roster state the page already maintains** (`archiveStats()` in `lib/aviary.ts`, Vitest-covered): zero fetches of its own, live detections tick the numbers in place, and "this week"/"today" mean *distinct species with `week`/`today` > 0* — the roster's own windows, no third boundary definition, and deliberately not new-lifers (that's New Arrivals' job). *Listening since* is derived, never a hard-coded go-live constant — the archive's first row is the honest start of the record and survives a reprocess. Three states, one footprint: em-dashes while the roster is in flight (a populated archive never flashes the day-one message during hydration), the **fresh-aviary hero** once loaded-and-empty ("waiting for the first visitor" — anticipation, not zeros; the celebration lives here alone, the Life List's empty state stays matter-of-fact), and the tiles otherwise. The fresh state is an overlay on the tile grid rendered `invisible`, so the tiles keep defining the band's height at every breakpoint and the flip to a live scoreboard — the band's whole payoff on a fresh install, and it happens live off the bus with no reload — changes content only, never layout (the New Arrivals reserved-footprint move at page scale). The day count rounds the local-midnight difference (DST skews it by at most an hour) and floors at 1, so a first bird landing past a mounted tab's midnight reads day one, not day zero.
  - **Mobile jump chips** (issue #265): below `lg` -- exactly where the rail stops sitting beside the grid and stacks under 40+ life-list tiles -- the masthead tab bar carries right-aligned `events ↓` / `arrivals ↓` anchor chips to `#latest-events` and `#new-arrivals`. Plain hash anchors on purpose (instant, pre-hydration, shareable); passed to `AviaryMasthead` as an optional prop only by the main page (the lamp precedent -- the profile and archive have neither the stacking problem nor the sections). Reordering the mobile stack was considered and declined: the Life List keeps the top spot as the page's identity. Same issue's addendum: at `lg` the hero band's grid uses the page grid's own tracks (`minmax(0,1fr)` ×3 + the rail's 340px, shared `gap-4`) rather than an even four columns, so the fourth tile shares the rail's column and the third ends exactly at the Life List's edge -- the seams align by construction, and any future rail-width change carries the band with it.
  - **Nothing reshuffles on its own** (house rule #1, made structural): the grid's sort order is *state computed only on a sort click* (`rosterOrder()` returns species keys; name/visits × asc/desc, ties always name-ascending) — live count updates land in the tiles in place, lifers append at the end whatever the sort says, and Today's Visitors bumps counts in place and appends newcomers (its pills sized at `text-[11px]` so the longer species names pair two-across the 340 px rail rather than each claiming a row — #207). Every event row reserves a fixed player slot (play / playing / "faded" / no-clip dot — one geometry), empty states reserve their panels' footprint.
  - **Clip playback, raw vs enhanced** (#190): one shared `<audio>` per view (a second click stops the first), keyed throughout on the **raw clip path** — a row is identified by the recording, not by which rendering of it happened to play, so flipping the mode mid-session can't strand a lit button or double-mark a faded one. A `raw`/`enh` switch sits in the Latest Events and Recent Visits label rows, **defaulting to enhanced**; enhanced mode asks for the pass's `-enh` sibling and, on a 404, retries the raw clip and says nothing. That silent retry **is** the existence check — the pass keeps no state about which clips it has processed, so asking for the file is the only honest way to ask whether it exists — and a clip earns the "faded" stamp only once the *raw* file is gone too, which is the one thing that actually means pruned. The route's guard was **not** loosened for any of this: `-` was already in the allowlist, so a sibling is an ordinary clip name by construction (a test asserts the guard still rejects traversal with an `-enh` suffix). One trap, found in the browser and worth keeping: a missing file reports itself **twice**, via the element's `error` event *and* the `play()` rejection, in an engine-dependent order — the first kicks off the raw retry and the second would then land, read the now-current state, and mark faded a clip that was at that moment playing fine. Each attempt therefore owns an `AbortController`: whichever signal arrives first aborts it and acts, the second sees an aborted controller and returns, and listeners registered against the signal detach with their attempt.
  - The profile page is deliberately a **frame**: a full-width header (names, first-heard + source, grouped totals), then the **magazine body** (#192 — the portrait + its CC-BY credit float left in the description's text flow, the Field Journal's magazine-wrap precedent; `flow-root` so a short description still holds the photo, float dropped on small screens) with **Recent Visits in a 340px right rail** (the dashboard rail width) that **shares the hero's bottom edge**: its wrapper contributes no height of its own on `lg` (its only child is absolutely positioned), so the grid row is sized by the hero alone and the rail fills it exactly, scrolling internally. Letting the rail size itself misaligned it in both directions — a bird with two visits ended short, one with forty ran past. The list keeps its max-height *below* `lg` only, where the panels stack and there is no hero to match: without that cap it renders every row at ~11,000 px and buries the chart — the full width below both columns is the reserved floor for Phase 3's visits-over-time chart. Bird rows in the Latest Events ticker lead with a small square portrait thumb (#192, the music search-result idiom; the glyph in the same reserved square when a species has no portrait), and give the common name its **own full-width line** — the timestamp sits in the meta row ahead of the source and confidence (#207), so long names read in full instead of truncating to share the line with the clock. The analysis prose lands in Phase 4. Pure logic (visit grouping incl. the pre-#175 collapse, roster/today shaping, event parsing, clip-path guard, sort comparators, param clamps) lives in `lib/aviary.ts` + `lib/bus.ts` with Vitest coverage; routes and components stay thin over it.
  - **The event archive** (issue #211, `/aviary/events`, `AviaryEvents` in `components/Aviary.tsx` — the shared-file pattern, so the clip player, thumbs, and masthead stay private): the browsable record the ticker isn't. The ticker keeps its 50/80 hydrate/cap and gains one always-rendered footer link ("browse the full record →"); the page walks the whole store newest-first under **sticky local-day headers** ("Sunday, July 19", today/yesterday called out, per-day event counts) — the date is structural, not a per-row stamp, which is what answers "which day am I in" mid-scroll. Rows are the ticker's vocabulary enlarged: h-16 thumb (`TickerThumb`'s one geometry, size parameterized), display-type name, time, source/confidence/wind stamps, play slot right-aligned. **Species pills, combinable** (most-visited first via `rosterOrder`, static thereafter — a filter bar that reshuffles as you use it is house rule #1 broken in a control); exactly one selected species earns a **solo band** (portrait, totals, first-heard, the enrichment lead, profile link — all off the roster payload, zero new backend). **Filters live in the query string** (`replaceState` writeback, read once on mount behind a `ready` gate so a shared link fetches its filtered view once, not twice) — the /weather being-a-page reasoning. **Pagination is the hydration/live merge trick turned sideways**: pages of 100 via an IntersectionObserver sentinel (600px rootMargin), cursor `before` = the client's own oldest row, *inclusive* on the wire, `audioEventKey` deduping the deliberate overlap — a same-second sighting straddling a page boundary is a no-op, never a dropped bird; `nextBefore()` steps past a theoretical all-one-second page rather than looping. A short page **is** the record's end for the active filter (the WHERE runs before the LIMIT), shown as a fixed-height "the record begins here" endcap sharing its reserved row with the fetching stamp. **Jump-to-date** (`dayAnchor()`: hand-parsed `yyyy-mm-dd` — `new Date(string)` reads bare dates as UTC, a whole day off west of Greenwich — to the last second of the viewer's local day, DST-safe by stepping to next midnight and backing off one) repositions the cursor; the `now` control is always rendered and merely disabled while live (the chart's idiom). Day boundaries, grouping (`dayGroups()` — groups consecutive runs, refuses to sort), and labels are all client-side; the server filters raw epochs only. **No bus client here, deliberately** (v1): it's an archive browser, not a second ticker — new arrivals appear on reload; and only detections ever appear, since sound events are bus-only by design (#182). Verification caveat, hard-won: IntersectionObserver callbacks never fire in the hidden in-app browser pane — the scroll must be verified in a real browser, not "fixed" in code.
  - **The visits-over-time chart** (Phase 3, issues #185 and #204, `components/VisitsChart.tsx`): visits on the species profile — daily bars or a 48-hour hourly curve, the viewer's choice — full-width under the magazine body, **pannable back through the whole record** — the station chart's interaction idiom (#106) copied and its code deliberately not (patterns are shared; `lib/weather.ts` stays unentangled). Grab and drag, mouse or finger: pointer events throughout, `setPointerCapture` so a drag leaving the chart doesn't strand, `touch-action: pan-y` so a horizontal drag pans while a vertical one still scrolls, a `TAP_SLOP_PX` tap placing the crosshair on touch, the y-axis **frozen on `pointerdown` and settled once** the gesture ends *and* any chunk it kicked off has landed, and a `now` control **always rendered, merely disabled while live** (a control appearing on pan would shove the legend — house rule #1). Window semantics are `clampWindow`'s: a span fixed per mode (30 days of bars or 48 hours of curve — see below), `null` = live, an **absolute right edge** once panned, and the right wall wins on a young record so the default window stays reachable on day one. Simpler than the weather desk in one way — there is no forecast, so the right wall is just today and the seam logic disappears.
    - **Two modes, one panel** (#204): a `30d`/`48h` segmented toggle in the chart's label row (EnhanceToggle's nameplate vocabulary — both states always rendered at a fixed width) switches between the **daily bars** and a **48-hour hourly curve**. They answer different questions and neither replaces the other: the bars say *how often* this bird has been around over a month, the curve says *when in the day* it shows up — the dawn spike, the dead afternoon, the evening return, all of which a daily total averages flat. **The chart opens in detail** (the 48-hour curve): the *when-in-the-day* rhythm is the question worth landing on, since the *how-often* the bars carry is already summarized by the standings band directly above — the opening fetch is sized for overview regardless (below), so the default costs no extra network. Each mode's span is fixed within itself; the older claim that `VISITS_SPAN_S` was "fixed forever" is now half-true and stated as such in the constant's own comment — a **clamp** still never resizes a window, but the **viewer** may. Switching keeps the viewer's place rather than snapping home (a rhythm noticed in last April is worth zooming into): a live window stays live, a panned one keeps its absolute right edge re-clamped against the new mode's walls, and the round trip is lossless on any record long enough to hold the 30-day span. On a young record the switch back lands on live, which is `clampVisitWindow`'s documented right-wall-wins rule doing its job, not a lost place. Detail's right wall is the **hour in progress**, not midnight — ending it at midnight would spend half the window drawing a future that hasn't happened. No new endpoint and no refetch: the opening fetch is sized for overview (30 days + a 120-day chunk), which covers 48 hours many times over, so the toggle never waits on the network.
    - **Bars for days, a trace for hours** — the mark follows the data's continuity, not a style preference. Between two days there is none to imply, so daily counts stay bars. Within a day there *is* (a bird present at 6:40 and 7:10 was plausibly around at 6:55), and at 48 bars the picket fence hides exactly the rhythm the mode exists to show. Single series either way, so no legend (the page title names the bird); the crosshair **snaps to a real bucket**, never an interpolated instant, by that bucket's own half-width (`nearestBar`'s `halfWidth`, half a day or half an hour).
    - **The curve is monotone cubic, and that is a correctness choice rather than a cosmetic one.** `smoothSegments()` is Fritsch–Carlson: tangents zeroed at every local extremum and otherwise clamped to 3× the local secant, which pens both Bézier control points inside their own segment's y-range. A cubic Bézier never leaves its control points' convex hull, so **the drawn curve cannot dip below a quiet hour or ring above a busy one** — a plain Catmull-Rom or natural spline through sparse counts does both, and a chart showing −2 visits at 3 a.m. is simply lying. Vitest asserts the hull property directly on spiky series rather than eyeballing the render. Points are planted at the **middle** of the hour they count; on the boundary the whole rhythm would read half an hour early.
    - **Hour bucketing is client-side too, for the same reason days are** — `hourBuckets()` mirrors `dayBuckets()` exactly one unit finer, including opening-detection attribution (a visit that opened at 6:59 and ran past seven belongs to the six o'clock hour). `hourStart()` floors via `Date`, **not** `Math.floor(ts/3600)*3600`: India (+05:30) and Newfoundland (−03:30) put their hour boundaries on the half-hour in UTC, and the arithmetic version would draw every bucket 30 minutes off the viewer's own clock. Stepping advances 3600 real seconds and re-floors, which needs no DST special case — spring forward yields 23 buckets because 2 a.m. never happened, fall back yields 25 with the repeated 1 a.m. hours landing in distinct buckets. The DST tests assert the buckets **tile the real seconds between two local midnights** rather than a hardcoded 23/25, so they hold in CI's UTC and on a desk in US Eastern alike. Note the one thing deliberately *not* reused: `species_analysis.hour_histogram()` buckets in the **server's** local time and would contradict this rule outright.
    - **The server groups; the client buckets, and that split is deliberate** — a correction to what #185 sketched (server-side local-day bucketing). The server cannot know the viewer's timezone, which is exactly the lesson Phase 1 encoded in `parseSince`, so `/aviary/visits/[species]` applies the 60-second visit grouping (which *must* match `gate.VisitTracker`, or a pre-#175 day reads 25 visits instead of 1) and ships **one epoch per visit opening**; `dayBuckets()` folds those into the viewer's local days. The payload is lighter than the day-objects it replaces, and the grouping rule lives in one place for both the chart and the roster. Bucketing steps by calendar day and re-floors, so DST's 23/25-hour days can't skip or double a bar (Vitest covers both directions).
    - **Empty buckets are honest zeros; pre-first-heard ones are nothing at all.** A day the bird didn't come is data (the crosshair reads "0 visits"); a day before the species was ever heard is absence of *record*, not of bird, so it isn't drawn — which is why `first_ts` rides along in the route's body. The y-axis floors at `VISITS_CEIL_FLOOR` so a quiet species reads quiet rather than drawing a full-height bar for one visit (the wind-axis-floor reasoning).
    - **The left wall is discovered, then confirmed**: chunks (`VISITS_CHUNK_S`, 120 days — far more generous than the station's 7, because the payload is an integer per visit) are fetched as a drag reaches past what's held, keyed off what the viewer *asked* for rather than the clamped result (the clamp pins `ts0` at the wall, so waiting for the clamped window to cross it would mean the request never fires). The floor asked-below is a **ref, not state** — a chunk resolving doesn't re-render synchronously, so a `pointermove` in between would re-request the same range. Unlike the weather archive the record's start is *known* (`first_ts`), so reaching it sets `exhausted` outright.
  - **The hero wears the bird** (#196): the species profile's top section carries its own portrait as a softened, faded backdrop — the music player's album-hero idiom (`AlbumView.tsx`, #157) on this palette, whose finding carries over rather than being re-litigated: a heavy blur reads as a colour smear, so this is a *lightly softened band of the actual photo* (`blur-md`, `scale-105` to keep softened edges off-frame) and the bird stays recognizable. Two scrims do the legibility work rather than crushing the opacity — the house bottom-up fade plus a right-anchored one under the text column, so the backdrop is visible at the top-left where the art is the point and gone at the bottom where the prose runs. **Crank the art, then scrim the text.** The first pass got that backwards — it dimmed the photo to 40% *and* used a light scrim, which spent the contrast budget evenly and showed the bird nowhere. The settled values (image at 0.78, text-column scrim at 0.78) come from a **parameter sweep over every portrait actually on the life list**, compositing the exact layer stack in a canvas and keeping the most visible configuration whose *worst* body-text contrast still clears AA: 2.7× the backdrop luminance of the first pass at **4.64:1 worst case** (flat-panel baseline is 6.81:1). Re-tune by sweeping, never by eye — a value that looks right on the jay fails on the robin, whose photo is far brighter, and single-image tuning is exactly how the first pass went wrong in both directions at once. A species with no portrait renders the flat panel exactly as before. The **description clamps with a read-more** (the `ArtistView` bio pattern): Wikipedia's lead ran four or five paragraphs, which buried the chart and the field notes below the fold, and short prose over softened art is legible where five paragraphs would not be. **The clamp is set by the photo, not by a constant** — it fills ~75% of the portrait's height, computed from the stored dimensions (the figure is `md:w-[300px]`; text-sm/leading-relaxed is 22.75 px a line) rather than measured, so it is known before first paint. A tall bird therefore earns ~13 lines and a panoramic one ~7, and neither leaves the gutter beside the photo conspicuously empty; bounds at 5 and 18 lines keep a very wide photo readable and a very tall one from dumping the whole encyclopedia, and unknown dimensions keep the original six. Whether the toggle is needed at all *is* measured, in a `useLayoutEffect` (pre-paint, so no control arrives a frame late) and **only while collapsed** — open, `scrollHeight` equals `clientHeight`, and re-measuring there would delete the "read less" control mid-read.
  - **The field notes** (Phase 4, issue #186, `listener/species_analysis.py`): two prose blocks per species on the profile — *the rhythm* (when this bird shows up) and *weather & timing* (how conditions move its odds) — written by `gemma3:12b` and read by the MCC through `/aviary/analysis/[species]`. **Statistics are computed in Python; the model only narrates them.** That split is the whole design: hand any LLM raw event rows and ask for the pattern and it invents percentages — a 12B local model, Opus and Grok alike — and the invention is invisible in the output. So every figure the prose may speak is computed here, pytest-covered, and stored beside the text as `stats_json`, which makes each sentence auditable after the fact. It also settles the model question: "turn eight facts into two charming paragraphs" is what the local model already does for Marlin and Willard, so the epic's "the local gemma is the wrong tool" note was reasoning from the discarded premise — and since a local model is free, the epic's other consequence (click-gated generation, because generation costs money) dissolved too. It is an ordinary enrichment pass, worklist-driven on a **visit-count watermark** (a bird nobody has heard since June has nothing new to say), and the MCC never writes `earl.db` — #183's rule intact. The Ollama client is the narrator's, borrowed not copied (the weather post's precedent); `complete()` grew an optional `timeout` because the narrator's 30 s ceiling is a live-show pacing number and this is a patient batch job. Since #217 the crank turns itself — the enrichment loop (below) runs the pass opportunistically, gated so a generation only ever spends itself on changed inputs.
    - **Weather claims are exposure-normalised rates, never counts.** "40% of visits were cloudy" is meaningless if 40% of the hours were cloudy; every bucket is visits-per-hour against the species' own baseline, and `weather.db`'s 5-minute rows *are* the denominator. **Hours with exposure but no visits count in that denominator** — sixty silent rainy hours are the evidence, and dropping them (the first implementation did) inflates the baseline and understates every effect. Exposure is counted **only within the species' own active hours**, which is the dawn-confound control: birds sing at dawn, so if rain falls at dawn more often, a rain effect and a dawn effect are otherwise indistinguishable. **Wind is never a behavioural claim** — Earl's own threshold rises above 15 mph (#175), so "fewer birds when windy" partly measures the instrument going deaf. Temperature bands are reported with a standing note that they track the seasons as much as the thermometer. The piezo outranks OpenWeather's word for "is it raining" (the ConditionGlyph rule), and the archive stores `weather.main`, so sky granularity is *cloudy vs clear* and nothing finer.
    - **Findings carry their sample size and thin ones are hedged, not hidden** — a bird heard twice is still worth reading about as long as the page says that's all we have. Validated against a fixture with a *planted* effect (70% of rainy-day visits deleted): the pass recovered "65% below its own average", the expected figure once the baseline itself includes the depressed days.
    - **Three failure modes were found by auditing the model's output against `stats_json`, and all three were fixed in the prompt rather than by reaching for a bigger model** — which is the argument for storing the stats at all. (1) It **inverted a direction**, writing "notably quiet" for a finding of *+82% more often*, reasoning from bird lore instead of the figure; directions are now uppercase keywords in front of the number. (2) A literal example hedge in the rules ("only a handful of mornings so far") became **boilerplate it stapled onto findings resting on hundreds of visits**; the rules now name the label and leave the words to the model. (3) The labels themselves then **leaked into the prose** ("appearances have been MORE OFTEN"), so the rules now say the annotations are never words to copy. A fourth: handed "amcrest: 744, rover: 1" it wrote "slightly more", so magnitudes are characterised in Python — anything the model can get wrong by describing, describe for it.
  - **The field desk** (issue #220): the profile's bottom half under the chart, where the prose used to sit alone. A **standings band** — this-week rank, all-time rank, and yard records — computed at load in `lib/aviary.ts` from payloads the page already holds and **never reshuffled by live events** (house rule #1; a scoreboard jumping under the reader is the worst reshuffle). Ranks are competition-style (ties share a number, the next skips) with a rivalry line naming the nearest species strictly ahead ("two visits behind the Blue Jay"); the week tally rides the roster (`tallyVisits` grew a third window off the existing `?today=` param — `weekWindowStart` subtracts fixed days from the client's midnight, a documented one-hour DST skew accepted for a leaderboard); records (busiest day, current streak, longest silence, earliest/latest hour, lifer number) derive from the visits route's openings in the **viewer's** local calendar via the chart's DST-safe helpers (`yardRecords`, `secondsIntoDay`). The **field notes render as a journal spread**: two facing pages (`JournalPage`) with entry headings over ruled lines, a drop cap, and per-page datelines signed "— Earl" — and each page carries a **margin figure drawn from `stats_json`**, which `/aviary/analysis/[species]` now ships parsed: the rhythm page's 24-cell hour strip (peak window lit in `--wing`, the rest at low opacity) and the weather page's effect pills (`weatherChips` — strongest first, capped at four, about-average and unknown buckets skipped, thin findings dashed and dimmed, and the whole row suppressed when the pass itself judged the sample insufficient — a confident chip over hedged prose would be the figure contradicting the writing). Since #276 the weather figure is **one horizontal row of pills in a fixed `h-[52px]` box matching the rhythm strip exactly** (it was a vertical stack of full-width rows from #257, which grew far taller than the facing strip and left the two pages' prose starting on different lines); equal figure height means the facing paragraphs start on the same line. Each pill carries a **hover tooltip** (`weatherChipExplain`, a pure Vitest-covered helper) decoding its effect in one plain sentence — "Seen 59% more often in mild weather than its overall average", the −100% case softened to "almost never", thin findings appending the caveat the dashed border signals — the tooltip is where the `· thin` words now live and the paragraph beneath still carries the interpretation the tooltip deliberately doesn't. **The figures render the stored stats verbatim, never recomputed**: the same numbers the prose was written from, so figure and prose cannot disagree — with the stated consequence that the hour strip speaks the *server's* clock like the prose does, while the chart above speaks the viewer's (#185's rule); two clocks, both deliberate. **First contact**: the hero's meta row plays `life_list.first_clip` — the retention-exempt lifer recording, stored since #175 and never before played — through the shared clip player; a pre-clip-era lifer wears the reserved no-clip slot. Every tile, strip, and chip row reserves its footprint in loading/thin/empty states, and the no-notes empty state keeps the pre-#220 panel exactly.
  - **Portrait framing** (#185): Wikipedia's bird photos are commonly **portrait-orientation** (measured on the real life list: Northern Cardinal 900×1182, Blue Jay 900×1227), and a centered `object-cover` crop into a landscape box cut off the **head** — the most identifying part of the bird. The pass therefore stores the thumbnail's `image_w`/`image_h` (already in the pageimages response, previously discarded), and the GUI frames from them: fixed boxes (4:3 tiles, square ticker thumbs) **crop from the top** whenever the source is taller than the box (`cropPosition()`), while the profile's floated figure **crops nothing at all** — it takes the image's own aspect ratio (`portraitAspect()`), reserved before load so the photo landing shifts nothing. Unknown dimensions fall back to the previous centered 4:3 in both cases, never a guess about a shape we don't have. A square source in a landscape box top-crops too — `object-cover` fills the width, so it overflows vertically for the same head-losing reason.
  - **Species enrichment** (Phase 2, issue #184, `listener/species_profile.py`): a repeatable pearl-side pass — repo venv, stdlib HTTP only — that fills a `species_profile` table in `earl.db` (created idempotently by the pass's own `connect()`; the sightings schema is untouched) and downloads portraits to the `species/` shelf under `MERLE_EARL_CLIPS`. **Source is the Wikipedia Action API, deliberately not `rest_v1/page/summary`** — the endpoint #184 sketched entered gradual deprecation in July 2026 (`music_bio.py` already litigated this) and never carried image licenses anyway; two calls per species (`extracts|pageimages|pageprops` with `redirects=1` — BirdNET's scientific names resolve to common-name articles via taxonomic redirects — then `imageinfo` extmetadata for license + author). The finished **attribution line is stored, not derived** (`photo: <artist> · <license> · via Wikipedia`) because CC-BY requires it and the GUI should stay dumb. **Provenance from day one**: `image_source` (`wikipedia`/`owner`), and a pass or `--refresh` never touches an `owner` row — the future feeder-cam upload survives every re-run (the artist-image rule, one stack over). Worklist = life-list species with no profile row, so a no-page species stays on it for future looks and re-runs are no-ops otherwise; disambiguation pages parse as nothing (a genus page is not a portrait). The MCC reads only the local store — **no render-time Wikipedia, ever**: the roster route LEFT JOINs the profile columns (falling back to the bare life list on a pre-pass store, so the aviary never blanks), `/aviary/portrait/[species]` serves the shelf by scientific name (`speciesImageName()` mirrors the pass's scrub byte-for-byte; `no-store` — refresh and owner-replacement rewrite in place), grid tiles show the italic scientific name under the common one (#226 — a tile-sized truncated lead earned nothing; the New Arrivals card's vocabulary, and since the sci name always exists the slot never shifts on enrichment), while the profile and the archive's solo band render the full prose + the credit. `sightings.py`'s retention pass **never prunes `species/`** (a permanent collection sharing the rolling window's directory — `prune_selection` skips the prefix, regression-tested). Since #217 the pass is a loop-owned *tick*, not a deploy step: the enrichment loop (below) drains the worklist within ~15 minutes of a lifer landing, with a ~daily retry gate (`profile_attempts` — its own table, deliberately not a stub profile row, because "no profile row" is the semantic both the worklist and the roster's LEFT JOIN rely on) so a species with no article isn't asked about ~100 times a day forever; `Servers/Pearl.md` keeps the by-hand `--refresh` recipe, which never consults the stamps.
  - **The enrichment loop** (issue #217, `listener/enrichment_loop.py`, the `earl-enrichment` unit on pearl): the Aviary tends itself — one loop service (never a systemd timer; the journal-spam rule) ticks every ~15 minutes and runs each pass's worklist whenever its dependency is available: the network for Wikipedia, a reachable Ollama host for the notes. **Profile before notes, deliberately** — `rhythm_prompt()` feeds the model the profile's description, so a lifer gets portrait, background, and first field notes in one tick, background included. **Scheduled by opportunity, not by clock**: Ollama lives on a workstation that's off overnight, so `MERLE_OLLAMA` is now a comma-separated preference-ordered candidate list (`ollama_candidates()`; `ollama_address()` still hands the live show the first entry immediately — the narrator never acquires a probe loop), probed via `/api/tags` (one cheap call answering "alive, holding which models"); no host answering is a quiet no-op tick, and the host coming back at breakfast is the same code path. **Regeneration triggers on changed inputs, never on the calendar**, in two stages: a cheap over-inclusive worklist (never written / 5+ new visits — down from 20, the fingerprint makes the wider net free / `PROMPT_VERSION` bump / an *outranking* model / profile touched since the last write-or-check), then the **stats fingerprint** — a hash of the full stats package plus the exact description snippet the prompt consumes — checked in `analyze_species()` before any LLM call. Unchanged inputs are a `current` skip that stamps `checked_ts` (bookkeeping so worklist over-inclusion settles) and never touches `generated_ts` — the page's "as of" stamp must not claim unearned freshness. **Model changes only regenerate upward** (`MODEL_RANK`, env-overridable via `MERLE_MODEL_RANK`; unknown models sort last): a lower-ranked host can never claw back a higher-ranked row, so two hosts holding different models settle instead of thrashing, and a future bigger box sweeps the archive upward exactly once with no migration. Generations are **capped per tick per host** (`MERLE_ENRICH_CAP`, positional against the candidate list, `0` = never generate there; default conservative because today's host is a workstation someone is sitting at), and an `llm-down` mid-drain stops the slice. The loop and both CLIs share the same per-species functions (the reusable-pass rule), `--refresh` forces past every gate, `MERLE_OLLAMA` unset still kills generation while the profile half keeps running, owner rows stay untouchable, and none of it runs from `merle-autodeploy` — it's a unit of its own, documented in `Servers/Pearl.md`.
  - **Species removal** (issue #216, `DELETE /aviary/species/[species]`): the misidentification eraser — BirdNET's confident mistakes (a rover motor logged as a Wood Duck) otherwise sit on the life list forever. A quiet affordance in the profile hero (the sort control's chrome in the faint ink — the palette's only red means *chipmunk*, never *danger*, so the destructive weight is carried by the confirm step, not by color) opens a confirm **overlay**, never in-place expansion (house rule #1), naming the species, its visit count, and its first-heard date; the dialog is mounted **outside every `.panel`** because the panel reveal animates a transform and a transformed ancestor becomes a fixed element's containing block. Confirming deletes every row for the `species_sci` from `sightings`, `life_list`, `species_profile`, and `species_analysis` (one transaction), then the files: each sighting clip, its `-enh` sibling **named unconditionally** (the enhancement pass keeps no registry — the filesystem is asked by deleting, and a sibling never written is just a missing file), the lifer's `first_clip` (its retention exemption is mooted by deleting the row and the file, not by touching `sightings.py`), and the portrait on the `species/` shelf. The doomed-file set is a pure Vitest-covered helper (`doomedFiles` in `lib/aviary.ts`) that re-runs the clips route's own traversal guard on every row-held path — the MCC's first delete must not let a hand-edited row escape the clips dir. **This is the MCC's first write to `earl.db`**, and it deliberately does *not* wear the readers' quiet-empty semantics: unset env or a missing store answers 404 (a write-mode open would *create* an empty `earl.db`, which a delete must never do), while an unopenable or unwritable store answers 500, loudly — a delete that silently removed nothing while the UI reported success would be worse than an error. Deployment: the `mcc-dashboard` unit's user needs write permission on the DB file *and its directory* (WAL sidecars) and on the clips tree. Rows delete before files, so a mid-walk crash leaves an honest roster and orphaned clips that age into the retention pass's ordinary horizon. The species heard again later is a **brand-new lifer** — intended behavior, and the listener is untouched.
