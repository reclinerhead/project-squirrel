# Merle Technical Guide

The living documentation of how project-squirrel (Merle) is built and why. This describes current state — chronology lives in git log and PR descriptions.

## Quick start — running the station

The station spans two machines. **pearl** (`192.168.1.64`, always-on Ubuntu) hosts the broker (Mosquitto), the narrator (Marlin), the weather post (`weatherpost/weather.py`), and the production dashboard (MCC, `http://pearl:3000`) — always up, nothing to start. **bluejay** (the Windows desktop) runs only the perception daemon, the one process that needs its GPU and camera, from the repo root (PowerShell). Bluejay's standing Ollama install also serves the narrator's LLM calls (port 11434) — not a Merle process, nothing to start. Everything meets on the bus and tolerates the others being absent:

```powershell
# Perception daemon (needs MERLE_RTSP_PASS for the camera and MERLE_MQTT
# for the bus -- both set User-level on bluejay;
# set MERLE_SOURCE=synthetic for the camera-free world).
# --host 0.0.0.0 so pearl's production dashboard can reach it across the
# LAN (loopback-only was the one-box era); needs a one-time Windows
# Firewall inbound allow on TCP 8000 -- the firewall silently DROPS
# blocked packets, so the symptom is a hang, not a refusal.
# --timeout-graceful-shutdown: pearl's 24/7 dashboard always holds an MJPEG
# /stream connection, which never completes -- uvicorn's graceful shutdown
# would wait on it forever, so Ctrl+C hung until the flag bounded the wait
# (a second Ctrl+C is ignored too; the only other escape is killing the PID).
# --no-access-log: that dashboard also polls /state about twice a second, and
# each poll printed a line that buried the daemon's own event prints.
.\.venv\Scripts\python.exe -m uvicorn vision.merle_daemon:app --host 0.0.0.0 --port 8000 --timeout-graceful-shutdown 3 --no-access-log
```

`.\start-merle.ps1` runs exactly that in its own Windows Terminal tab (`-Synthetic` for the camera-free world). For dashboard *development* on bluejay, run `pnpm --dir mcc dev` by hand — it's deliberately not in the launcher, since the copy that matters is pearl's.

Rehearsal without live animals — republish archived events onto the bus with original (speed-scaled) timing:

```powershell
.\.venv\Scripts\python.exe -m tools.replay_events --last 100 --speed 4
```

**Everything Python is run as `-m package.module` from the repo root, never as a
file path** (issue #123). This is not style: Python puts the *script's* directory
on `sys.path`, so `python vision/merle_daemon.py` dies with
`ModuleNotFoundError: No module named 'bus'`, while `python -m
vision.merle_daemon` puts the repo root there and every import resolves. It is
also what keeps the CWD-relative defaults (`weather.db`, `merle.db`,
`models/current.pt`) landing exactly where they always have.

### Services on pearl (systemd)

The pearl-resident processes run as systemd units at `/etc/systemd/system/` — the narrator (Marlin), `willard-weather.service` (Willard), `frame-archiver` (the still-shot filing clerk), and `merle-autodeploy` (the deploy watcher, issue #95) — enabled so they survive reboots. Mosquitto is its own stock `mosquitto.service`. All Merle units follow one pattern: run as the login user, `WorkingDirectory=` the repo checkout (`/home/todd/project-squirrel`), `ExecStart=` the repo venv's python (`venv/bin/python` on pearl — never system python) invoking a **module, not a file** (`-m narration.narrator`, `-m weatherpost.weather` — see the `-m` note above; `frame_archiver.py` is still a root module and needs no package prefix), `Restart=on-failure`, and `Environment=` lines carrying the process's env (each service sets `MERLE_MQTT=localhost:1883` — the broker is local; the narrator adds `MERLE_OLLAMA`, the weather post adds `MERLE_OWM_KEY`). **Every unit needs `Environment=PYTHONUNBUFFERED=1`**: the scripts log with bare `print()`, and under systemd stdout is a pipe, so Python block-buffers — `journalctl` shows nothing for hours and a TERM on stop discards the buffer. `WorkingDirectory` doubles as data placement: the weather post's `weather_history.json` lands in the repo dir because its default path is relative.

Pearl also serves the production MCC as `mcc-dashboard.service` (port 3000,
`next start` + a `fast-stop.conf` drop-in — see The MCC dashboard below and
`Servers/Pearl.md` for the ops detail).

Deploying new code — **merging the PR is the deploy** (issue #95):
`merle-autodeploy` (`Servers/autodeploy.sh`, a root loop-service that demotes
git/pnpm to todd) polls origin/main every 60s on pearl *and* merle, pulls
`--ff-only`, restarts that box's Merle units, and rebuilds + restarts the MCC
only when the merge touched `mcc/`. A failed MCC build never restarts the
dashboard (old code keeps serving); a dirty checkout is skipped loudly, never
clobbered; quiet polls log nothing, so `journalctl -u merle-autodeploy` reads
as a deploy history. Manual fallback with the watcher stopped: pull + restart
for the Python services (they run from source); the MCC via
`Servers/deploy-mcc.sh` (pull → install → build → restart, failing loudly at
each step) — `next start` serves the compiled `.next/`, so pull + restart is
*not* a deploy for it. Runbooks: `Servers/Pearl.md` (The deploy watcher) and
`Servers/Merle.md`.

Everything else: `systemctl status <unit>` for health, `journalctl -u <unit> -f` for live logs, `systemctl cat <unit>` to see (and crib from) an existing unit when adding the next service — new unit file, then `sudo systemctl daemon-reload && sudo systemctl enable --now <unit>`.

## Vision pipeline (current)

- **Camera**: Amcrest PoE at `192.168.1.102`, RTSP main stream over TCP (`OPENCV_FFMPEG_CAPTURE_OPTIONS=rtsp_transport;tcp|fflags;nobuffer|flags;low_delay` — TCP because UDP drops packets under load and smears frames; `nobuffer`/`low_delay` trim FFmpeg's demuxer/decoder buffering). Camera image settings live in its web UI; `cap.set()` is ignored for RTSP — **including `CAP_PROP_BUFFERSIZE`, which the FFmpeg backend silently ignores** (the set call returns `False`), so FFmpeg queues undrained frames without bound; low latency has to come from the reader draining the stream (see the frame sources section). Credentials come from the `MERLE_RTSP_PASS` env var (`MERLE_RTSP_USER` optional, defaults `admin`) — never committed.
- **Model**: YOLO26s fine-tuned on driveway data. The deployed weights live in `models/` (the promoted-weights shelf), loaded from `models/current.pt` by default or `MERLE_MODEL` if set — `runs/` is training scratch and is never loaded directly. Weights are not in git (`*.pt` ignored). See `models/README.md` for the promote-a-new-model steps. Inference at `imgsz=1920` so distant animals survive the downscale from 4K, and in **FP16** (`perception.QUANTIZE = 16`, passed as `quantize=` to every inference call site — live.py's track, the daemon source's track, and its warmup predict; `half=True` is the deprecated spelling and warns per call). FP16 exists for GPU *co-tenancy*: Ollama serves the narrator's LLM from the same GPU, and token generation is memory-bandwidth-bound (each token streams the full weight set from VRAM at ~20% compute — there is no utilization cap to raise), which starved FP32 inference from 15fps down to ~7 during narration. Halving the detector's memory traffic buys back headroom (solo speedup is modest, 19.7→17.4 ms/frame; the point is behavior under contention); accuracy impact is nil in practice, and ultralytics keeps FP32 on CPU. **Gotcha**: ultralytics locks precision when the first inference call builds the predictor — later calls can't change it, so any new first-touch call site must pass `quantize` too.
- **Important: the model is NMS-free** (end-to-end head). The ultralytics `iou=` argument is a no-op; duplicate boxes on one animal are possible and must be removed explicitly — `dedupe_boxes()` in `label_utils.py` (greedy, class-agnostic, IoU ≥ 0.7) on the labeling path, and `dedupe_detections()` in `perception.py` (same threshold and approach) on the live tracking path, inside `TrackMemory.ingest()` — a duplicate that reaches ByteTrack is confident enough to mint a parallel track riding the same animal (issue #24).
- **Tracking**: ByteTrack (`bytetrack_squirrel.yaml`). `DETECT_FLOOR=0.10` deliberately low — weak detections sustain existing tracks through confidence dips (tracks only *start* at ≥ 0.5). Lost tracks coast for 15 frames (~1s) to paper over single-frame misses and stay in memory as stitch/necromancer targets until `PRUNE_FRAMES` (900, ~60s — was 450/~30s until issue #74's gap analysis caught stationary feeders re-appearing on their own grave 44–52s after vanishing). `track_buffer: 180` (~12s of id memory) because stationary feeding animals flicker out of detection for many seconds — shorter buffers re-minted ids, inflating the census and (before the species-level event debounce) manufacturing phantom arrive/depart events.
- **Identity stitching** (`TrackMemory`, issue #22): when the flicker outlasts even the 12s buffer, ByteTrack re-mints the animal a new id anyway — one squirrel became several census "visitors", and its still-coasting ghost double-counted against the crowd threshold. The fix rides on the fixed camera: a brand-new id whose box lands with IoU ≥ `STITCH_IOU` (0.4) on a not-matched-this-frame track of the same voted species *is* that track come back, so it adopts the old identity (aliases map re-minted → canonical ids forever and chain-flatten; votes/box/census fold into one record; ingest is two-pass — known ids first — so a track matched in the current frame can never absorb a real neighbor). Everything downstream (sightings, census, crowd counts, drawn labels) sees one continuous id with no changes of its own. Accepted trade-off: a different animal taking the exact same spot within the ~60s prune window merges with its predecessor — rare, and the error it replaces ran 5x the other way. **Rover-era caveat**: stitching and every frame-denominated constant (`COAST_FRAMES`, `PRUNE_FRAMES`, `track_buffer`) assume the fixed 15fps camera; a moving ~60fps camera is a different tuning regime (IoU association weakens, windows silently shrink 4x if reused unscaled).
- **The necromancer pass** (`TrackMemory._necro_target`, issue #74 Phase 2.4): the stitch's distance-gated sibling, for the animal that *moved* while the tracker had lost it. When no stitch target exists, a brand-new id of species S born within reach of where an S track was lost (not matched this frame, not yet pruned) adopts that track's identity through the same alias mechanics — census, tenure, and events all inherit; ByteTrack's fresh id stays internal; nearest grave wins, and an overlapping corpse via the stitch always beats a nearer non-overlapping one. Reach scales with the dead track's box (a near animal is bigger and moves more pixels per body length), and is **two-tier**, straight from the fixture gap analysis: a *vanished* corpse (lost past the coast window) gets `NECRO_REACH` (2.0 body-lengths — it had time to wander); a *coasting* corpse gets only `NECRO_REACH_COASTING` (1.0 — this catches ByteTrack failing the IoU association mid-hop, the single biggest leak at 5 of 10 births, while a real second animal beside a briefly-missed one stays unmergeable). The same analysis caught stationary feeders re-appearing on their own grave 44–52s after vanishing, so `PRUNE_FRAMES` doubled to 900 (~60s). Measured on the crowd fixture (~5 squirrels + a cat scatter): census 19 → 9, canonical births 27 → 10, median track life 12.5s → 88s, arrival/departure events byte-identical — raw ByteTrack mints deliberately unchanged (the tracker isn't touched; `canonical_fragmentation` in `metrics()` is the number that reflects this layer, and the census target). Residual: a lone squirrel wandering 4–11 body-lengths between detections (43–138s gaps on the quiet fixture) is correctly refused — that's re-ID (the Big Chonk epic), not a spatial gate's job. Accepted trade-off, same direction as the stitch's: a different same-species animal claiming a dead one's patch within ~60s merges into its census identity; the overcount it replaces ran ~4x the other way.
- **Churn instrumentation** (issue #74, Phase 0): measure before tuning. `TrackMemory` counts every raw ByteTrack id minted (pre-dedupe — a duplicate box's id never reaches the bookkeeping, but the tracker did mint it and tracker churn is what's being measured), every canonical track born, every stitch, and every death (with matched-frame lifetime and whether the track was tenured or one-blink junk), plus per-track confidence min/mean/max — the instrument that shows the dips killing stationary feeders. `TrackMemory.metrics(fps)` rolls these into the numbers every Phase 2 tracking change is graded against, over a 10-minute window (`METRICS_WINDOW_FRAMES`, matching the issue's per-10-min success criteria): ids/minute, median death lifetime, mean concurrency, and the **fragmentation ratio** (ids minted ÷ mean concurrent animals; `None` on an empty pavement rather than dividing by ~zero). Lifecycle logging rides an injected `log` callable (a line per birth/stitch/death, not per frame — cheap enough to leave on): the daemon's RTSP source passes `print` unless `MERLE_TRACK_LOG=0`; the pure default stays silent for live.py and the tests.
- **Runtime provenance** (issue #74, Phase 0): whether the daemon is on the main stream or the sub-stream, and at what imgsz, must never be a mystery. `frames.rtsp_url()` is the ONE place the camera URL is built (the daemon source and the fixture recorder both call it; `subtype=0` = the Amcrest main stream) and returns a password-redacted twin for logs. Every frame source answers `provenance()` — source kind, redacted URL, native resolution (the capture's claim at open, overwritten by the first real frame: the honest source, which also tracks a mid-run camera settings change), imgsz, quantize, model path, class roster; the synthetic source reports the same shape with camera-only fields honestly null. The daemon logs one `[provenance]` line once frames flow and serves the dict on `/state` alongside `churn` (the metrics above at the measured loop fps; `None` for trackerless sources).
- **The fixture workbench** (issue #74, Phase 0.5): `record_fixture.py` captures a few minutes of the same main-stream feed the daemon watches (same URL construction, same FFmpeg options) to an MJPG `.avi` in `debug_frames/` — each frame an independent JPEG at q95, so no motion-compression artifacts stack on the camera's H.264 (deliberately dependency-free since there's no ffmpeg on PATH; the cost is disk, a few GB per fixture). `replay_fixture.py` runs a fixture through the FULL live stack — `model.track` (same conf/imgsz/quantize/tracker yaml) → `extract_detections` → `TrackMemory` → `SpeciesPresence` — with the clock derived from frame index ÷ fps, and prints the churn metrics, census, and events: the before/after ruler for every tracker change, on identical frames. Both are GPU/camera workbench tools (desk-tested, not in CI); the logic they drive is what `test_perception.py` covers. The event debounce itself moved from `Worker._species_presence` into `perception.SpeciesPresence` (same state machine, injected clock, canonical `ARRIVE_AFTER_S`/`DEPART_AFTER_S` defaults) precisely so the replay and the daemon can never drift; `merle_daemon` re-exports the constants as `ARRIVE_AFTER`/`DEPART_AFTER` for its tests.
- **Census tenure** (issue #24): a track only counts as a *visitor* after `CENSUS_AFTER_FRAMES` (30, ~2s matched) — one-blink junk tracks are still tracked, drawn, and recorded, they just never tally. Applied in two places with the same constant: `TrackMemory.seen` (live.py's run total) records a track only once tenured, and the `sightings`-backed queries (`storage.species_totals` / `census_by_day` / `day_hours`) take a `min_frames` filter the daemon passes. Filtering happens at **query time** — the `frames` column already existed — so `sightings` stays the honest raw record and historical days clean up retroactively. Stitched frames accumulate on the canonical track, so a re-mint mid-visit doesn't reset the visitor clock.
- **Classes**: training is **2-class — `0=squirrel, 1=turkey`**. Chipmunks are deliberately deferred (too small/rare for the overhead cam; they return when the rover gets a close-up camera). The **Roboflow dataset stays 3-class** (`chipmunk/squirrel/turkey`, chipmunk annotations preserved); `make_2class.py` converts each export (see Training flywheel). Within an export's lifetime the mapping is **append-only** — new classes get new trailing IDs, never reorder (every YOLO label file on disk depends on the mapping). *Note: the deployed `train-16` weights predate this and are still 3-class until the first 2-class model is promoted.*

## Training flywheel

1. **Hard-frame banking** (live.py): when a live track's confidence sits in the flicker band (0.15–0.50), the raw frame (never annotated — drawn boxes would poison training data) is saved to `hard_frames/` with a YOLO-format `.txt` sidecar of every current box, deduplicated. Low confidence means "unsure WHAT", not "unsure WHERE" — so the boxes are usually right and human labeling becomes review-and-nudge.
2. **Backfill/re-label** (`prelabel.py`): same idea for any folder of stills.
3. **Dedup** (`dedup.py`): near-duplicate frames moved aside by comparing *label geometry* (the camera is fixed, so whole-image hashing would call everything a duplicate). Frames containing rare classes are always kept.
4. **Review** in Roboflow (upload jpg+txt+classes.txt together; boxes arrive pre-drawn). New harvests go **100% to the train split** — random splitting of temporally-correlated frames leaks near-duplicates into valid and inflates metrics.
5. **Convert the export to 2-class** (`python -m tools.make_2class training/<export>`): strips chipmunk boxes (never renames them to squirrel — tiny ambiguous blobs would poison the class), renumbers squirrel→0 / turkey→1, drops chipmunk-*only* frames entirely (an unlabeled animal in a kept frame trains "background"), and keeps mixed frames with the chipmunk as intentional background. Writes a new `<export>_2class` folder and self-verifies (squirrel/turkey box counts unchanged, no out-of-range ids) — deletes its output and fails loudly on any mismatch. Roboflow's paid "Modify Classes" step, done free and reproducibly. The 69 chipmunk source images are tagged `chip-only`/`chip-mixed` in Roboflow for review and for the rover era.
6. **Train locally** (ultralytics) on the `_2class` folder. Augmentation happens online during training (fliplr, hsv, mosaic) — no pre-baked augmented dataset versions needed.
7. **Evaluate on the same ruler**: `yolo val` for both old and new weights against the *same* valid split before claiming improvement. Current baseline: **train-18** (first 2-class, deployed), 0.864 mAP50 / 0.838 recall on the 0707 2-class valid split — squirrel 0.908, turkey 0.820 (turkey's read is thin, 30 instances). *Class-set change caveat: train-18 is NOT same-ruler'd against 3-class train-16 (0.936 all-classes on 0705) — different split and class set, so the 2-class lineage starts fresh here.*

## MCC + daemon architecture (epic #1, in progress)

Design rules:

- **The daemon owns all state** (SQLite file, image folders). The MCC talks to it over HTTP only (via its `/daemon` proxy route, target `MERLE_DAEMON_URL`) — never touches the DB or filesystem directly. This keeps the daemon portable — it moved from same-box-as-the-MCC to "MCC on pearl, daemon on bluejay" with an env var, and a future mini-PC/Jetson is the same move.
- **Images and clips live on the filesystem; SQLite stores metadata and paths.** No blobs in the DB.
- **Video to the browser is MJPEG** (`/stream`, downscaled to `STREAM_WIDTH` (1080p) then encoded at q85) — browsers can't speak RTSP. Full-res single frames via `/snapshot`. WebRTC only if remote/multi-viewer needs appear. The downscale is load-bearing (issue #49): camera-native 4K JPEGs are ~1.7MB each, ~26MB/s at 15fps, and every byte transits the MCC's `/daemon` proxy per open tab.
- **Runtime is local-only.** The MCC is not hostable on Vercel (needs LAN camera + GPU). Remote access later = Tailscale.

### Storage (`storage.py`)

The daemon's persistent memory is a single SQLite file (stdlib `sqlite3`, no server, no extra deps — back it up by copying the file). Timestamps are ISO-8601 strings passed in by the caller, never generated inside storage, so every function is pure and deterministic for tests. Three tables:

- **`sightings`** — one row per tracked animal, keyed by `(session_id, track_id)`. ByteTrack ids only mean something within a single daemon run (they reset on restart), so `session_id` (the run identifier) disambiguates. Accumulates `frames`, `first/last_seen`, and running `max_conf`; `species` holds the latest voted class.
- **`events`** — notable moments (hard frame saved, crowd snapshot, clip recorded). `kind` classifies; `details` is a free-form JSON blob so new event types need no schema change. Per the filesystem rule, an event records a frame's *path*, not the frame.
- **`training_runs`** — one row per training round for the dashboard's metrics-over-time panel. Headline `map50`/`recall`/`map50_95` are columns; per-class detail is JSON in `metrics`. Keyed by `run_name` (PK) so seeding is idempotent. Seeded with train-15/train-16 (3-class, 0705) and train-18 (first 2-class, 0707) — the class-set change means train-18 isn't on the same ruler as the earlier rows.

Pure-logic tests in `test_storage.py` run in CI (stdlib-only, no model/camera). `connect()` enables WAL (on-disk only) so the perception thread can write while request threads read.

### The daemon (`merle_daemon.py`) and frame sources (`frames.py`)

The daemon is a FastAPI app. A background `Worker` thread runs the perception loop — pull a frame + its detections from a **frame source**, annotate, JPEG-encode (twice: full-res for `/snapshot`, downscaled to `STREAM_WIDTH` for `/stream`), publish to a lock-guarded `SharedState` (bumping `seq` per frame), and persist sightings/events — while request handlers read that state. Endpoints:

- `GET /state` — session id, control flags, the model's class roster (`species`, in class-id order), live counts/tracks/fps, run totals, recent events, plus `provenance` (what the source is actually connected to) and `churn` (the tracker fragmentation metrics; `None` for trackerless sources) — see Churn instrumentation and Runtime provenance under Vision pipeline. The roster is advertised so the dashboard can render a fixed row per class instead of only the species currently counted — the daemon stays the single source of truth for what the model can see.
- `GET /history?days=N` — the Station Records feed (one fetch, slow-poll — never the 1s loop; N clamped 1–90): per-day distinct-visitor census (`storage.census_by_day` — counted on the day a track *first* appeared, padded so quiet days exist), hard-frame counts per day (from `hard_frames/` file mtimes — live.py banks there with no DB record, so mtimes are the one honest source), and the `training_runs` table. `GET /history/day?day=YYYY-MM-DD` — that day's arrivals bucketed by hour.
- `GET /stream` — the annotated feed as `multipart/x-mixed-replace` MJPEG (an `<img src>` in the browser; no player). Async + `is_disconnected()` so a closed tab frees the generator. Serves the **downscaled** copy, and only frames the worker hasn't already sent to that client (`next_stream_part` gates on `SharedState.seq`) — a stood-down station streams ~zero bytes instead of re-sending the frozen frame at 15fps forever, which is what had `next-server` on pearl relaying 26MB/s at ~50% CPU while nothing moved (issue #49). A fresh client always gets the current frame immediately, so a tab opened during stand-down shows the last frame, not a broken image.
- `GET /snapshot` — the latest annotated frame, one JPEG.
- `POST /control` — `start`/`stop` the loop, `record_on`/`record_off`, `set_crowd_threshold`. Recording writes the **annotated** stream to `debug_frames/clip_*.mp4` (mp4v) from the worker thread — for watching/sharing a moment. (live.py's `V` key still writes *raw* clips, which is what you sample for training stills.) The writer is closed cleanly on `record_off`, lost feed, or shutdown, and a `clip_recorded` event is logged.

The **frame source** (`frames.py`) is the seam that keeps perception swappable. `FrameSource.read()` returns `(frame, [Detection, ...])` (each `Detection` has a `coasting` flag); the daemon's annotation/encoding/persistence has one implementation regardless of source, so the synthetic and real paths can't drift. Selected by `MERLE_SOURCE` — **default `camera`** (the real feed, so `uvicorn vision.merle_daemon:app` just works); set `MERLE_SOURCE=synthetic` for the camera-free world. The source is built lazily **at server startup** (the module-level app passes a factory, not an instance), so importing the daemon never opens the camera or loads the model — that's what keeps tests/CI camera-free even with a camera default. Sources:

- `SyntheticFrameSource` — camera-free (a couple of squirrels + a visiting chipmunk, motion a pure function of a frame counter so it's deterministic). Powers the tests, local dev, and MCC frontend work with no camera or model. Imports no ultralytics, so CI stays light.
- `RTSPFrameSource` — the real Amcrest feed through the same model + tracker as live.py, sharing `perception.py`. Reads the password from `MERLE_RTSP_PASS`, the model from `MERLE_MODEL`; lazy-imports ultralytics so the daemon stays importable without torch. Coasting tracks are flagged (drawn grey, and not counted as matched frames in `sightings`). **Latency** (issue #29): FFmpeg queues an RTSP stream without bound (`CAP_PROP_BUFFERSIZE` is a no-op on that backend), and a consumer reading at exactly the camera's 15fps can never drain a backlog once one forms — model load + first-inference CUDA warmup with the stream already open used to bake in a permanent ~6-7s delay. Two structural answers: the model is loaded **and warmed up (one dummy `predict`) before the RTSP connection opens**, and a `FreshestFrameReader` thread drains `cap.read()` continuously from the moment it opens, keeping only the newest frame. `read()` blocks until a frame *newer than the last consumed* arrives (never re-serving a frame), timing out to `None` after `READ_TIMEOUT`; the camera paces the reader once caught up, and any mid-run stall (clip writing, a slow frame) drains at decode speed (~11ms/frame for 4K H.264) instead of ratcheting the delay up. Expected camera→dashboard latency is ~0.5–1s, the practical floor for RTSP + H.264. **Self-healing**: a failed read (camera restarted after a settings change, network blip) triggers a throttled reconnect (`RECONNECT_INTERVAL`) inside the reader thread instead of freezing forever. The worker treats a `None` frame as transient — it flags `live.signal=false` and keeps polling rather than exiting, so a dropped stream never kills perception (it used to `break`, which froze the feed until a manual restart). The dashboard shows a distinct "reconnecting" veil while `signal` is false. `test_frames.py` locks down the reader's contract (newest-frame, never-twice, timeout-as-no-signal, reconnect) with a fake capture, camera-free.

### Shared perception (`perception.py`)

The tracker bookkeeping and box-drawing are shared by **both** live.py and `RTSPFrameSource` so they can never drift (a real bug we already hit once between two files). It holds the detector/tracker config (`DETECT_FLOOR`, `IMGSZ`, tracker yaml, coasting windows, the hard-example flicker band, class colors), `extract_detections()` (pull ID-carrying detections out of a `model.track` result, ignoring the ID-less leak), `TrackMemory` (the coast/prune/vote bookkeeping + the `seen` run-total census), and `draw_tracks()` (boxes + labels; `scale=1.0` is tuned for 4K — the daemon passes `frame_height/2160`). Imports no ultralytics, so `test_perception.py` covers the bookkeeping camera-free. live.py is now a thin consumer of it; the model inference and RTSP capture (the untestable-in-CI parts) live in `RTSPFrameSource` and are verified against the real camera.

`test_daemon.py` drives the app via FastAPI `TestClient` against the synthetic source + an in-memory DB (no camera/model), so the full HTTP surface runs in CI. The live MJPEG stream can't be consumed cleanly through TestClient (infinite multipart blocks it), so the frame framing is unit-tested via `mjpeg_frame()`, the downscale via `encode_stream_jpeg()`, the new-frames-only gating via `next_stream_part()`, and the stream is verified end-to-end by running uvicorn.

Deps for the daemon are in `requirements.txt` (fastapi, uvicorn, opencv, numpy); ultralytics/torch stay out (installed per-machine, GPU-specific). CI installs headless opencv.

### The event bus (Mosquitto + `bus.py`)

Live event distribution rides an MQTT broker (Mosquitto), decoupling producers from consumers: the daemon publishes, and narrators/dashboards/future rover processes subscribe without anyone knowing who else exists. **SQLite stays the durable archive; the bus is the live transport** — a message dropped while the broker is down is never a lost record, just a moment nobody narrated. The daemon publishes fire-and-forget (QoS 0, auto-reconnect in the background) and runs identically with no broker at all.

- **Broker**: Mosquitto on **pearl** (`192.168.1.64`, the always-on Ubuntu box), configured at `pearl:/etc/mosquitto/conf.d/squirrel.conf`. Nothing broker-shaped runs on bluejay anymore — Mosquitto is uninstalled there, and the repo-root `mosquitto.conf` is just a pointer to pearl. Two listeners: `1883` plain MQTT (Python processes, paho-mqtt) and `9001` WebSockets (the browser — Next.js rewrites can't proxy WebSockets, so the dashboard connects to the broker directly). Anonymous auth; the bus never leaves the LAN.
- **Topics** (constants in `bus.py`, so a typo'd string can't split the system): `driveway/events` (daemon → world, one JSON event each), `narration/lines` (narrators → world — and mention-listening narrators subscribe to it too, issue #80; see The narrator below), `narration/journal/<mqtt_id>` (narrator → world, retained: each narrator's field journal window — its last 50 spoken lines, republished whole on every new line. Per-narrator since issue #80: a single shared retained topic republished whole by each narrator meant every republish clobbered the other's window, so a fresh tab rehydrated with only whoever spoke last; the dashboard subscribes `narration/journal/+` and merges. The topic name comes from `bus.narration_journal_topic(mqtt_id)`; see The narrator below), `narrators/<id>/status` (retained `online`/`offline` presence; `offline` is each narrator's MQTT Last Will, so a crash flips the dashboard lamp with no cleanup code), and the retained weather set — `weather/current` / `weather/forecast` / `weather/history` / `weather/report` (Willard's LLM-narrated on-air segment, issue #45; absent entirely when the weather service runs without an Ollama) plus `weather/status` presence (`online`/`offline` with the same Last-Will contract as the narrators, but in the weather namespace so the Field Journal's narrator wildcard never picks up a reporter; see The weather post below).
- **Retained topics carry state, event topics carry moments.** `EventPublisher.publish(topic, payload, retain=False)` defaults off: replaying a stale arrival to every new subscriber would be worse than dropping it. The weather topics set `retain=True` because weather *is* state — the broker hands a late joiner (fresh dashboard tab, restarted narrator) the latest report instantly, which is the entire delivery mechanism for weather (no HTTP path, no poll loop, no daemon involvement).
- **Event kinds on the bus** (same shape as the `events` table: `ts`, `kind`, `details`): `arrival` and `departure`, plus `crowd_snapshot` and `clip_recorded`. Every event goes to SQLite and the bus through one `Worker._event()` helper so the two can't diverge.
- **Event still shots** (issue #90): `arrival`/`departure`/`crowd_snapshot` each carry a `frame_id` in `details` (minted by `merle_daemon.mint_frame_id` — session + compact timestamp + kind + a per-session counter, filesystem-safe by construction; the counter exists because two events can fire on the same frame), and the daemon publishes the frame the event fired on to `driveway/frames/<frame_id>/full` (the annotated stream-downscaled JPEG — the copy already encoded every loop for `/stream`, so near-zero extra cost) and `.../thumb` (~320px, `THUMB_WIDTH` — the daemon encodes it because it owns cv2; consumers stay image-dep-free). **Raw JPEG bytes, not JSON** — `EventPublisher.publish_bytes()`; topic helpers `frame_topic()`/`frame_topic_parts()` live in `bus.py` like everything else. Fire-and-forget and non-retained, same ethos as events: a dropped frame is a moment nobody archived (the event row still exists, id and all), never a lost record. A departure's frame shows the scene ~12s after the animal left — the debounce is when the event honestly happened, and empty pavement is the honest picture of a departure. `clip_recorded` carries no frame (it already records a clip path). SQLite stores only the id, per the no-blobs rule; the bytes land on pearl (next entry).
- **The frame archive** (`frame_archiver.py`, issue #90): a tiny pearl-resident service — flat script, the `weather.py` conventions — that subscribes `driveway/frames/#` and writes the bytes to `<MERLE_FRAMES_DIR>/<frame_id>.jpg` / `<frame_id>.thumb.jpg` (default `frames/` under the unit's `WorkingDirectory`). On pearl and not behind the daemon's HTTP surface **because daemon-down is the steady state** for the 24/7 MCC: the Field Journal rehydrates from retained topics while bluejay sleeps, and its thumbnails must survive the same nap — the MCC's `/frames` route reads this folder locally. Filenames derive from the topic through `bus.frame_topic_parts()` plus a strict `[A-Za-z0-9_-]` id allowlist (the path-traversal guard — pure, tested; never trust the wire). Retention is a rolling window: files older than `MERLE_FRAMES_KEEP_DAYS` (default 14) are pruned hourly (`prune_selection()` is pure with injected clock/listing); the journal shows a quiet placeholder for anything pruned. Requires `MERLE_MQTT` (no default, the `bus.py` convention). Runs as the `frame-archiver` unit (see `Servers/Pearl.md`); when the USB NAS arrives, migration is repointing `MERLE_FRAMES_DIR` (and the MCC unit's copy of it) at the mount — nothing else changes.
- **Arrival/departure are SPECIES-level and debounced** (`perception.SpeciesPresence`, driven by the Worker), not track-level. ByteTrack re-mints a new track id when it loses a stationary animal past its buffer and re-acquires it — track-level events turned every one of those into a phantom departure+arrival pair (a real spam problem on the live feed). Instead the Worker watches per-species counts of *matched* (non-coasting) tracks: a count increase must hold `ARRIVE_AFTER` (2s) to fire `arrival {species, count}`, a decrease must hold `DEPART_AFTER` (12s — longer than any realistic churn gap) to fire `departure {species, count}`; any wobble back resets the clock, so id churn produces zero events. `duration_s` rides on a departure only when the count hits 0 (the species' whole visit — with several present you can't know which individual left). The per-track `sightings` table and Run Census get their id stability from `TrackMemory`'s identity stitching (see Tracking above) rather than from this debounce. **Every count the daemon emits — species presence, the crowd trigger and its `crowd_snapshot` payload, `/state`'s `counts` — tallies *matched* (non-coasting) tracks only**; a coasting ghost is a briefly-lost track, not another animal. Coasting boxes still draw grey and still ride `/state`'s `tracks` list — display only.
- `MERLE_MQTT` (`host` or `host:port`) points every Python bus process (daemon, narrator, replay) at the broker, and is **required — there is no default**. `bus.py` used to fall back to `localhost:1883`; with the broker on another machine that silent default produced a daemon that looked healthy while publishing into the void, so `broker_address()` now raises at startup when the var is missing. It binds the two machines together: on bluejay it's a User-level env var (`192.168.1.64:1883`, same treatment as `MERLE_RTSP_PASS`, and `start-merle.ps1` pins it as a belt-and-suspenders default); on pearl the narrator sets it explicitly to its local broker.
- The browser side is `NEXT_PUBLIC_MERLE_MQTT_WS` in `mcc/.env.local` (not in git): `ws://192.168.1.64:9001`. Without it `busUrl()` falls back to the page's own hostname on port 9001 — a relic of the broker-on-bluejay era that survives as a guard for any future same-host broker, including its `localhost`→`127.0.0.1` pin (Windows browsers resolve `localhost` to IPv6 `::1` first, and the WebSocket to Mosquitto over `::1` connects at TCP level but never completes the MQTT handshake (`connack timeout`) — the same trap the `/daemon` proxy route dodges).

### The narrator (`narrator.py`)

The scene narrator: **one voice per process, LLM prose with template fallback, real pacing** — and since issue #80 the show is a two-hander: **Marlin** (the studio host, on pearl next to the broker) and **Jim** (the field correspondent, on merle — the Pi 5 that will be the rover's brain, positioned there on purpose). Each narrator is its own `narrator.py` process subscribing to `driveway/events` and publishing spoken-style lines to `narration/lines` — it never plays audio itself; a consumer (the dashboard's TTS) speaks. A narrator needs neither the GPU nor the camera, just the bus (the LLM runs remotely on bluejay's GPU). Design decisions:

- **Persona vs bible**: a persona YAML (`personas/marlin.yaml`, `personas/jim.yaml` — name, `mqtt_id`, `tts_voice` hint, `personality_prompt`, pacing knobs) is one voice; `character_bible.yaml` is shared world canon (seed-pile location, Big Chonk lore). Kept separate from day one so multiple narrators never need untangling — Jim reads the same bible and already knows where the seed pile is. Since issue #88 the knobs encode the *roles*: Jim carries announcer levels (0.9 chattiness / 0.4 threshold / 20s cooldown — the field man is the first voice on any new development) while Marlin defers the raw play-by-play entirely (see Announcer roles below). Each process still carries its own Editor min-interval, so the ceiling stays one line per narrator per 30s.
- **One pacing gate** (`worth_speaking()`): cooldown first, then per-kind interest scaled by the persona's `chattiness` against its `interest_threshold`. Silence is most of the show — with the default knobs, most events pass unremarked. The gate is identical across prose tiers.
- **The editor's desk** (`Editor`, issue #74): the narrator-side defense against a noisy bus. Tracker id churn on a steady feeding scene floods `driveway/events` with phantom departure+arrival pairs, and v1 handed each one to the talent — a nonstop Ollama-call storm on bluejay's GPU. The Editor sits between the wire and the roster (scene-level state, one desk for the whole show — personas keep their own cooldown/interest knobs downstream, unchanged) with two defenses. **Hysteresis**: an arrival/departure becomes narratable only once the species' reported count has held at its new value for `MERLE_NARRATE_STABLE_S` (default 20s); a departure undone by an arrival of the same species settles back to the old count and cancels outright — the id-churn signature's explicit outcome is silence, never a hedged line. **Rate limit + burst collapse**: the desk hands the talent at most one event per `MERLE_NARRATE_MIN_INTERVAL_S` (default 30s — a hard ≤2 generations/min ceiling regardless of upstream behavior; deliberately independent of the persona cooldown, which tunes character while this caps cost). Whatever stabilized or happened during a cooldown collapses into ONE `scene_update` event summarizing where the scene ended up — a full `{counts}` snapshot including unchanged species (the scene is the story, not the diff), with quiet-pavement phrasing when empty — which then narrates through the normal gate and prose tiers (its own `INTEREST` entry, templates, and `event_summary` branch; a lone ripened change still narrates as the real arrival/departure, `duration_s` and all). Non-presence moments (`crowd_snapshot`, `clip_recorded`, unknown kinds) skip hysteresis — the daemon already debounces them — but ride the same slot, keeping only the latest per kind. Dropped play-by-play is fine: SQLite has the record; the bus is live transport. The plumbing changed with it: `on_message` now only parses and enqueues (stamped with arrival time, so hysteresis judges when a change was reported, not when the loop read it), and the main thread runs a ~1s pacing loop (`EDITOR_TICK_S`) that drains the queue through the Editor and owns the blocking Ollama call — which therefore no longer runs on paho's network thread, retiring the old "timeout must stay under the MQTT keepalive" constraint. Editor logic is pure (injected clock, no I/O) and covered in `test_narrator.py`.
- **Two prose tiers, one swap point** (`generate()`, issues #9/#20). **Tier 2 (LLM)**: when `MERLE_OLLAMA` (`host` or `host:port`, port defaults to 11434) is set, each line is synthesized by an Ollama server — `MERLE_OLLAMA_MODEL` picks the model (default: `OLLAMA_DEFAULT_MODEL` in `narrator.py`, currently `gemma3:12b`); model choice is deployment config, not persona, because it tracks what's pulled on the Ollama host. The system prompt is the persona's `personality_prompt` (Marlin's is an exaggerated Marlin Perkins / Wild Kingdom homage) + bible canon + output rules that live in code (`LINE_RULES`) so persona files stay pure character; the user prompt (`build_user_prompt()`) is an optional recent-lines memory block over a dry factual `event_summary()` — the standing hook for extra bus context, which now carries the weather sentence. **Tier 1 (templates)**: Mad-Libs fill-ins from the event + bible, and the fallback whenever the LLM is unset, unreachable, times out, or returns nothing — the show never goes silent. `MERLE_OLLAMA` unset is the kill switch and keeps a bare checkout working.
- **Weather context** (issue #26): the narrator subscribes to the retained `weather/current` on its existing MQTT connection and caches the latest payload; when a fresh report is in hand the user prompt gains a labeled current-conditions paragraph — one deliberately dry sentence ("It is 78F with a light breeze under overcast clouds.") plus a one-line usage nudge (`WEATHER_GUIDANCE`); all flavor stays in the persona prompt. The paragraph-plus-nudge shape is desk-tested necessity, not decoration: gemma3:12b ignored the same sentence appended bare to the event summary 4 generations out of 4, and wove it in naturally 4 out of 4 once labeled and nudged. A report older than `WEATHER_STALE_S` (30 min = 3 missed 10-minute polls) is ignored, so a dead weather service degrades to exactly the no-weather prompt — never to narrating yesterday's rain. Weather rides the LLM prompt only (templates stay v1) and is context, never a speaking trigger — the pacing gate doesn't know it exists.
- **Recent-lines memory** (issue #28): each `Narrator` keeps an in-process deque of its last 10 spoken lines (`MEMORY_LINES`; ~400 extra prompt tokens, no knob needed) as `(ts, event_kind, text)`, captured in `speak()` right where the bus payload is built so memory and the bus can never disagree — template fallback lines count (they went on air), a failed LLM call leaves no trace. The prompt gains a "recently on air" block above the event summary, ages rendered with the `human_duration()` vocabulary; the variety/continuity guidance travels in the block header (not `LINE_RULES`), so an empty memory — including every restart, which deliberately starts blank — degrades the prompt to exactly the pre-memory shape. The deque is deliberately separate from the journal window below: the journal is the show's record for the dashboard, the deque is prompt seasoning, and seeding one from the other would couple what the audience sees to what the model is told.
- **Mention triggers** (issue #80): the opt-in persona knob `answers_to` (a list of names; absent/empty = the machinery is entirely off and the process is byte-identical to the one-voice era — the #26/#28 degradation convention). When set, the narrator additionally subscribes to `narration/lines`: a line whose `mqtt_id` differs from its own and whose text matches any `answers_to` name (case-insensitive, word-boundary — "Jimmy" is not a mention of Jim; `mentions_name()`/`colleague_mention()`, pure and tested) becomes a synthetic `kind: "colleague_mention"` event with `details: {narrator, mqtt_id, text}`. `INTEREST["colleague_mention"]` is 0.95 — the marquee trigger clears any sane threshold — but the event rides the existing Editor as a non-presence *moment*, so it respects the narration rate limit like everything else; if a busy burst collapses it into a `scene_update`, that's showbiz. `event_summary()` quotes the colleague's line and cues the follow-up; a Tier-1 `colleague_mention` template pair keeps the acknowledgment airing when the LLM is down. **Never a direct reply channel** — everything rides the bus. Loop safety with both narrators listening (issue #88 cashed in #80's "revisit" note): a narrator ignores lines carrying its own `mqtt_id`, and a line whose `event_kind` is itself `colleague_mention` never triggers — **never reply to a reply** — so a chain is bounded at announcement → one follow-up per listener. A standing instruction still reaches the colleague through the heard-lines memory (Jim carries out Marlin's assignment in his *next* report); it just can't trigger.
- **Announcer roles** (issue #88): with both narrators live, both gates cleared the same event — a double announcement seconds apart. The fix is a deterministic role split, not coordination: the opt-in persona knob `defer_events_to` (a list of colleague `mqtt_id`s; Marlin carries `[jim]`) subscribes the narrator to the existing retained `narrators/+/status` presence (parsed Python-side by `bus.narrator_status_id()` and handled *before* the JSON path — status payloads are raw strings), and `wants()` refuses raw events while any listed colleague reads `online` (`defers_event()`, pure): the field has the announcements. Mention follow-ups are never deferred — reacting to the field report *is* the studio's beat (Marlin also carries `answers_to: [Marlin]`, and Jim's prose tosses to the studio by name). When every listed colleague is dark (`covering_field()`), announcements resume through the normal gate math and the LLM prompt gains one dry `COVERING_NOTE` paragraph — the persona hangs the coffee-break ribbing on it — so the show keeps running when merle/the rover is off, with the Last Will flipping the roles within seconds in both directions. Colleague presence lands in `Narrator.colleagues` from paho's network thread (per-key dict assignment, the `latest_weather` atomicity pattern).
- **Broadcast-context memory** (issue #80): alongside the own-lines deque, a mention-listening narrator keeps a small bounded record (`HEARD_LINES`, 5) of recent lines *heard* from other narrators — `Narrator.hear()`, a plain tuple rebuilt per line so paho's network thread can write while the pacing loop reads (atomic assignment, the `latest_weather` pattern, no lock). The prompt gains a "recently on the broadcast, other voices" block (`heard_block()`) between the own-lines memory and the event summary, with don't-repeat/build-on-it guidance riding the block header — this is what makes a mention follow-up context-aware and non-redundant against what Marlin already said. Empty block (every persona without `answers_to`, and Jim before anyone speaks) ⇒ prompt byte-identical to today, same as the memory/weather blocks.
- **The field journal window** (issue #58, per-narrator since #80): the last `JOURNAL_LINES` (50, matching the dashboard's `JOURNAL_LIMIT`) published lines, oldest first, published retained to `narration/journal/<mqtt_id>` (`bus.narration_journal_topic()`) as `{lines: [...]}` and republished whole on every new line — the weather/history pattern, so a fresh dashboard tab rehydrates the Field Journal straight from the broker with no HTTP path. Persisted to a flat JSON file (`MERLE_NARRATION_JOURNAL`, default `narration_journal.json` in the unit's `WorkingDirectory` on pearl — the `MERLE_WEATHER_HISTORY` convention) so a narrator restart doesn't blank the journal either; missing and corrupt files both mean "start over" (the journal is the show's recent record, not a record of record — SQLite keeps the events). The window is also republished on every (re)connect: the file, not the broker, is the source of truth, which heals the retained copy after a broker restart (Mosquitto only keeps retained state across restarts with persistence configured). File first, then the retained publish, so a crash between the two loses a broadcast, never the record. Deliberately a flat file, not SQLite — bounded, rewritten whole, refetchable (the `weather_history.json` argument). `narration/lines` itself is unchanged and stays non-retained: it carries moments (TTS speaks each once); the journal carries state. Continuity is the model's job, not code's: the block gives timestamps and event kinds and lets Marlin *speculate* about returning individuals in character — real identity stitching across the bus is issue #22's problem. Tier 1 gets only the cheap version: remember the last template index per event kind and re-roll once on an immediate repeat.
- **The Ollama call is blocking and non-streaming** (`stream: false`, one `POST /api/generate`): the bus contract is one JSON object per spoken line and the dashboard TTS speaks whole lines, so nothing downstream could use a token stream. Since issue #74 it runs on the main pacing loop, not paho's network thread, so `OLLAMA_TIMEOUT_S` (30s; desk-tested ~8s warm, ~15s cold-load) no longer has to stay under the MQTT keepalive; events arriving mid-generation wait in the queue and mostly collapse at the Editor afterwards, which is the desired pacing anyway. `keep_alive: 30m` keeps the model resident between sparse events. Rejected alternative: streaming over MQTT would have needed a new topic contract plus reassembly in the dashboard for no audible benefit. The `Ollama` class splits into a generic `complete(system, prompt, num_predict, temperature)` core and the narrator-shaped `narrate()` wrapper — `complete()` is shared with the weather post's Willard segment (issue #45), so there is exactly one Ollama client in the codebase.
- **Embedded producer**: the producer/orchestrator lives inside `narrator.py` as `Producer`, deliberately shaped around a *roster* (a set of voices) even though the roster is one — `cast(event)` picks who speaks, so solo-beat/banter-beat casting slots in later without a rewrite. What narrators do when a standalone producer is absent is a known future question, deferred to the promotion issue.
- **Still-shot pass-through** (issue #90): a spoken line caused by an event whose `details` carry a `frame_id` copies the id onto the `narration/lines` payload — and therefore into the journal window — so the Field Journal can show what he was narrating (the event-time frame is the honest moment: by speak time, 20–50s of hysteresis and rate limit later, the scene has changed). The Editor's burst collapse carries the **newest ripened change's** `frame_id` onto the `scene_update` (best-effort; a moments-only collapse carries none); a lone ripened change or moment passes its own through untouched. `colleague_mention` lines never carry one — their details are the colleague's line, not a moment on the pavement. Absent `frame_id` ⇒ payloads byte-identical to today (the #26/#28 degradation convention): old journal files, stale retained windows, and template-tier lines all keep parsing.
- Pure logic (gate, scoring, templates, persona loading, LLM config/prompt-building/sanitation/fallback) is covered by `test_narrator.py`; the MQTT plumbing and the live Ollama call are desk-tested against the real services.
- **Deployment**: Marlin runs on pearl as `narrator-marlin` (see `Servers/Pearl.md`), Jim on merle as `narrator-jim` (see `Servers/Merle.md` — the full bare-Pi-to-enabled-unit runbook; the Pi installs only the narrator subset, `paho-mqtt` + `pyyaml`, never the vision stack). Both units set `MERLE_OLLAMA=192.168.1.79:11434` (bluejay). Ollama on bluejay binds the LAN address; if Windows Firewall blocks inbound 11434, a narrator silently degrades to templates — check the log's startup "narration tier" line and per-call `[ollama]` fallback messages when lines sound suspiciously Mad-Libs.
- **Rehearsal**: `replay_events.py` republishes archived SQLite events onto the bus with original relative timing (`--speed`, `--kinds`, long silences clamped by `--max-gap`) — the narrator can't tell the difference, which is the point of the bus.

### The weather post (`weather.py`)

Conditions at the seed pile (issue #25, re-instrumented in #51): a pearl-resident service (needs neither GPU nor camera, just the bus, the gateway, and internet) with **two sources and one truth**. The driveway's own station — an **Ecowitt GW2000B gateway + WH90 sensor array** on the LAN — is the system of truth for everything it can measure: polled every 60 s over its unauthenticated local HTTP JSON API (`/get_livedata_info`), it supplies temp, feels-like, dew point, VPD, humidity, wind speed/gust/max-daily-gust/direction, solar, UV, the piezo rain suite (raining-now bit, rate, event/day/week/month/year totals), barometric pressure (abs + rel, from the gateway's internal WH25), indoor temp/humidity, and the WH90's battery/voltage (radio signal comes from `/get_sensors_info`, fetched on the slower cadence). **OpenWeather's classic free APIs keep exactly two jobs the station can't do**: the 5-day/3-hour forecast (hourly) and the sky garnish — condition text, sunrise/sunset — from a demoted 10-minute `/weather` call whose measured fields are discarded. Measured fields **never** fall back to OpenWeather: a gateway outage is a skipped poll and a chart gap, never a silent source switch. Publishes the retained `weather/*` topics; consumers are the dashboard's Weather Post panel + station view and the narrator's prompt context (issue #26, `weather/current` only).

- **Config** (env, the MERLE_MQTT conventions): `MERLE_ECOWITT` (gateway `host` or `host:port`, e.g. `192.168.1.210`) and `MERLE_OWM_KEY` are both **required, no default** — a service that can't name its station (or would poll 401s while looking healthy) fails at startup instead. `MERLE_WEATHER_LOC` takes `zip`, `zip,CC`, or `lat,lon` (default `49001,US` — the station's home turf; lat/lon is recognized because both halves parse as floats, which a country code never does). `MERLE_WEATHER_HISTORY` overrides the history-file path.
- **Gateway parsing** (`parse_station`): the GW2000B keys live data by opaque ids (`0x02` temp, `0x0E` rain rate, `"3"` feels-like…) and wraps every number in a unit-suffixed string (`"29.24 inHg"`, `"46%"`); `STATION_COMMON_IDS`/`STATION_RAIN_IDS` are the Rosetta stone (verified against the real device) and `station_num()` strips the units. A response where nothing parses is treated as a failed poll, not a report of Nones. The `weather/current` payload is a **superset of the pre-#51 shape** — every old field name survives with its old meaning, so the narrator and any open dashboard tab ride through a deploy; `ts` is our fetch time (the station has no observation clock).
- **Timestamps on the weather topics are unix epoch seconds**, not the ISO strings used elsewhere: the dashboard formats them in the viewer's locale and the narrator compares against `time.time()` for staleness — nobody wants to parse.
- **Forecast points carry precipitation** (issue #56, un-summed in #65): besides `pop` (probability 0..1), each `weather/forecast` point has `rain_rate_inhr` — the step's `rain["3h"]` mm converted to an **average in/hr over the 3-hour window ending at the point's `ts`**, sharing the observed piezo trail's unit — and `snow_3h_in`, the step's `snow["3h"]` mm as plain inches (its own strip charts it per step; no station instrument shares a scale with snow, the piezo is snow-blind). OWM omits both objects entirely on dry steps; the fields default to 0, not null: a step the forecast left dry is a real forecast of zero, unlike a station gap (which stays an honest hole in the observed trail).
- **The 48h rolling window** is the live state the service owns: polls run every 60 s but the trail keeps **5-minute resolution** (`should_record` gates appends; ~576 points of the 12 charted fields, tens of KB), pruned by age *measured from the newest point* (deterministic for tests), republished whole to `weather/history` on every append, and persisted to a JSON file on pearl so restarts don't blank the dashboard's observed trail. Old history files load fine — pre-#51 points simply lack the new fields, which chart as gaps that heal within 48h. Deliberately a flat file, not SQLite — bounded, rewritten whole, refetchable.
- **The seasonal archive** (`weather_archive.py`, issue #105) is the permanent record beside that window, and the two are not alternatives: every point that passes `should_record()` is appended to the window **and** recorded to a SQLite `observations` table that never prunes. The window stays the retained-bus payload (a fresh tab rehydrates its trail from the broker with no fetch — the contract that keeps weather off HTTP); the archive is a second consumer of the same tick, read over HTTP only when someone asks for the deep past. A failed archive write is logged and skipped — one missing point is a gap the chart already draws honestly, and never worth killing a service whose job is the bus. Startup backfills the window into the archive (idempotent on every restart, not just the first, so a fresh archive inherits up to 48 h of real readings). Config is `MERLE_WEATHER_DB` (default `weather.db` under `WorkingDirectory`), and an unopenable path fails at launch rather than publishing weather for a week while recording none of it. Four decisions worth the ink:
  - **On pearl, not in `merle.db`.** `merle.db` lives on bluejay next to the daemon, and daemon-down is the *steady state* for the 24/7 dashboard — archiving through the daemon's HTTP surface would punch holes in the record exactly when bluejay naps, which is most of the time. `weather.py` runs on pearl 24/7, so the rows live where the writer lives. This is `frame_archiver.py`'s reasoning applied to a different kind of byte: pearl-resident writer, pearl-local store, an MCC route reading off local disk, never a `/daemon/*` endpoint. `storage.py` and `merle.db` are untouched.
  - **5-minute rows, not an hourly rollup.** Issue #72 proposed hourly because that's all its forecast scorer needs, but 5-minute data is a strict superset — hourly means, min/max, and vector-averaged wind all derive from it at read time — and it's what the station already records and the charts already draw. Hourly would throw away resolution we have and cap any pressure curve at a day's zoom. ~105 k rows/year, which SQLite does not notice. **#72's hourly rollup is therefore a query, not a store**, and these rows are the `observations` table its scorer joins forecasts against.
  - **Epoch seconds, not ISO-8601 — a deliberate divergence from `storage.py`.** Every other table in this project stores ISO strings; this one stores integers. The weather bus is epoch end to end, the points arrive as epoch, and the chart consumes epoch, so converting at the store boundary and back at the route boundary would be two lies for zero benefit. (#72's forecast rows keep ISO because they juggle multi-source issuance times; a single-source observation stream has no such problem.) `ts` is the PRIMARY KEY, which SQLite aliases to the rowid — so the range scan is already indexed, and re-recording a point the archive has is a no-op rather than a conflict: `roll_history`'s dedupe rule enforced by the store. Append-only, no UPDATE path. Columns are exactly `HISTORY_FIELDS`; a test asserts the two can never drift apart.
  - **The archive is the one irreplaceable file the stack owns.** Every other piece of state here is a cache of something re-fetchable — `weather_history.json` refills in 48 h, frames re-archive, `merle.db` re-derives. A deleted observation is a moment of driveway weather no API sells back. `Servers/Pearl.md` says so where it can be acted on.
- **Resilience**: any gateway or OpenWeather failure means a skipped report (logged, the next 60 s pass retries — garnish and forecast timers stay put on failure rather than waiting out their full cadence), never a dead service; OWM errors are logged without the URL, which carries the API key. The bus side inherits `EventPublisher`'s run-with-or-without-broker contract.
- **Deployment**: runs on pearl as `willard-weather.service` (see Services on pearl under Quick start). The unit carries `MERLE_MQTT=localhost:1883`, `MERLE_ECOWITT=192.168.1.210`, `MERLE_OWM_KEY`, and — for the on-air segment — `MERLE_OLLAMA=192.168.1.79:11434` (bluejay's GPU, same as the narrator unit); `WorkingDirectory` is the repo checkout, which is where the default-pathed `weather_history.json` lives. One-time hardware note: the gateway's altitude/relative-pressure calibration is set in its own web UI — until then `rel` equals `abs` and "sea-level pressure" is a polite fiction.
- **Willard's on-air segment** (issue #45): every `REPORT_INTERVAL_S` (30 min, first one on the first loop pass), the service narrates the conditions + next-24h outlook through the narrator's LLM tier — same `MERLE_OLLAMA`/`MERLE_OLLAMA_MODEL` env vars, same kill switch (unset = no `weather/report` topic at all, everything else unchanged), and the same client code: `weather.py` imports `narrator.Ollama` and calls its generic `complete()` rather than growing a second copy that could drift. The persona is an exaggerated **Willard Scott** homage (booming folksy showmanship; the squirrels and turkeys are the viewership), a constant in `weather.py` rather than a `personas/*.yaml` — that system carries narrator-only machinery (`mqtt_id`, pacing, TTS), and Willard is a reporter, not a narrator. The prompt follows the narrator's desk-tested labeled-paragraph shape: a dry `current_facts()` paragraph (since #51 including the station's dew point, rain-falling-now + rate, rainfall today, UV when ≥1, and the barometer — each only when measured, Willard never narrates a hole), a dry `outlook_facts()` paragraph from `forecast_digest()` (next-24h high/low, strongest wind or gust, peak precipitation chance, expected rainfall in inches — each step's #56 rate × its 3 hours, summed — expected snow in inches on its own ledger (#65: Willard says "two inches of snow", never folds it into the rain number), and conditions in order of appearance), a dry `extended_facts()` paragraph from `extended_digest()` (issue #60: one terse weekday-labeled line per local calendar day after today, all ~5 days the API publishes; tomorrow deliberately overlaps the rolling 24h outlook — today in detail, the week at a glance, the way a broadcast desk talks), then the cue; all flavor lives in the persona, and the rules tell Willard to close with a sentence or two picking out the days that matter, never reciting the list (desk-tested live: he singles out the soaker and the nice one). Output is `sanitize_report()`-cleaned (collapses whitespace within paragraphs but *keeps* the paragraph break, unlike the narrator's one-line `sanitize_line`) and published retained as `{ts, text, model}` (`ts` epoch seconds, the weather-namespace convention). Failure = a skipped broadcast retried on the next 10-minute pass (the forecast-retry pattern); a current report older than `REPORT_MAX_AGE_S` (30 min) skips the broadcast rather than narrating yesterday's rain. The blocking Ollama call is safe here — it runs on the main loop's own thread (not paho's; `EventPublisher` has its own network thread) with 10 minutes of slack per pass. — `EventPublisher` grew an optional `status_topic` (Last Will `offline` + retained `online` on every (re)connect, offline published on `close()`), so any future bus process gets presence for one constructor arg. The Last Will fires on any socket death without an MQTT DISCONNECT — a crash, but also systemd's SIGTERM — so `systemctl stop` flips the dashboard within seconds with zero signal handling; the `KeyboardInterrupt` handler covers graceful manual runs. Freshness (`STALE_AFTER_S`) remains the *online-but-not-reporting* signal and the whole judgement when no retained status exists (a pre-#31 service).
- Pure logic (config/location parsing, gateway live-data + sensor-roster parsing, OWM response → bus payload mapping, the station/garnish merge, window roll/prune/thin/persist, segment prompt assembly/digest/sanitation) is covered by `test_weather.py` — the Ecowitt fixture is lifted verbatim from the real GW2000B; the fetch loop, MQTT plumbing, and live Ollama call are desk-tested against the real services (`weather/current` payload round-trips a fresh subscriber intact).

### The MCC dashboard (`mcc/`)

The daemon's face: a Next.js App Router app, one page, one client component (`components/Dashboard.tsx`) that polls `/daemon/state` every second (10s while the daemon is asleep) and renders the live MJPEG stream plus the instrument rail (current counts, run census, controls, event log, and coming-soon placeholders for future panels).

- **Production runs on pearl** as `mcc-dashboard.service` (`next start`, port 3000) and deploys via `Servers/deploy-mcc.sh` — the script enforces pull → install → build → restart, because `next start` serves the compiled `.next/`, not source: pull + restart re-serves the old code, and building after the restart rewrites `.next/` under the live server. A `fast-stop.conf` drop-in sets `TimeoutStopSec=5` (Next's graceful shutdown waits on browser keep-alive connections, up to systemd's 90s default — the MCC is a stateless proxy with nothing worth waiting for). The unit's `MERLE_DAEMON_URL` points at bluejay, which is why the daemon there runs with `--host 0.0.0.0` (see Quick start; the Asleep panel's wake command prints exactly that). Ops detail lives in `Servers/Pearl.md`.
- **The rail panels have fixed species slots.** On the Pavement and Run Census render one row per species in the daemon's `/state` roster (stable roster order via `rosterCounts()` in `lib/daemon.ts`), zero or not — a zero-count row dims instead of disappearing. Rows never insert, remove, or reorder as counts change, so a species blinking in (or a one-second misidentification) lights its gauge rather than shoving everything below it down and back up. Species counted but absent from the roster (older daemon, future classes) are appended so nothing is hidden.

- **All daemon traffic goes through an explicit proxy route** (`app/daemon/[...path]/route.ts` → `MERLE_DAEMON_URL`, default `127.0.0.1:8000`; on pearl it points at bluejay). The browser stays same-origin — no CORS in the daemon — and a phone on the LAN reaching the MCC also reaches the daemon through it; the infinite MJPEG stream passes through un-buffered as a piped body. It was a `next.config.ts` rewrite until issue #35: Next's internal proxy logs every failed upstream attempt, and with the MCC running 24/7 on pearl while the daemon only runs during bluejay test sessions, daemon-down is the *normal* state — at a 1s poll that was ~86k journal lines/day. The route instead answers a quiet `503` (any non-OK response is already the client's "asleep" path) and journals only the **transitions**: one line when the daemon drops, one when it returns. A client-side abort (closed tab) is neither logged nor treated as a transition. Upstream connects are bounded by a **3s headers timeout** (cleared once headers land, so it can never cut the infinite stream body): a dead daemon on localhost refuses instantly, but cross-machine Windows Firewall silently *drops* packets to a listenerless port and the connect would otherwise hang for minutes — the dashboard's first symptom was a permanently "loading" page on pearl. **Bus traffic is the one exception**: HTTP proxying can't carry WebSockets, so the browser connects to Mosquitto on pearl directly (`NEXT_PUBLIC_MERLE_MQTT_WS` in `mcc/.env.local`; `lib/bus.ts`).
- **Field Journal** (in `Dashboard.tsx`): the narration panel. Subscribes over mqtt.js to the retained per-narrator journal windows via the `narration/journal/+` wildcard (issue #58, namespaced in #80) — the entries' source of truth, every narrator's window replayed by the broker the moment the tab subscribes, so a reload rehydrates the whole show the way the Weather Post rehydrates its state. Windows live in a ref keyed by the topic-derived `mqtt_id` (`journalTopicId()`) and merge chronologically (`mergeJournals()` — stable sort on the ISO `ts`, capped at `JOURNAL_LIMIT` keeping the newest), so one narrator restarting or republishing can never blank the other's history; the retired bare `narration/journal` topic parses to null, so a stale pre-#80 retained blob is ignored. Also subscribes `narration/lines` and `narrators/+/status` (presence lamps, one per narrator: a dot + the name, no on/off-air text (issue #84) — the pulsing green `lamp` dot reads as "live" on its own, so a roster of two fits the panel label; `online` is the pulsing led dot + led name, `offline` a static faint dot + faint name, and any other retained payload shows `name · status` verbatim with the breathe/turkey dot — a future narrator can be "on coffee break"). The live lines topic keeps two jobs: TTS (speaking is a *moment* — a replayed window must never re-speak) and an instant prepend ahead of the window republish, which also keeps a pre-#58 narrator working through a deploy. Entry keys are derived from line content (`toJournalEntries`), not minted on arrival, so a republished window keeps React's DOM nodes and the filed-entry animation plays only for genuinely new lines (entries render in the display face — the narrator's *voice* against the mono telemetry). Each empty state says exactly which command to run (bus down vs no narrator hired), and a failed connection surfaces its reason inline (`busError`). A default-muted speaker toggle TTS-speaks new lines via `speechSynthesis`, matching the persona's `tts_voice` hint against installed voices by substring (`pickVoice`). Pure parsing/shaping (`parseLine`, `parseJournal`, `toJournalEntries`, `statusTopicId`, `busUrl`, `pickVoice`) lives in `lib/bus.ts` with Vitest coverage.
- **Journal still shots** (issue #90, presentation reworked in #96): any entry whose line carries a `frame_id` renders a lazy-loaded thumbnail **floated left in its text flow** — the prose wraps down its right side magazine-style (`flow-root` on the entry body contains the float, so a one-line entry still holds open the print's full height; the print sits left in both the panel and the broadcast view since issue #107) — served by `app/frames/[id]/route.ts` from `MERLE_FRAMES_DIR` (the folder `frame_archiver.py` fills on pearl — `?thumb=1` is the ~320px copy the panel shows; bare `/frames/<id>` the full-size variant the lightbox consumes). This is the one image path that does **not** ride the `/daemon` proxy: the whole point of archiving on pearl is that thumbnails survive bluejay's nap, so a fresh tab rehydrating the journal from retained topics gets its images too — the browser never reaches bluejay for journal images. The route answers quietly — 404 for missing/pruned files, unset dir, or an unsafe id (`frameFilename()` in `lib/frames.ts` mirrors the archiver's `[A-Za-z0-9_-]` allowlist; Vitest-covered) — and marks hits `immutable` (a frame_id's bytes never change, so window republishes re-render without re-fetching). Layout is the no-layout-shift rule made concrete: an entry's image presence is known at arrival (`frame_id` on the parsed line — optional in `lib/bus.ts`, key absent when the wire had none), so entries with images reserve a fixed 144×81 slot (`FrameThumb`) from first paint and entries without never gain one; a missing image fills its reserved slot with a quiet "faded" stamp, never a broken-image icon. The error state lives in the component, which rides the entry's stable content-derived key, so a window republish keeps it.
- **The journal lightbox** (issue #96, `yet-another-react-lightbox`): a live print is a button (`cursor-zoom-in`, hairline hover brightening, keyboard-focusable) opening the full-size `/frames/<id>` in a lightbox themed to the station (`.yarl__root` overrides in `globals.css`: near-opaque pine-black backdrop, ink-scale chrome, the entry's byline — narrator + broadcast clock — as a stamp-lettered caption, the print in a panel hairline frame). **One instance serves both surfaces** (#102): it lives in `FieldJournal`, the broadcast view is its sibling and opens it through a callback — a second lightbox would mean a second slide list to keep honest. The slide list is **every still in the current window**, opened at the clicked entry, so ‹/› page through the morning's photos; the carousel is finite (the window has real ends — a disabled arrow says so honestly). **Slide order follows the reading order of the surface that opened it** — newest-first from the panel, oldest-first from the broadcast view (`broadcastView` names the live surface, since the view covers the panel and only one is ever clickable) — otherwise the arrows run backwards through the column the viewer just clicked in. The open slide is tracked by the entry's stable content-derived **key, not an index**, re-derived each render via the controlled `index`/`on.view` pair: neither that order flip, nor entries prepending, nor a window republish can swap the photo under the viewer, and an entry aging out of the window closes the lightbox rather than showing a neighbor. Escape, backdrop click, and the close button all exit; opening and closing shift nothing underneath (the overlay is fixed-position, the no-layout-shift rule). A "faded" print is inert — the archive prunes both sizes together, so there is nothing to open. **Escape belongs to the topmost layer**: the broadcast view's own Escape handler is on `window`, which the lightbox's Escape bubbles up to, so the view stands down (`stillOpen`) while a print is up — one keypress closes the photo, the next the reading room. Its scroll lock is a separate effect with its own deps for the same reason: unlocking/relocking around a still opening would race the lightbox's own `body.overflow` restore.
- **The broadcast view** (issue #89, `FieldJournalView` in `Dashboard.tsx`): an expand control in the Field Journal masthead opens the station-view treatment for the narration — a full-screen CSS overlay (the `WeatherStationView` mechanics: `fixed inset-0 z-50` over opaque `--bg`, body scroll locked, Escape or the close button exits), live from the same entries state the panel holds (one bus client, no new subscriptions), with the presence chips and TTS toggle in its masthead (`PresenceChips` is shared by both mastheads). It reads like a conversation where the panel reads like a log: a centered `max-w-[1500px]` column (issue #107 — the station view's measure, itself the main dashboard wrapper's, so the app's two full-screen overlays agree on the page's width; the room it buys goes to the prints), **oldest-first and pinned to the newest line at the bottom** — scrolling up to reread unpins, returning to the bottom re-pins — and **threading is data-driven**: entries whose `event_kind` is `colleague_mention` are the follow-ups, indented under the report they answer with a `↳` + "follow-up" stamp, no narrator hardcoding. **The flash is for news, not history** (issue #107): `journal-filed` bundles `journal-in` + `journal-flash` and both run *on mount*, which the panel gets right for free (its list stays mounted, so only a genuinely new entry animates) but this view did not — it unmounts on close, so reopening re-ran every entry's mount animation and lit the whole thread up as new. It now snapshots the keys it opened with (`openedWith`, a `useRef` captured at first render); those render with plain `journal-in` and only lines arriving while the room is open get the full `journal-filed` flare. Stable content-derived keys mean a republished window never remounts and so can never re-flash, and reopening re-snapshots — which is what makes read history stay quiet. **Voice colors** (both views): each narrator wears a stable accent on its name stamp and entry rail — hue carries identity, intensity carries recency (newest rail full strength, the rest dimmed to 45%), body text stays ink so readability never pays for attribution. The named cast is art-directed (`voiceColor()` in `lib/bus.ts`, Vitest-covered: Marlin squirrel-orange, Jim turkey-khaki); guest voices get a deterministic palette pick, led last since it moonlights as the live signal. **Its stills are the panel's magazine wrap writ large** (issue #102): the same `FrameThumb` at `size="broadcast"` — ~40% of the reading column, ~580px at the full-width measure (the float drops on narrow screens, where a 40% print would leave a two-words-per-line ribbon beside it), pulling the **full-size** archived variant rather than the panel's thumb, which is only ~320px wide and would land far under its display size, soft on any screen. Lazy loading means only the prints in view fetch, and those are the exact bytes the lightbox opens next, so it comes up warm-cache instant. The reserved fixed 16:9 box is load-bearing twice over: the no-layout-shift rule, and the pinned scroll — the pin sets `scrollTop = scrollHeight` in an effect, so an unreserved image loading afterwards would grow the thread and strand the pin mid-column. Deliberately a reading room, not an instrument wall — more lands on this page later.
- **Weather Post** (in `Dashboard.tsx`, next to the Field Journal): current conditions + a hand-rolled SVG trend chart — observed temps trail solid to the left of a dashed "now" line (from `weather/history`), the forecast extends dashed to the right (from `weather/forecast`, bridged at the last observed point so the lines meet), wind on its own scale underneath. 24h back + 48h forward puts "now" at the 1/3 mark. **The forecast's temperature is station-calibrated at render time** (issue #71, `blendForecast` in `lib/weather.ts`): the offset between the last observed temperature and the forecast's opinion of that same moment (linearly interpolated) is added to every forecast temp, decaying linearly to zero across `BLEND_HORIZON_S` (12 h) — MOS-style bias correction anchored to the instrument of record, not cosmetic smoothing, closing the ~5 °F cliff the OWM grid cell's bias against the driveway's microclimate used to draw at the seam. Temperature only (an additive offset on the station's sheltered wind would go negative — a multiplicative wind blend is a possible follow-up); it runs inside `trendSeries`, so the panel chart and the station view both get it for free, and the hover chip reads the calibrated values because the chart and its readout must never disagree. Render-time rather than publish-time on purpose: the seam is drawn against a station point minutes old, so a blend baked in at the hourly forecast publish would re-gap whenever temperature moves fast — and the bus payload stays OWM's actual forecast (the never-silently-switch-sources ethos; Willard narrates the raw numbers). Since #51 the conditions block reads the station: five fixed right-column rows (wind, humidity + dew point, barometer + trend glyph + UV, rain today + rate-when-falling, sun times), each rendered with em-dash placeholders so data landing never shifts anything. **Outdoor owns the stage; indoor is one quiet strip** ("indoors · at the gateway", WH25 temp + RH) below the conditions — stacked, not tabs: ~15 outdoor readings vs 2 indoor ones would make a tab switch land on a nearly-empty pane and kill the at-a-glance comparison. Barometric pressure groups with outdoor/atmospheric despite being measured indoors — pressure is sky, not room. The barometer's tendency (`pressureTrend`, `lib/weather.ts`) compares the newest observed reading against the one nearest 3h earlier — real points, never interpolation — with ±0.02 inHg as meteorological "steady" and no opinion on a trail shorter than half the span; since #67 the core is the series-agnostic `seriesTrend(history, now, value, eps)`, and the station view runs it over temperature (±1.5°F), dew point (±1°F), and humidity (±3 points) too — temp/humidity arrows mostly ride the sun, honest but diurnal, while a moving dew point means the air mass itself is changing. Dew point and humidity wear theirs in their WxStat sub-lines, the hero temperature on its feels-like line (stacked vertically beside the number since #78, making room for the icon); only the barometer's appears in the small panel. **A condition icon rides left of the temperature in both views** (issue #78): eight hand-drawn SVG glyphs (`ConditionGlyph` in `Dashboard.tsx` — sunny, mostly sunny, partly cloudy, cloudy, stormy, raining, snowing, super windy) in the house line-work style — structure strokes inherit `currentColor` so stale/off-air dims them with the text, the subject of each sky (sun disc, rain, lightning) wears its accent (`--turkey`/`--led`) only while the report is fresh, and clouds fill with `--panel` so overlaps occlude. Which sky wins is `conditionIcon()` in `lib/weather.ts` (Vitest-covered), precedence drama-first: Thunderstorm > Snow > Rain/Drizzle-or-piezo (`raining === 1` outranks OWM's word — the driveway instrument beats the grid cell) > windy (sustained ≥ 20 mph or gusts ≥ 30, plus Squall/Tornado) > the four cloud-cover states read from OWM's `condition`/`description` (the rest of the atmosphere group — Mist, Fog, Haze — reads as cloudy for now; night/moon variants are a deliberate follow-up). The icon slot (`ConditionBadge`, 44px panel / 58px station view) reserves its footprint whether or not a sky is known — no report means an empty reservation, never a collapsed one. Its own mqtt.js client (same shape as the Field Journal's — panels stay self-contained; Mosquitto shrugs at a second WebSocket) subscribes to the retained weather topics, so a fresh tab paints weather instantly with zero HTTP. Beneath the chart's time axis sits **Willard's on-air segment** (issue #45): the retained `weather/report` narration rendered in the display face with a "willard, on the air" stamp + broadcast clock — the Field Journal convention (voice = display type, telemetry = mono). A segment older than `REPORT_STALE_S` (90 min = three missed half-hour broadcasts) is history, not news, and yields to the between-broadcasts placeholder; the block reserves a fixed min-height either way (no-layout-shift rule) and lives inside the dimmable content, so it fades with the rest of the panel when Willard is off duty. `parseReport` lives in `lib/weather.ts` with Vitest coverage. The masthead bills the reporter Field Journal-style — green lamp + "willard with the weather" when fresh (Willard is the weather post's on-air name; purely a dashboard label for now, a real persona if he ever narrates), "on coffee break" the moment the retained `weather/status` reads `offline` (the veil then shows when he last checked in, clock + relative age — the chart's "now" line means nothing once he's off duty — plus the `systemctl start` command), a turkey-lamp stale warning when the report is older than 30 minutes with no offline announcement, "off the air" when there's no report at all — while the conditions text ("broken clouds") sits with the temperature in the data block. The panel reserves its full footprint before data arrives (the no-layout-shift rule); when off the air, the data skeleton dims to 30% and the what-to-do message overlays it (the Live Watch veil idea), with bus-unreachable and no-service-publishing kept distinct. The chart is interactive (issue #40): hovering (or touch-scrubbing — `touch-action: pan-y`, so vertical page scroll survives) shows a crosshair + readout chip that **snaps to the nearest real report/forecast point** (`nearestPoint`) — never interpolated values — with day + time, temp, wind/gusts, condition, and an observed/forecast stamp; the chip rides the roomier side of the crosshair and everything hover-related is an absolute overlay, so it can't shift layout. Because the viewBox is stretched (`preserveAspectRatio="none"`), pointer math runs against the container rect and the marker dots/labels are HTML overlays, not SVG text/circles (which would distort). The time axis carries 12h ticks (`timeTicks`) with faint gridlines, plus near-black night bands (`nightBands`) built by repeating the current report's sunrise/sunset at 24h offsets — day-length drift is invisible at chart scale — so the 72h window reads as days at a glance. Scale whispers sit on the chart edges: temp min/max on the left (squirrel-tinted), wind ceiling on the right (the two series ride different scales, so a shared axis would lie). Pure parsing/shaping (`parseCurrent`, `parsePoints`, `trendSeries`, `blendForecast`, `tempRange`, `windCeil`, `seriesCeil`, `pressureRange`, `pressureTrend`, `linePath`, `nearestPoint`, `timeTicks`, `nightBands`, `compass`) lives in `lib/weather.ts` with Vitest coverage; temp axis pads to a 16°F minimum span so a flat day reads flat, wind axis floors at 10 mph so calm days hug the baseline, pressure holds a 0.3 inHg minimum span for the same reason.
- **The weather archive's read path** (issue #105, `app/weather/history/route.ts`): `GET /weather/history?from=<epoch>&to=<epoch>` → `{points: [...]}`, the deep past the retained bus window can't carry (a payload republished whole on every append has to stay bounded). Like `/frames/[id]` and unlike everything else the dashboard fetches, it does **not** ride the `/daemon` proxy — the writer is on pearl precisely so the record survives bluejay's nap, so the reader is a route off the same local disk. The body is byte-shaped like a `weather/history` bus payload, so `parsePoints()` reads both and **the chart cannot tell an archived point from a retained one**; the client half is `fetchArchive()` in `lib/weather.ts` — thin `fetch`, hand-mirrored types, throw on non-OK, `cache: "no-store"` (the `lib/history.ts` convention; deliberately *not* the frames route's `immutable`, which is honest only because a frame's bytes never change — a time range's contents grow). The range is **clamped to 90 days** at both ends of the wire (`parseRange` here, `clamp_range` in Python — the `/history` precedent: "a typo can't bucket ten years"), anchored at `to`, so `from=0` returns the newest 90 days rather than scanning the table; an inverted range is left inverted because it honestly selects nothing. **Unset `MERLE_WEATHER_DB`, a missing DB file, a malformed range, and an empty range all answer a quiet `{points: []}`, never an error** — on day one an empty archive is the normal state, and unlike a pruned frame (a 404), "no data here" is a chartable answer the gap-drawing chart already handles. The route has **no default path**, deliberately: `mcc-dashboard`'s `WorkingDirectory` is the `mcc/` subdirectory while `willard-weather`'s is the repo root, so a shared *relative* default would name two different files and serve an empty archive with no error anywhere — the unit carries an absolute path, the `MERLE_FRAMES_DIR` coupling exactly. No UI consumes it yet; #106 is the deep-history chart it was built for.
- **Gotcha — the SQLite driver is `node:sqlite`, the stdlib module** (issue #105's open question, resolved by checking rather than assuming). It needs **no dependency and no native build on pearl**, which is what settles it against `better-sqlite3` (proven and synchronous, but a native module wanting a build toolchain on the box). The catch is that it has been experimental and flag-gated: unflagged from Node **23.4**, and it stops emitting an `ExperimentalWarning` somewhere in 24.x (pearl's 24.18 is quiet; a 24.14 desk still warns on import — harmless, and the reason a stray warning in a build log isn't a bug). Pearl runs 24.x, so it just works; **if pearl's Node is ever downgraded below 23.4, this route is what breaks**. Two consequences already visible in the repo: `@types/node` had to go from `^20` to `^24` (the types ship with the runtime, and `next build` type-checks the route), and the route is untested in CI — the `web` job pins Node 22, where the import would need a flag. That's tolerable only because nothing in Vitest imports the route; its pure half (`parseRange`) lives in `lib/weather.ts` and is covered there. The database is opened **read-only, per request** rather than cached in a module: the MCC doesn't own the file, on a fresh pearl this route can serve traffic before `willard-weather` has created it, and a cached handle would pin that day-one miss until the next deploy — a local SQLite open is microseconds, and this is a hand-fetched panel, not the 1 s `/state` poll.

One crosshair spans the whole stack; the readout chip gains a station line (rain rate, baro, solar, UV) on observed points, and on forecast points the expected rain rate, snow inches, and chance (`pop`, shown from 5% up; rates that would print as zero stay quiet). Right rail: Willard's segment with display-face room to breathe, then "the station itself" — the indoor strip, absolute pressure, WH90 battery (5-segment meter, LED→turkey→chipmunk as it drains) + voltage, radio signal (4 segments), last-report age. The battery row prints its **voltage first and the meter last** (issue #103) so the two green meters share a right edge and read as one column — a meter placed before its voltage is the only thing in that box that can't line up with the row beneath it. Indoor readings and hardware health share that box deliberately: both are "the station", not "the weather".

That null doubles as the snap-back control's home. **Span is fixed; only position moves** — `clampWindow` (Vitest-covered) preserves the span on every clamp, and when the walls are closer together than 144h the *right* wall wins, which is not an edge case but the young-archive's normal state: the default window must stay exactly reachable on day one. `trendSeries` and `dayTicks` now take an **explicit `[ts0, ts1]`** rather than deriving one from `now`; `now` survives in `trendSeries` only as the seam that splits observed from forecast. The panel chart (`WeatherPost`) passes the same live window it always had and is **deliberately not pannable** — six days in ~400px is a smear, a legibility decision rather than an oversight.
  - **The right edge is pinned at `now + 120h` and that is not a bug.** Five days is *everything* OpenWeather's classic free tier publishes and `parse_forecast` caps nothing — the UI truncates no data, so there is no scrollable space to invent behind it. More forecast means a different tier or source (#72's territory).
  - **The left wall is discovered, never announced.** The route has no "oldest reading" endpoint (and #106 adds no backend), so the client simply clamps at the oldest point it holds and asks for one `ARCHIVE_CHUNK_S` (7 days) below it whenever a drag *reaches* for something older. The request keys off what the viewer **asked** for, not the clamped result — the clamp pins `ts0` at the wall, so waiting for the clamped window to cross it would mean the fetch never fires at all. An empty answer means the record stops there and sets `exhausted`, which also stops the asking: without it a drag held at the wall would refetch forever. A **7-day** chunk is what keeps a station outage from reading as the end of the record; a longer one still would. Fetch failure and unset `MERLE_WEATHER_DB` take the same path — the chart is exactly the chart it was before the archive existed and panning stops at the retained window's edge.
  - **Panning the first day costs no network.** The bus already publishes 48h and the view charted only 24h of it, so the second day was being thrown away; it now pans for free. The archive is merged **additively** onto the retained trail (`mergePoints`, deduped by ts — the two genuinely overlap, and the bus wins a tie), never replacing it: the trail still rehydrates from the retained topic with zero fetches. Ranges are fetched once and cached in component state, history being immutable once past. The dedupe that matters is a **ref, not state**: a chunk resolving doesn't re-render synchronously, so a `pointermove` landing between the fetch settling and React catching up still sees the previous render's `oldestKnown` and would re-request the same week — `inFlight` can't catch that, since it's already back to zero.
  - **The axes settle when the gesture does.** Every value scale is computed from the points in view, so a drag would rescale them every frame and the chart would breathe — the no-layout-shift rule violated inside the chart's own frame. They freeze at what's on screen on `pointerdown` and settle **once**, when the drag is over *and* any chunk it kicked off has landed: one controlled rescale when the viewer stops moving, never a jitter under the finger and never a jump the instant a fetch returns.
  - **Night bands stop at a 7-day horizon rather than drifting into fiction** (the owner's call). They're built by repeating *today's* sunrise/sunset every 24h, and day length drifts ~2 min/day: a week out that's ~15 minutes ≈ 2px at this scale, invisible; a month out it's an hour, which is visibly wrong, and wrong-but-confident is the one thing this chart doesn't do. So panned deep the chart simply has no night shading — an honest "we don't know the sun times back there". The browser is only ever told today's sun times, never the station's **lat/lon**, so computing them per-day for real needs that on the bus: a backend change, and its own issue. The repetition index `k` *is* the band's distance from today in days, so the horizon needs no clock of its own and the panel's live window can't reach it.
  - **Interaction**: the first drag in the codebase, so it sets the pattern — pointer events throughout (one path for mouse and touch, as the hover scrub already did), `setPointerCapture` so a drag leaving the chart doesn't strand, and `touch-action: pan-y` so a horizontal drag pans while a vertical one still scrolls the overlay (the browser fires `pointercancel` when it claims the gesture, which ends the drag cleanly). Mouse hover scrubs as before and a press-drag pans, release handing hover back. On touch the two collide, so they split: **drag pans, tap places the crosshair** — a finger resting on glass isn't hovering, and today's touch-scrub is a happy accident of pointer events that a tap preserves rather than drops. A press travelling under `TAP_SLOP_PX` is a tap. The snap-back control is a `now` button in the chart's legend line, **always rendered and merely disabled while live** — a control that appeared on pan would shove the legend sideways (house rule #1) — wearing the masthead's chrome and the day-stepper's disabled idiom rather than a new vocabulary. Zoom/span presets are the acknowledged follow-up once the archive holds enough months for dragging 144h at a time to feel tedious.

The wrapper feeds it from `useWeatherFeed()`, the bus subscription + staleness/presence judgements extracted from `WeatherPost` when this page became its second caller (each caller still runs its own mqtt client — the panels' self-contained idiom). The overlay's close affordance (and Escape) navigates to `/` — standing alone there's nothing to dismiss, so "close" means "the full dashboard". The segment now holds a page *and* an API child (`/weather/history`) — fine in the App Router, deliberate rather than surprising.
- **Gotcha — import the browser build of mqtt.js.** `Dashboard.tsx` imports `mqtt/dist/mqtt.esm`, not the default `mqtt`. The default entry is mqtt.js's **Node** build (Buffer + node streams); Turbopack bundles it for the browser, but it can't serialize packets there, so the CONNECT never sends and the client dies with `connack timeout` (~30s) — the connection opens but never completes the MQTT handshake. The `dist/mqtt.esm` build is self-contained for browsers. Types are borrowed for that subpath in `lib/mqtt.browser.d.ts`. An `error` handler on the client is mandatory: mqtt.js is an EventEmitter, and an unhandled `error` throws in the browser and wedges its own reconnect loop.
- **Design language: "Ranger Station, Night Watch"** — pine-black panels with topographic-contour background, Fraunces display type + Sometype Mono telemetry, and species accent colors that are the *actual box colors* the vision stack draws (squirrel `#FF7031`, chipmunk `#FF3838`, turkey `#CFD231`), so the UI and stream read as one instrument. Tokens live as CSS variables in `app/globals.css`.
- **The palette has two kinds of accent, and the distinction is the rule** (issue #113). The three species colors mean *"this species"* and are the vision stack's literal box colors — never pick them for decoration. **`--rain` `#6bb6e5` is the first accent that isn't a species**: it names something the station *measures* rather than something the model *detects*, which is the line that keeps the species rule intact. It also gives `--led` its one job back — green means **live** (presence lamps, on-air stamps, the falling-now dot), and it had been moonlighting as rain since #56. It was **chosen by the validator, not by eye**, and the finding is worth keeping: *any* blue light enough to match `--led` (OKLCH L 0.829) or `--turkey` (0.835) falls under the 0.10 chroma floor and reads **gray** — blue's gamut peaks darker than green and yellow, so "match the palette's brightness" is a trap. `#6bb6e5` (L 0.746, C 0.12) is the brightest blue that still reads blue, and it lands between `--squirrel` (0.711) and `--led` — inside the palette's own range. Its CVD separation from every other ink is ≥ 34 ΔE against a target of 12; the palette's genuinely tight pair remains the pre-existing chipmunk↔squirrel (ΔE 10.1 deutan), which is why `SPECIES_ORDER` keeps orange and red from ever touching.
- **Precipitation wears its intensity as shade** (issue #113): each bar's ink tracks **its own value** — rate for the observed piezo bars, chance for the forecast ghosts, inches for snow — so a 90% Friday reads as a tall vivid slab and a 20% Tuesday as a short ghost without hovering. Redundant with the bar's height *on purpose*: the shade makes no claim the height doesn't already make, so it cannot be misread and needs no legend of its own. (The tempting alternative — height = chance, shade = expected *volume*, restoring what #65 banished to the hover chip — is a real candidate that needs a legend to explain it, and so wants its own issue.) Three details are load-bearing: **the ramp mixes toward `--panel` rather than varying opacity**, because the observed bars are 0.7 viewBox units wide but 5-minute points sit ~0.56 apart — they *already overlap*, and stacked translucent bars compound into intensities the station never measured (the warning that has sat in the rain strip since #56); solid fills overlap by overwriting. **The faint end is floored** so "unlikely" never reads as "no data" — and the floors differ per ink because white and blue don't vanish at the same rate: `RAIN_SHADE_FLOOR` 0.4 and `SNOW_SHADE_FLOOR` 0.3 both land at ~2.1:1 against the panel (measured, not guessed). **The forecast's ceiling stays under the observed trail's full voice** (`FORECAST_SHADE_CEIL` 0.85): a prediction, however confident, must never shout as loud as a measurement — while snow reaches *full* white, because the piezo is snow-blind so its strip has no measured sibling to out-shout. Direction is "vivid = more": on pine-black, fading a colour blends it toward the near-black panel, so faint reads **dimmer and greyer**, not lighter — which is why "solid white for heavy snow" and "stronger blue for a sure thing" are the same ramp described from opposite ends. `precipShade`/`precipFill` are pure and Vitest-covered; the normalisation is what earns the test (hand it the wrong ceiling — `rainMax` vs a fixed 100% vs `snowMax` — and every shade is wrong while still looking plausible).
- **The strips' headings read over their own data** (issue #113). They sit in the top-right corner because #56 put them in the forecast half, on the premise that the station leaves the future blank — **a premise #65 destroyed** when it rescaled the ghosts to chance on a fixed 0–100% strip, where a confident Friday is a full-height bar growing straight into that corner. So they stop relying on it: 11px in `--ink-dim` (was 10px in `--ink-faint`, the palest ink on the board) wearing a `--panel/80` **scrim**. The scrim is the part that matters — measured against the loudest possible bar the heading went from **1.80:1 to 4.98:1**, clearing WCAG AA, and bigger-and-brighter alone wouldn't have: the old label was only 3.23:1 even over bare panel. Both the heading and the left-edge scale whisper stay absolute overlays, so none of it can shift the chart.
- **Each forecast peak and valley wears its temperature** (issue #113, `tempMarks` in `lib/weather.ts`): the question a viewer actually has — how hot Friday gets, how cold tonight — answered without hovering point to point. **They are turning points, not per-calendar-day min/max, and that distinction is the whole design.** Calendar bucketing is right for *speech*, which is why `weather.py`'s `extended_digest` does exactly that for Willard's "days ahead"; it is wrong for labelling a *curve*, and it fails precisely when the chart is most worth reading — on an evening cold front the temperature falls all night, so Friday's coldest sample is 23:00 (a bucket edge, still dropping) and Saturday's is 06:00: two labels seven hours apart on one continuous slide into one valley. Turning points give that valley the single label it deserves with no special-casing of midnight. **Willard and the chart therefore answer different questions** — "how cold does Friday get" versus "where does this curve bottom out" — and will visibly disagree on exactly that night, and *both are right*; that divergence is deliberate, not a bug to file. The labelled point is always a **real** forecast point: averaging the two lows would invent a temperature nobody forecast, which is the one thing this chart doesn't do (`nearestPoint` snaps, `linePath` splits on null rather than bridging). Endpoints are never marked — the first and last points aren't turning points but where the data ran out. **The observed-side bridge is dropped outright rather than merely left unlabelled** (issue #103): `coming[0]` is `trendSeries`' seam stitch, a 5-minute trail sample wearing the forecast series' clothes, and leaving it in the array to act as a *neighbour* is enough to do the damage — it is the one sample guaranteed to sit beside the first forecast step, so on any morning climbing toward its high the last reading before the seam lands above that step and manufactures a valley the weather never had, labelled at the now line where the trail is steepest and cuts straight through the glyphs. That is precisely the 5-minute noise this function is forecast-only to avoid, readmitted one point at a time. `trendSeries` therefore reports `bridged` (only the drawn line wants that point; nothing reading the forecast can tell by looking), and `tempMarks` slices it off **before** the null filter — the bridge is `coming[0]` whatever it measured, so filtering first would eat a real step instead and the phantom would simply move one point right. The cost is a genuine dawn valley losing its label whenever it falls on the first step: that step has no forecast predecessor to turn against, which makes it an endpoint like any other, and silence beats a turning point judged against a different series. A `TEMP_MARK_MIN_GAP_S` (10h) rule thins a shower's 2°F wiggle into the real afternoon; because extrema always *alternate*, the same-kind rival is two marks back, not one — comparing only against the previous mark would never catch the pair that matters. Labels read the **blended** temps (#71 — `trend.coming` is already calibrated) or they'd disagree with the line they sit on, and they're HTML overlays for the same reason as the snapped dots (the viewBox is stretched; SVG text would squash). Forecast only: the observed trail is 5-minute data where every passing cloud is a turning point. **Caveat worth knowing**: OWM's free `/forecast` is 3-hour steps, so the labelled peak is the *sampled* peak — the true daily high may sit between two samples and run a degree or two warmer.
- **Daemon-down UX**: a failed `/state` poll shows the "Merle is asleep" panel with the wake command; when polls recover, the `<img>` is remounted (key bump) to reconnect the stream. While asleep the poll backs off from 1s to 10s (issue #35) — the daemon being down is the steady state on pearl, and a control click still polls immediately, so waking stays snappy.
- **Live Watch** (`VideoFeed`) has a YouTube-style fullscreen toggle (bottom-right, hover-revealed) using the browser Fullscreen API on the feed container; double-click toggles, Escape exits natively. The stream freezes on its last frame whenever it isn't live, so three distinct veils cover it: **stand down** (engine idle, `running=false`), **reconnecting** (`live.signal=false`), and **asleep** (daemon unreachable) — a frozen frame is never mistaken for a live one.
- **Station Records** (Phase 4, in `Dashboard.tsx`): the history shelf under the main grid, fed by `/history` via `lib/history.ts` (60s refresh, not the 1s poll). **Field Census** — 14-day stacked bars of distinct visitors/day in the species accent colors, stack order squirrel→turkey→chipmunk (the order that keeps adjacent pairs colorblind-separable — validated, not eyeballed; 2px surface gaps + legend + tooltips + the selected-day text breakdown carry identity beyond color alone). Direct labels only on the peak and selected day. Clicking a day (or ‹›) selects it: per-species totals as text plus an hourly arrival strip from `/history/day`. **Hard-Frame Harvest** — window total + per-day mini bars. **Training Rounds** — lineage table, newest first (`runsNewestFirst`), with the not-same-ruler-across-splits caveat printed right on the panel. ## Music catalog (epic #115, Phase 0 — issue #120)

`music_catalog.py` (the store) + `music_index.py` (the read-only indexer) are the foundation every later music phase reads. **`music.db` lives on pearl**, next to its writer — `weather_archive.py:22-29`'s rule, unchanged: `merle.db` sits on bluejay where daemon-down is the steady state, and music must play when bluejay is asleep. It is **never on the NAS**: SQLite's locking is unreliable on SMB/CIFS, which is a corruption risk rather than a performance note. Config is `MERLE_MUSIC_DB` / `MERLE_MUSIC_ROOT` (`get(X,"").strip() or DEFAULT`); any future MCC-side reader takes an **absolute path with no default** (`mcc/app/weather/history/route.ts:20-25` explains why). Timestamps are **unix epoch seconds**, siding with `weather.db` over `merle.db`'s ISO-8601: ratings and history get ranged far more than read by a human.

**The audio files are an immutable input — we never write tags back to them.** Tags are read once at index time and every derived field lands in the catalog, which makes it rebuildable and disposable and lets metadata iterate against a 612.7 GB library without risking it. The share is mounted read-only (`//hummingbird/music on /mnt/music type cifs (ro,...)`), so the principle is enforced by the mount rather than by discipline.

**Identity is a hash of the audio stream, not of the file, and that was measured rather than argued.** A path is a location — files get renamed. But hashing whole file *bytes* has a subtler failure: this library is iTunes-managed (the share carries `.itl` and 2,241 `.itc2` files), so tag rewrites are live, not hypothetical. Retagging one m4a and one mp3 changed the whole-file hash both times and left the audio-stream hash **byte-identical** — a whole-file hash would mint a new identity on every tag edit and orphan that track's ratings and history, the exact failure the content hash exists to prevent. *(Rejected: whole file bytes — trivial, and wrong for this library.)* Spans are located with stdlib parsing, no dependency: `mdat` for m4a/mp4, ID3v2-syncsafe-to-ID3v1/APEv2 for mp3, post-metadata-blocks for flac, the `data` chunk for wav — probed at 32/32 across the real formats. FLAC's STREAMINFO carries an **MD5 of the decoded audio** (present on 8/8 samples), which survives even a re-compression, so it wins where non-zero. The hash is **blake2b-128** (stdlib): content-addressing for identity has no adversary, so SHA-256 buys nothing, and at GB/s the hash is free against a wire that delivers ~61 MB/s. A file whose span won't parse is **not dropped** — it lands with `needs_attention` set (Phase 1's bucket), a queryable number rather than a mystery.

**`tracks` is keyed by hash; locations live in `track_files`, one-to-many.** A 26k-track library rips the same recording twice (an album and a greatest-hits), and the real 400-file smoke pass found **369 tracks in 400 files** — 31 duplicates on the first try. Collapsing them is correct: rating a track rates the recording, not whichever copy played. Folding path into `tracks` would make a duplicate either a spurious second identity or a row whose path flip-flops every pass. A rename is therefore a re-hash that re-links to the same id, leaving a stale row that **`--prune` reclaims** — never automatically, because the indexer cannot distinguish "the files moved" from "the share isn't mounted", so `prune_is_safe()` refuses below a 50% floor. `forget_paths()` only ever drops locations: a track we can't find is not a track that never existed.

**The analysis columns are populated since #136** — `bpm`, `replaygain_db`, and `dynamic_range_db` went from <9% (tags) to 92.3% (measured), and `needs_attention` holds its first real entries: 15 files that hash fine but won't decode. See the audio-analysis section below.

**Both feedback tables are live**: `play_history` since #129 (a row per play, the moment anything sounds) and `ratings` since #135 (the daemon's `POST /rate`). Explicit and implicit feedback now accumulate together, which is what Phase 3 reads as rules and Phase 4 as evidence. Implicit feedback cannot be backfilled or bought, which is why the table predates its writer and why Phase 2a shipped Denon-first rather than waiting for the browser path. The tables are also what makes **schema evolution** non-deferrable: `CREATE TABLE IF NOT EXISTS` does not alter an existing file, and these tables accumulate for months before Phase 3/4 read them — exactly when a schema change lands on irreplaceable data. Hence `PRAGMA user_version` + an ordered, append-only `MIGRATIONS` list, built before it's needed. **The catalog is rebuildable from the NAS; ratings and play-history are not** — they are to music what `weather.db` is to the station.

**The walk skips the share's `#recycle` bin** (`EXCLUDED_DIRS`, issue #129). The first pass indexed it — 3,096 locations of *deleted* tracks, one shelf-render away from the GUI once real rows arrived — and the fix is two-part by design: the exclusion stops new bin entries at the walk, and a `--prune` pass reclaimed the ones already cataloged (23,494 locations remain; the pass saw 88% of known locations, comfortably over `prune_is_safe()`'s floor). Locations only, as always — any track that also exists outside the bin kept its identity, ratings, and history.

**Measured, replacing epic #115's estimates:** the library is **612.7 GB / 26,590 audio files** (m4a 62.3% · mp3 32.1% · flac 4.2% · wav 1.4%), not ~500–600 GB / ~27k. The indexer runs at **~61 MB/s hashing, ~56 MB/s with tag reads → a ~3.0 h first pass**; a raw sequential read of large files clocks 99.8 MB/s, so the real walk runs at ~60% of best-case streaming — per-file open/seek overhead across 26k mostly-12 MB files, not the hash and not the chunk size (1/4/8 MB read chunks landed within 1%, since the mount's `rsize=65536` governs). The **second pass is seconds**: the hash cache is keyed on `(path, size, mtime)` and re-reads nothing. Both fields are load-bearing — the measured retag left file **size unchanged** and moved only mtime, so a size-only cache would skip that file forever. Two consequences: **`vers=2.0` costs nothing** (the wire is the limit, so there's no throughput to win by revisiting it — #115's open question, closed), and **pearl reads at wire speed**, so D2's transcode cache stays on pearl and hummingbird's escalation triggers remain unmet.

**The pass runs on pearl, not bluejay, despite bluejay's faster CPU** — because it is wire-limited, and that was measured: pearl reads the share at 99.8 MB/s to bluejay's 90.2, both at gigabit saturation, while blake2b hashes at GB/s. pearl also already has the mount and already owns `music.db`, so it writes rows directly with no export/import hop. *(Rejected: bluejay + JSONL + `INSERT OR IGNORE`. That is the right shape for **Phase 1**, where beat tracking is genuinely CPU-bound and bluejay is several times pearl — but for Phase 0 it buys a slower pass and an extra moving part.)*

## Music audio analysis (epic #115 Phase 1a — issue #136)

`jukebox/music_analyze.py` (bluejay, offline) → JSONL → `jukebox/music_import.py` (pearl). BPM, ReplayGain, and dynamic range for every track, computed from the audio because the tags don't carry them (all three were under 9%). **Coverage went 0 → 23,440 of 25,382 tracks (92.3%)** in a single 1:32:43 pass; the missing 7.6% is not a shortfall but the 1,927 tracks whose only copy lived in the `#recycle` bin and lost its location to #129's prune — **of tracks that have audio, coverage is 99.94%**.

- **One ffmpeg invocation does both jobs.** The `ebur128` filter measures loudness *and* passes the audio through, so the same decode feeds beat tracking on stdout (`-f f32le -`; nothing is ever handed a path to write). Reading twice — once for loudness, once for beats — would double the wire cost for nothing. *(Rejected: `librosa.load()` for the decode, which hands non-WAV/FLAC to audioread, which shells out to ffmpeg anyway — the same read with less control and no loudness.)* ffmpeg is not optional: the library is 62% ALAC, which libsndfile cannot decode.
- **The epic's "CPU-bound" premise is false at scale, and this is the number that matters.** Measured on bluejay's 16-core i9-12900KF: 8 workers → 2.64 tr/s, **16 → 3.18**, 24 → 3.29 (+3% for 50% more workers). The full run sustained **3.99 tr/s ≈ 92 MB/s**, which is bluejay's measured wire ceiling (90.2 MB/s, Phase 0) — so past ~16 workers **the wire is the limit again**, not the CPU, and more workers buy contention. D2's "put the CPU work on the fast CPU" reasoning still holds; its stated reason stops applying at the knee.
- **The JSONL is the resume state, deliberately — not the catalog.** A bluejay-side snapshot won't know a track is analyzed until the import runs on pearl, possibly days later. Records are flushed per track under a lock, so a `taskkill /F` mid-run loses only the ~16 in flight (demonstrated: killed at 875 tracks, restarted, resumed at exactly 891 with the 16 in-flight redone). A torn final line is skipped, not fatal.
- **`ex.map` + `with ThreadPoolExecutor` is a trap for an interruptible pass**: `map` submits every future up front and the context manager's `shutdown(wait=True)` drains the whole queue, so Ctrl-C raises in the main thread and then waits out the entire run. `shutdown(cancel_futures=True)` plus a `stop` flag that no-ops dequeued work is what makes an interrupt interrupt.
- **The importer UPDATEs, it does not INSERT.** The epic said `INSERT OR IGNORE`, but Phase 0 put these on `tracks` as nullable columns — same idempotency, different verb (proved on the real 25,382-row table: a second import leaves the fingerprint byte-identical). It updates *unconditionally* so a future better tempo algorithm can land without a migration; the *analyzer* is what skips finished work (`--force` re-does it). An id the catalog doesn't know is counted and skipped, never inserted — `tracks` rows are the indexer's to create.
- **Two silent failures the catalog could not have caught.** Phase 0 reported **0 `needs_attention`** because hashing only *locates* the audio span; it never decodes. This pass decodes every byte and found **15 genuinely broken files** (0.064%) — 13 that ffmpeg refuses outright (three consecutive Genesis tracks on *The Lamb Lies Down on Broadway*, a bad rip) and 2 truncated stubs whose tags lie about their length (The Prodigy's "Claustrophobic Sting" claims 434 s in 0.3 MB, where ALAC would need ~40 MB). A valid container with corrupt guts hashes perfectly and looks healthy.
- **Path remapping, not a second catalog.** `music.db` stores locations as *pearl* sees them (`/mnt/music/...`); the analyzer runs elsewhere, so `remap_path()` swaps the prefix for `MERLE_MUSIC_ROOT` (`\\hummingbird\music`) rather than teaching the store about a second machine's filesystem. A path outside the catalog root returns None and is skipped loudly — better than building a nonsense path and blaming the NAS.
- **Measured, for whoever tunes this next**: BPM median **117.5** (p10 89, p90 152; only 4 tracks over 220, so no double-time epidemic), ReplayGain median **−7.7 dB**, dynamic range median **5.4 LU**. **The loudness war is visible in our own library**: average gain by release decade runs −2.5 dB (pre-1990) → −5.2 (90s) → −7.6 (2000s) → **−8.4 (2010+)**, i.e. masters got ~6 dB hotter across four decades.
- **Nothing downstream depends on the analyzer existing** — verified with the analyzer stopped: pearl serves `/state` and the app on :3001 unchanged. After the backfill bluejay can be powered off forever.

## Music catalog normalization (issues #163 genre, #152 artist)

One pass (`jukebox/music_genre.py`), one rules file, two normalized columns. The library's **178 feral iTunes genre strings** (`Electronica\Trip-Hop` in three separator spellings, `FLAC` as a genre, `Industrial Jungle Pussy Punk`) normalize into a **22-tag canonical vocabulary** in `tracks.genre_norm`, and its **26 case-split artist identities** (`Panic! at the Disco` / `Panic! At the Disco`, measured across 748 identities) collapse into a canonical display name in `tracks.artist_norm`. Raw `tracks.genre` / `artist` / `album_artist` are never touched — they are provenance, the indexer keeps writing them verbatim, and every decision stays reversible because the input is still there. **Every surface reads only the normalized columns** (pills, browse filter, playlist affinity; artist grouping, ids, art keys), which is the enforcement: a weird new tag has no UI existence until deliberately mapped, and a new casing cannot split an artist.

- **The artist stage (#152) is the #163 materialization move applied to identity**: fold each track's artist identity (`COALESCE(album_artist, artist)`, trimmed + `str.lower()` — deliberately *not* `casefold()`, because simple lowercase is what a TS twin could implement identically and casefold's `ß`→`ss` mappings couldn't be), tally casing frequency, write the winner. Tie-break, deterministic (the catalog has a live 12–12 tie, Run DMC): most tracks → a casing seen in `album_artist` beats one only seen in track `artist` → lexicographic. The rejected alternative was fold-key resolution at read time — three lowercase implementations (Python/TS/SQL) that must never drift, one of which (SQLite `lower()`) is ASCII-only and wrong by construction; materializing means **no folding exists downstream at all**. `artist_display` in the rules file is the taste veto (`GWAR`, not the library's majority `Gwar`) and the pin against a future import flipping a winner; an override must fold to the same identity — it picks a casing, never renames.
- **Identity flows through one SQL fragment, twinned**: `music_catalog.ALBUM_ARTIST_SQL` / `db.ts`'s `albumArtistCol` put `artist_norm` first with the raw COALESCE as fallback, so a pre-pass snapshot degrades to case-split grouping — never to a crash or broken art. `ALBUM_KEY_SQL` (art keys) and `albumIdOf`/`artistIdOf` (URLs) therefore mint canonical, case-collapsed identities; art rows written under a dead casing simply re-enter `albums_missing_art` / `artists_missing_art` and the next art-pass run re-extracts them (self-healing by worklist — run it once after the first normalize on a populated catalog). Track rows keep the raw per-track performer credit; `Track.albumArtist` carries the identity for surfaces (search grouping) that need the canonical name off a track alone.
- **The stages feed forward**: genre's artist-majority inheritance and `artist_overrides` key on the folded identity, so a majority split across `Gwar`/`GWAR` tallies as one artist and override keys match case-insensitively.

- **The rules are data, not code** (`jukebox/genre_rules.yaml`, the owner requirement): `vocabulary` + `map` (canonical → raw strings; a raw maps once) + `artist_overrides` (judgment calls no string can express; also where the future curation of unresolved artists lands) + `clusters` (the playlist engine's genre families — a custom vocabulary needs custom families, so they travel together) + `tuning` (the 0.6 inherit threshold and the `triage` junk-drawer list). The repo file is the owner's live ruleset, so **rule changes are commits**; a second deployment (the metal-heavy friend library) points `MERLE_MUSIC_GENRE_RULES`/`--rules` at its own file, never at code. YAML over JSON specifically for comments — `FLAC: # that's Burial` is half the file's value. Validation fails loudly before any write: targets in vocabulary, no duplicate raws, no unknown sections, mapped-or-triaged-never-both. `test_music_genre.py` pins the shipped file as valid (the typo canary — a bad edit fails CI, not pearl mid-pass).
- **The pass** (`jukebox/music_genre.py`, the #153 reusability rule whole): syncs the file's map into the `genre_map` table one-directionally (`source='file'` rows upserted, stale ones pruned; `external` rows — the future bulk-metadata pass's alias lane — never touched by the sync), then recomputes every track's expected pair and writes only diffs. Same run twice = zero writes; post-ingestion run touches exactly the new rows; **a rules edit re-run IS the "reprocess under my new rules" job** — there is no separate backfill tool.
- **Precedence**: `owner` (hand-set, protected in the UPDATE's WHERE like the art store — no automated pass can clobber it) > `artist_overrides` > string map > `external` (a per-track backfill value yields only to an explicit rule) > **inheritance** (a track whose raw maps nowhere takes its artist's genre when ≥60% of the artist's mapped tracks agree; the artist key is the album-artist COALESCE so compilation guests don't fragment the majority) > honest NULL. Provenance lives in `genre_norm_source` (`mapped|inherited|external|owner`); both columns ride migrations v5/v6 (one ALTER per step — the focal_y lesson) and sit outside `TRACK_COLUMNS`, so the indexer's tag-refresh upsert can't wipe them.
- **The deferred set is named, not lost**: NULL-genre tracks plus the five `triage` strings (`Pop\Rock` is AllMusic's catch-all — Violent Femmes/Skinny Puppy/Fear Factory/Prince side by side — plus `Pop/Rock`/`Styles`/`Popular`/`Other`) stay `genre_norm` NULL for the bulk-metadata-import pass (bios + genre via MusicBrainz/Last.fm, routed through the same `genre_map` as `external` rows). Triage strings stay off the unmapped report; anything else unmapped prints at the end of the pass with counts — the fix is one rules line + a re-run.

## Music playback daemon (epic #115 Phase 2a — issue #129)

`jukebox/music_daemon.py`: a pearl-resident FastAPI service (**port 8090**, the `music-daemon` unit — `Servers/Pearl.md`) that streams catalog tracks over HTTP and drives the **Denon AVR-X4000** over UPnP/DLNA. The epic's spike inverted the design's build order — the Denon plays ALAC natively and untranscoded (verified, audible, parses the container's own duration), while Chrome refuses ALAC outright (`SRC_NOT_SUPPORTED`) — so 2a shipped the Denon first (starting the unback-fillable `play_history` clock), and the browser became its own phase with a transcode path (2b, next section).

- **DLNA means the renderer pulls.** `POST /play` hands the Denon a URL (`SetAVTransportURI` + `Play`); the Denon then fetches `GET /stream/{track_id}` itself. The daemon is two halves meeting at a URL: a Range-capable file server and a small SOAP client. **The Denon receives the file's bytes untouched** — nothing sits between `open("rb")` and the socket, which is the point of that output. `MERLE_MUSIC_STREAM_BASE` (required, no default) is the daemon's own LAN-visible URL, because guessing your own address is how you stream to a renderer that can't reach you.
- **Range is hand-rolled and non-negotiable** (`parse_range`, exhaustively tested): `http.server` doesn't implement it, no Range means no seeking, and the Denon really does open with a probe GET then a Range request into the audio. Suffix ranges are real (trailing-index reads); multi-range is answered with the whole file (RFC 7233 allows ignoring Range; no real client sends it).
- **Capability is a table we own** (`OUTPUT_FORMATS`), not a negotiation — the Denon answers `GetProtocolInfo` with HTTP 500, so asking is not an option. The answer to "does this need transcoding?" is the catalog's `format` column, read before the stream starts. Nothing sniffs bytes at runtime.
- **Discovery, never hardcoding**: SSDP `M-SEARCH` at startup (retried on first `/play` if the renderer was off), control URLs read from each device's description XML — the epic's warning about UUID-embedded paths, and this AVR has already changed address once between probes.
- **One history row per play, adjudicated by the watcher thread**: `completed` when the transport stops with the last position observed *while playing* near the end (the Denon zeroes `RelTime` the instant it stops — reading position after the stop would call every completion a skip-at-zero), `skipped` when the listener moved on (`/play` over a playing track, `/stop` mid-song, or a stop at the AVR's own remote). Unknown duration or position is always a skip: implicit feedback errs toward not crediting a listen that didn't happen.
- **Presence on the bus for free**: `bus.EventPublisher("music", status_topic=MUSIC_STATUS_TOPIC)` — retained `online`/`offline`, Last Will, own `music/` namespace (a jukebox is not a narrator; the Field Journal's wildcard must not light up). `systemctl stop` flips it within seconds with zero signal-handling code — verified with a real SIGTERM.
- **Two flags are load-bearing on the uvicorn line**: `--no-access-log` (the GUI polls `/state`; issue #125's flood, preempted) and `--timeout-graceful-shutdown 3` — the **renderer** holds `/stream` open for the whole song, so an unbounded graceful drain turns SIGTERM into a zombie that outlives its replacement. Observed live on pearl mid-playback; the vision daemon's flag, same lesson, different client.
- **Hardware findings, recorded so nobody re-discovers them**: the Denon refuses AVTransport `Seek` (HTTP 500) on external streams — the GUI's restart-track gesture re-issues play instead of seeking to 0, and a scrub snaps back to the truth on the next poll. Real browser seek arrives with the 2b browser output, where Range actually serves it.
- **The daemon is the catalog's only writer** (issue #135). `POST /rate` `{track_id, value}` records a thumb; the music app's own handle is `readOnly: true` by construction, so a rating comes here rather than minting a second writer. Validation lives in `music_catalog.rate()` next to `RATING_VALUES` — one copy of the legal set, or the two drift. Two traps it closes: **`bool` subclasses `int`**, so a JSON `true` satisfies `in RATING_VALUES` and would file itself as a thumbs-up (rejected explicitly); and **`value: 0` is legal to send but illegal to store** — it's the control's third click, so it dispatches to `unrate()` and deletes the row. An unrated track is the *absence* of a row, never a stored zero: Phase 3's filters read what exists, and a 0 would be a third state every one of them would have to know to ignore. The timestamp is the daemon's, never the client's — a browser clock is a guess, and this is the table we can't rebuild.
- The id allowlist (`music_catalog.valid_track_id`, `[A-Za-z0-9_:-]+`) runs before any catalog query — `frame_archiver.py`'s traversal-guard genre, wire ids never meet the filesystem. The share stays `ro`; every open is `"rb"`.

## Music browser output (epic #115 Phase 2b — issue #149)

The third output is the listener's own browser: the music app's hidden `<audio>` element streams `GET /stream/{id}?output=browser` straight from the daemon (not through the Next proxy — piping audio through the app server would re-buffer it and break Range). The daemon drives nothing; the tab is the transport.

**The lossless rule is absolute: no lossy transcode anywhere in the music path, ever** (owner requirement, recorded in #149's do-not-change list). The library was ripped lossless on purpose, so the browser's per-output policy (`music_cache.needs_flac`, decided from catalog columns, never by sniffing) is:

| Catalog says | Browser gets | Why |
|---|---|---|
| mp3 / flac / wav | **raw bytes**, the Denon's exact serving code | decoded natively |
| m4a/mp4 + `codec='aac'` | **raw bytes** | already lossy; repacking to FLAC inflates without gaining a bit |
| m4a/mp4 + `codec='alac'`, `NULL`, or anything else | **FLAC repack** | Chrome refuses ALAC; ALAC→FLAC is lossless-to-lossless (identical PCM, proven by `ffmpeg -f md5` decoded-PCM equality). Unknowns take this path because its worst case is wasted bytes, never lost ones |

- **The `codec` column** exists because `format` is just the extension and `.m4a` holds both ALAC (the majority) and lossy purchase-AAC. The indexer probes the mp4 `stsd` sample-entry fourcc (`music_index.mp4_codec` — a real atom walk, not a byte scan; `b"alac"` appears in magic cookies and cover art) at index time for new files; `python3 -m jukebox.music_index --codecs` backfills rows indexed before the column existed (header reads, minutes over SMB). It rode in as the migration seam's **first real migration** (v2) — which surfaced the seam's latent fresh-file trap: `connect()` now stamps a brand-new file straight to `SCHEMA_VERSION` instead of replaying ALTERs against a schema that already mirrors them, and `migrate()` tolerates "duplicate column" for the DDL-landed-stamp-didn't crash window. Only m4a/mp4 rows carry a codec; elsewhere the extension *is* the codec and a second column would be drift risk.
- **The cache** (`jukebox/music_cache.py`) lives at `MERLE_MUSIC_CACHE` (`/srv/media-cache/music` on pearl — music's subdirectory of the **shared** media-cache LV; tenants get sibling dirs and their own retention, see `Servers/Pearl.md`). **The cache key is the track id** — the audio-stream content hash — so staleness is structurally impossible: a re-rip is a new hash and a new entry; a retag changes nothing; the old entry becomes an orphan and is swept. There is no invalidation logic to get wrong.
- **Cached files, not live pipes**, because a pipe can't serve Range and no Range means no seeking (the epic's hard UX floor). The cold click takes the middle path: ffmpeg writes `<id>.flac.part`, the response **tails the growing file** (`iter_growing` — playback starts within the first frames, chunked, Range ignored per RFC 7233), and success renames to `<id>.flac` — rename-then-signal, so presence of the final name *is* the completeness contract. Degradation on that first cold serve: ffmpeg's placeholder STREAMINFO means no duration/scrub until the file completes (seconds); every later serve — and the MD5 acceptance check — reads the patched file. `POST /precache {track_ids}` warms the next queue tracks (gated to 2 concurrent so warming never starves a cold click, which starts immediately), so queue advances are cache hits and the transcode delay is unobservable in playlist listening.
- **Eviction is a pure plan** (`plan_evictions`, tested): stale `.part`s (crashed transcodes), then **orphans before any live entry** (hash no longer in the catalog — the re-rip leftover), then oldest-mtime LRU down to `MERLE_MUSIC_CACHE_CAP_GB` (default 40). mtime is the recency clock on purpose — the daemon `touch()`es on every serve because the mount is `noatime`. Swept after every transcode and at startup. A full or missing cache degrades (log, fall back), never fails playback.
- **Unset `MERLE_MUSIC_CACHE` is the kill switch** (the `MERLE_OLLAMA` convention): the daemon offers no browser output at all — `/state`'s picker row says `available: false` honestly rather than shipping an output that's broken for the ALAC 62%. Set-but-missing dies loudly at startup (the LV didn't mount).
- **Play-history parity**: a browser tab can't be polled, so the provider **reports** each session's end — `POST /report {track_id, position_s}` on `ended`, on skip-away, and on output switch. The daemon adjudicates with the *same* `outcome_for` the Denon watcher uses and stamps its own clock: one arbiter, two transports, rows land as `output='browser'`.
- **Client side** (`music/components/PlayerProvider.tsx`): one provider, two transports. Browser mode plays the hidden `<audio>` (position from `timeupdate`, advance from `ended` — the 2s daemon poll stays off; both transports share one `advanceAfterEnd` so repeat behavior can't drift), seeks for real, warms the next two queue tracks on every playhead move, and builds srcs from the daemon's `stream_base` (served by `/state`; server config never bakes into client bundles). Switching outputs mid-track **stops rather than migrates** — the transports don't share a position. The Denon path is byte-identical to 2a: `?output=` defaults to `denon`, so every URL the daemon ever handed a renderer means what it did before.

## Music cover art (epic #115 GUI track — issue #153)

Real artwork on every surface, from the library's own files — no network fetch anywhere in the path.

- **Sources and coverage (measured)**: embedded art on ~83% of tracks, WMP-era `Folder.jpg` in 38% of album dirs, union ~90% of the 1,872 albums. The extractor (`jukebox/music_art.py`, runs on pearl, minutes) takes the **largest embedded picture** across an album's tracks, falls back to `Folder.jpg`/`Cover.jpg`, and writes nothing for the remainder — those keep the generated SVG and stay on the worklist for the next run.
- **Focal analysis** (issue #159): every stored cover gets `focal_y` — the vertical interest centroid (FIND_EDGES over a 64×64 grayscale downscale, 1px border cropped so the frame doesn't drag everything to center; clamped [0.2, 0.8], uniform covers land at exactly 0.5), computed **inline at extraction** since the image is already open, pytest-covered. Migration v3/v4 added the column to `album_art`/`artist_art` (two steps, one ALTER each — `migrate()`'s duplicate-column tolerance would strand the second ALTER of a combined script); `--focal` backfills pre-#159 rows from the `.large.webp` variants (measured: ~2,500 rows in under half a minute, local reads) and is a no-op forever after — a fresh install never runs it. Analysis is not a clobber: the focal-only setters skip the owner guard on purpose, since *where the interest sits* is orthogonal to *which image the owner chose*. The album page anchors its backdrop band with `object-position` from `Album.artFocalY` (rides the index entry through `getAlbum`; null = center = pre-focal behavior); square renders (grids, the anchor cover, the lightbox) stay dead-center.
- **The reusability rule** (owner requirement): every enrichment pass — index, `--codecs`, analysis, art — is worklist-driven, idempotent, and standalone. Art's worklist is "albums with no `album_art` row", so the future ingestion pipeline re-runs the same pass and touches exactly the albums it just added. Nothing assumes "full library".
- **The store** is `/srv/media-cache/music-art/` — a **sibling** tenant on the media-cache LV, deliberately not inside the FLAC cache's dir (its sweep deletes files it doesn't recognize). Content-addressed: `<blake2b128>.orig` (untouched bytes) + `.thumb.webp` (~160px) + `.large.webp` (~600px), generated at extraction — the serve path does zero image work. Lossy sizes are fine; the lossless rule covers audio, not pixels. All derived and rebuildable, no backup — **except owner-override images**, which are irreplaceable and live with the `music.db` backup story.
- **Provenance is structural** (`album_art` / `artist_art`, `source ∈ embedded|folder|derived|owner`): the upsert's `WHERE source != 'owner'` means no automated pass can ever clobber the listener's own pick — the rule lives in SQL, not caller discipline. The tables ride `SCHEMA` with no version bump (`CREATE TABLE IF NOT EXISTS` self-applies; the migration seam is for ALTERs only).
- **Artist images**: zero exist in the library, so v1 **promotes** the artist's most-rated album cover (summed thumbs, tie-break lowest album key — byte-stable across runs), recorded as `source='derived'`. The owner's own photo is a one-row `owner` update when the picker UI lands; external photo fetch is deferred to the Phase 1b bio-fetcher.
- **The album key is a cross-language contract**: `music_catalog.ALBUM_KEY_SQL` and `music/lib/db.ts` both derive `COALESCE'd artist + U+241F + COALESCE'd album` (with `artist_norm` leading the artist half since #152) — the exact string `albumIdOf` base64urls into URL ids. Paired fixture tests (`test_music_catalog.py` / `catalog-rows.test.ts`) pin identical fixtures on both sides.
- **Serving** (`/api/art/[hash]/[size]` in the music app): hash allowlisted (32 hex) before any path is built, plain file read from `MERLE_MUSIC_ART` (no default — unset means SVGs everywhere, the kill-switch convention), `Cache-Control: immutable, max-age=1y`. Content-addressed URLs change exactly when pixels change, so browsers cache every cover forever and a repeat page load makes zero art requests.
- **Rendering**: `art_hash` rides the existing row mappers onto `Track`/`Album`/`Artist` (optional field, so pre-art shapes — fixtures, the daemon's `/queue` payload — type-check unchanged), and `CoverArt` renders the `<img>` absolutely over the generated SVG, which remains the loading state (art fades in) and the fallback (missing art, 404s via `onError`). **The fade settles cached images via a mount-time ref** (`el.complete` check) — an immutable-cached image finishes before hydration attaches `onLoad`, and waiting on the event left loaded covers invisible on hard reload (issue #155). Zero layout shift by construction. `lib/db.ts` guards every art query on the tables existing, so a pre-art snapshot degrades to "no art, full library" — never to an empty catalog.
- **The album page anchors on its cover** (256px, 384px at sm+ — the `large` asset earning its size), and clicking it opens a hand-rolled **lightbox serving `/orig`** — the untouched original bytes' one surface. OutputPicker's backdrop idiom: click-away and Escape close; no dependency.

## Music playlist engine (epic #115 Phase 3 — issue #139)

`jukebox/music_playlist.py`: *"play me stuff like this"* as pure arithmetic over the analysis axes — **no LLM anywhere in this phase**, the epic's bar verbatim (Phase 4 makes it smarter, not possible). The engine is pure end to end: candidates, clock, and RNG are injected, no SQL inside scoring, no `time.time()` inside ordering. The daemon exposes it as one endpoint, **`POST /queue` `{seed, n, exclude}` → an ordered list of track shapes** — it generates lists only, starts no playback, holds no state; the daemon stays one-track-at-a-time on the transport verbs and the GUI owns the queue it fetches.

- **Two measured facts shape the scoring** (prototyped live against the real catalog, 2026-07-16, epic #115 comments). (1) **BPM is lattice-quantized at ~4–5% resolution** — librosa reports tempo on the lattice `60·22050/(512·lag)`, so exact BPM ties across genres are the *normal* case, tempo distance under one bin (~0.03 in log2) is clamped to zero, and the GUI must never render "112.3" as precise. (2) **Because tempo ties constantly, the tiebreak carries the ranking and must know genre *families*** — the first draft's exact-tag bonus put Queensrÿche at #2 for a Stereo MC's seed. Affinity works on the head token plus a cluster table, never string equality and never as a hard filter. Since #163 the daemon feeds the engine `genre_norm AS genre` (canonical tags, NULL neutral) and injects the families from `genre_rules.yaml`'s `clusters` section the way candidates and RNG are already injected — the module's `GENRE_CLUSTERS` raw-tag set survives only as the no-rules fallback.
- **The recipe, with provenance** (weights as live-tuned; retuning is a constant change plus a test update): octave-folded log2 tempo distance ×22 (half/double time is the same groove; 10% off ≈ 3.0) · |ΔDR| ×0.55 · |Δgain| ×0.30 · |Δyear|/18 capped at 30 · genre family −2.5 same head / −1.8 same cluster. Era and loudness deliberately double-count a little (#136 measured the loudness war by decade — known, accepted). **The regression baseline is the Stereo MC's "Lost In Music" benchmark** on the epic: early-90s electronic/hip-hop nearly wall to wall.
- **The seed is a flexible intent** — `resolve_seed(seed, candidates) -> (target, exclude)`. A track seed is that track's features (and excludes itself); an artist seed is the artist's **median** centroid (one 287-BPM outlier must not drag it) plus modal genre head; `{"target": {...}}` passes a prebuilt feature point through — the mood/weather seam Phase 5 fills (`narrator.py`'s roster-of-one move: the shape exists now so later seeds plug in without a rewrite). An unknown artist is an empty queue, never an error.
- **Ratings are rules here, evidence in Phase 4.** Strong-down (−2) is a hard filter **in the daemon's candidate `WHERE` clause**, never the scorer — the ban is enforced by the query, not entrusted to arithmetic, and a test proves absence over many seeded runs. Down/up/strong-up are fixed score adjustments (unmeasured starting values; the table had ~0 rows at ship). An empty ratings table is indistinguishable from "no opinions yet" — the quality floor is the analysis axes alone. Skip-weighting from `outcome` is named and deferred to Phase 4.
- **Anti-repetition**: the candidate query excludes anything in `play_history` within `COOLDOWN_HOURS` (24; the query derives its cutoff from the engine's constant so the two can't drift, and the pure rule `too_soon()` is the same policy under an injected clock), no artist twice within `ARTIST_SPACING` (6) consecutive slots, and the caller's `exclude` list keeps a refill from repeating what the GUI's queue already holds. A library too small for the spacing runs dry honestly — a short queue, never a repeat.
- **Determinism with variety** (`lib/queue.ts`'s injected-entropy precedent): each slot samples from the top-K (40) still-eligible candidates weighted `exp(−Δscore/1.5)`, ranking ties broken on track id. Same inputs + same RNG seed → byte-identical queue (tested); production passes a fresh `random.Random()`.
- **Radio in the GUI** (`PlayerProvider.startRadio`): one affordance — Radio from a track row or the artist header — seeds the engine, replaces the client queue, plays. Continuous play is a **refill loop, not a server queue**: at ≤3 upcoming tracks the provider fetches the next window of 25 with everything it holds excluded. Responses come back in the `TrackRow` wire shape and feed through the same tested `trackFromRow` mapper as every other surface; an empty or failed fill leaves the current queue playing out. Any explicit `playTracks` exits radio mode.

## Music app (epic #115, GUI track — issue #116)

The music player UI is a **second top-level Next app, `music/`, a peer of `mcc/`** — epic #115's D1, decided in #116. #110 reserved it a launchpad tile as an independent app, MCC's shell lives inside the 4,142-line `Dashboard.tsx`, and "Merle Control Center" doesn't stretch to cover albums; the rejected alternative (a `/music` route inside MCC) would have started with a shell-extraction refactor that has nothing to do with music. Same stack and conventions as MCC (Next 16 App Router, Tailwind 4, pnpm, Vitest on `lib/` pure helpers, components untested by policy), with tokens **copied, not abstracted** — a shared package is premature until two apps demonstrably need the same change twice.

- **The data seam is real now** (issue #129; the fixture era's design paid off exactly as intended). **`lib/api.ts` is server-only and async**: per-request `node:sqlite` reads of `music.db` (the `mcc/app/weather/history/route.ts` pattern whole — `MERLE_MUSIC_DB` absolute with **no default**, read-only, unset-or-missing degrades to an empty catalog, never a 500), shaped by pure mappers in **`lib/catalog-rows.ts`** (Vitest), ordered by the *same* tested `browse.ts`/`shelf.ts`/`search.ts` helpers the fixtures used — the album/artist index (~2k rows) loads whole and sorts in JS rather than porting `sortKey`'s leading-article rules into SQL; only the visible window hydrates tracks. **Artist/album ids are base64url of the names** (an artist *is* a name in this catalog — the *canonical* name since #152, `artist_norm`'s case-collapsed identity; the pages build raw `/album/${id}` hrefs and this library really contains "AC/DC"); track ids are the indexer's content hashes, shared verbatim with the daemon's `/stream/{id}`. **The m4a→alac/aac split believes the catalog's `codec` column** (#149's stsd atom walk, wired through the seam in #157 — the same provenance the daemon's `needs_flac` policy trusts); the bitrate heuristic (>500 kbps — ALAC runs ~700–1100, store AAC ≤320) survives only as the fallback for null-codec rows, where its worst case is a mislabeled badge, never a playback decision. The raw container rides along as `Track.container` for the album header's format pill (`lib/format-badge.ts`, tested: `ALAC · M4A` where codec and wrapper differ, a single token where the extension is the codec, majority label with stable tie-break at album level). Client components can't import `api.ts` (node:sqlite fails their build, loudly): search debounces into `/api/search`, the `[id]` pages split into server wrappers + client views, the seed queue arrives as a layout prop, and the player talks through **`/api/player/[verb]`** — an allowlisted verb proxy to `MERLE_MUSIC_DAEMON` (`state` on GET; `play`/`pause`/`stop`/`seek`/`rate`/`queue` on POST), because the daemon's address is server config and a second origin would drag CORS into a two-box LAN app. `lib/fixtures.ts` stays on disk feeding the pure-helper tests, nothing else.
- **`PlayerProvider` plays for real.** The queue stays client-side on purpose (one-track-at-a-time is the daemon's whole contract; since #139 Phase 3's engine *generates* sequences and this provider consumes them — radio mode's refill loop, see the playlist-engine section) but time and truth come from pearl: a 2s poll of `/api/player/state` supplies the position, and "the track ended" is *observed* — the daemon's watcher adjudicated the history row and cleared its slate — then the queue advances and posts the next play. An unreachable daemon freezes the bar at the last known position rather than inventing progress. Restart-this-track re-issues play (the Denon refuses Seek — see the daemon section); a scrub snaps back to the truth within a poll.
- **Surfaces**: persistent bottom player bar (now-playing cluster, transport + seek, queue toggle / output picker / volume / quality badge), queue slide-over (history · playing-from · next-up, with per-track remove — user-driven edits are in scope; *agent*-driven queue mutation stays the epic's non-goal), sound-output popover (Phase 2's three confirmed targets verbatim), everything-search in the chrome (grouped artists/albums/tracks overlay), artist pages (hero, read-more bio, top tracks, discography), album pages (#157/#159: recognizable own-cover backdrop — lightly blurred, double-scrimmed so the bottom-left text zone stays legible while the top stays scrim-free as the art's showcase, and **focal-anchored**: `object-position` follows the extraction pass's interest centroid so the band features the cover's subject, not dead center; TIDAL-scale title with the artist as the clear second line, text bottom-aligned *on purpose* — the backdrop owns the space above; quality + format pills sharing one border weight; and the retired heavy-blur wash demoted to a top-anchored fade behind the tracklist, quieter than the hero ever ran it, focal-anchored the same way).
- **The front door scales by curation, not enumeration** (issue #118). Home is genre pills + three bounded shelves — recently played (**real `play_history`** since #129; the shelf is simply absent until listening exists), **Today's dig** (the rediscovery shelf: a date-seeded deterministic shuffle over everything *not* recently played — anti-recency is a hard filter, and the seed is computed in a `force-dynamic` **server component** so the client never runs Date-in-render), and recently added (**by newest file mtime** since #129 — the honest "added" proxy until the catalog has ingest timestamps). The genre pills render the **canonical 22-tag vocabulary** since #163 (`lib/db.ts` reads `genre_norm`; the feral-taxonomy era — `ELECTRONICA\DUBSTEP`, `FLAC`-as-a-genre — ended when the normalization pass landed). Each shelf is **exactly two rows at every width** (`.shelf-grid`: 7×2 desktop down to 3×2 phone, extras `display:none`). The clamp is CSS, not JS, for two reasons: measuring width in JS would desync the server and client renders (hydration), and a reflowing shelf shifts the page under a thumb. A horizontal scrollbar was the first draft and hid half the shelf behind a gesture. First-paint card count is bounded (~50) regardless of catalog size.
- **The full catalog lives at `/albums` and `/artists`** — the browse pages TIDAL refuses to build. Both **load a window at a time as you scroll**: the page server-renders the first 60 and `components/Browser.tsx` appends each next window from **`/api/albums` / `/api/artists`**. The client holds what's been scrolled to, never the catalog — at 27k tracks, shipping the library to the browser to slice it there is the exact failure these pages exist to prevent. The feed contract is **offset+limit in, `{items, total, nextOffset}` out** — `LIMIT ? OFFSET ?` wearing a URL, so Phase 0 swaps the fixture call for a query against `music.db` inside the same handler (`mcc/app/weather/history/route.ts` is the precedent) and nothing upstream changes. Three load-bearing details: **`nextOffset === null` is the only end signal** (inferring "done" from a short window breaks when the last window is exactly full); **there is a real Load-more button, not just a scroll sentinel** (auto-load is invisible to keyboard users, so the button is the honest control and the observer the convenience — both call one re-entrancy-guarded loader); and a fetch failure **degrades** (the list keeps what it has, the button offers Retry) rather than throwing.
- **Filter/sort/letter stay URL state**, all `<Link>`s, re-rendered on the server — only the scroll-append is client-side. Changing a filter must start a new list, not append to the old one, which is what the `key` on `<Browser>` enforces. `sortKey` drops a leading "The " (record-store filing, tested). The rail renders only letters present, and a letter jump sets **where the window starts** (`?letter=S`), not a page — with infinite scroll there are no pages, and an offset in the URL would rot on re-index. It hides under newest-sort, where a letter jump is meaningless. **A jumped list that reaches the end says "end of the list", never "that's all of them"** — the entries before the jump were never loaded and the footer must not claim otherwise. Genre is one-per-album at the UI seam — the dominant `genre_norm` across the album's tracks (canonical since #163; a pre-normalization snapshot degrades to Uncategorized-heavy, never to raw tags leaking back — `hasGenreNorm()` guards the column like `hasFocal()` does); an **artist** filter means "has any album in this genre", because artists span genres in a real library. Pure mechanics live in `lib/shelf.ts` + `lib/browse.ts` under Vitest.
- **The palette is "same station, different room."** Base tokens are MCC's verbatim; the species accents and `--rain` are deliberately **absent** (they name detections and measurements; nothing here does either). `--led` keeps its one job — green means LIVE — worn by the now-playing row and its equalizer glyph. The one new accent is **`--hires` `#e0b25c`** (OKLCH ~L 0.78 C 0.11 h 82): it names a *quality the catalog indexed* (>16-bit lossless) the way `--rain` names a measurement, colors only the quality badge, and the badge always carries its label in text so no meaning rides on hue. Tiering (`lib/quality.ts`): hi-res gold > 16-bit lossless quiet > lossy faint; an album wears its best track's badge.
- **Four-level rating** (`lib/rating.ts`): split thumbs; click sets ±1, same-thumb again escalates to ±2, third click clears, opposite thumb replaces. The transition table is exhaustively tested because a strong-down becomes Phase 3's hard `WHERE` clause — the control mints contract data. **It persists since #135**, and the shape is worth knowing: **the catalog's answer rides on every `Track`** (a scalar subquery in `TRACK_COLS`, so all three track queries get it including `topTracks`' `GROUP BY`, where a joined column would have to be grouped), and `PlayerProvider`'s map holds **only this session's edits** — `ratingFor(track)` is `edits[id] ?? track.rating`. Hence it takes the *track*, not an id: an id alone can't answer for an album-page row that was never in the queue, which is what a queue-only map got wrong. The write is optimistic like `startTrack`, and **rolls back only if the click it fired for is still the one showing** — the control is built to be clicked in a burst (set → escalate → clear), so a slow failed POST must not stomp a newer click.
- **Generated cover art** (`lib/cover.ts` + `CoverArt`): fixtures have no artwork, and gray squares would let layout bugs hide — each album hashes its id into two hues + a pattern (rings/beams/blocks/waves), deterministic so an album looks the same everywhere including its own blurred backdrop. Real art replaces the component's internals, not its callers.
- **No-layout-shift discipline** (the house #1 rule): the idle player bar renders the playing state's exact skeleton; toggle-button active dots and output-picker checks are present-but-`invisible` when off; rating controls are always mounted and only fade on hover; search results are an absolutely-positioned overlay.
- **Dev**: **port 3001, pinned in the `dev`/`start` scripts themselves** (`next dev -p 3001`) rather than only at the launcher — a bare `pnpm dev` in `music/` would otherwise default to 3000 and collide with MCC. The app owns its port however it's started. Dev env (`music/.env.local`, not in git): `MERLE_MUSIC_DB` pointing at a **backup-API snapshot** of pearl's catalog (never `cp` a live WAL file), `MERLE_MUSIC_DAEMON=http://192.168.1.64:8090` — browse reads the snapshot, play drives the real daemon, and history rows land on pearl's copy, so a stale dev snapshot only means a stale recently-played shelf. **Ratings make that split sharper (#135), and it's worth knowing before it fools you**: a thumb clicked in dev writes to *pearl's* catalog and is read back from the *snapshot*, so it renders correctly until you reload and then appears to vanish — which looks exactly like the bug #135 fixed. The data is on pearl; the snapshot is stale. Re-snapshot to see it, and don't debug the reload in dev without checking pearl's table first. CI: the `web-music` job in `tests.yml` mirrors `web` with `working-directory: music`. **Production is `music-app` on pearl:3001** (issue #131 — the `mcc-dashboard` pattern whole: `next start`, fast-stop drop-in, live `music.db` by absolute path, daemon proxied over loopback; `Servers/deploy-music.sh` by hand, `merle-autodeploy` on merge via `MERLE_DEPLOY_MUSIC=1`, and `music-daemon` rides `MERLE_DEPLOY_UNITS` so jukebox merges restart playback too). Still deferred from #110: the Caddy route/hostname and the launchpad tile. Deferred with names, not dropped: Explore/pregenerated playlists, similar artists, mini-player, playlist management, queue reorder/clear, a volume control that does anything.

## The front door (epic #110 Phase 1 — issue #141)

Everything on pearl used to be reached by memorized `IP:port` pairs. Since
#141 there is **one front door: Caddy on pearl:80**, and the house speaks
names — `pearl/mole` (Pi-hole), `mcc.lan` (:3000), `music.lan` (:3001).
The names are Pi-hole Local DNS records, and they use the **`.lan` suffix**
because that's the search domain pearl's DHCP already hands out — so a
desktop can type `mcc/` and the resolver fills in the rest. (The epic
proposed `.home`; `.lan` won because it was already deployed to every
client's resolver config, not chosen.) The Caddyfile lists each site under
both spellings (`http://mcc, http://mcc.lan`) because the browser's Host
header carries what was typed, not what DNS resolved.

Pi-hole's own web server (v6: embedded in `pihole-FTL`) moved off 80/443 to
**loopback:8081** to release the ports — Caddy proxies `/mole` *and* `/api`
(the v6 admin is a shell over its API; proxy one without the other and you
get a login page that can't log in). The `/mole` path is not a Caddy
rewrite: the UI's own home moved there (`webserver.paths.webhome =
"/mole/"`, renamed from the stock `/admin/` in #143 to match the Mole
tile), so the app generates `/mole/...` links itself and the proxy stays
dumb — plus the `/var/www/html/mole → admin` symlink, because FTL serves UI
files from `webroot + webhome` (see Pearl.md § Pi-hole for the trap).
`/api` ignores webhome and stays put. DNS (:53) and DHCP (:67) were untouched.
Plain HTTP, `auto_https off`, nothing on 443: TLS/auth is the epic's
deferred single-choke-point payoff, not a current feature. The broker's
WebSocket (:9001) is deliberately *not* proxied — browsers speak MQTT to it
directly (`NEXT_PUBLIC_MERLE_MQTT_WS` is absolute). **Phase 4 decided this
stays so** (#147): the launchpad's lamps dial `ws://192.168.1.64:9001`
straight (the `bus` key in `tiles.json`), keeping one idiom with the MCC; a
`ws://pearl/bus` Caddy proxy is deferred until something actually needs it —
TLS would be that something.

Canonical Caddyfile: `Servers/Caddyfile` in the repo; live copy
`/etc/caddy/Caddyfile`, synced **manually on purpose** (a reverse-proxy
config that deploys itself on merge could take every web surface down at
once — see Pearl.md § The front door for the two-line sync).

## Homestead — the launchpad (epic #110 Phase 3 — issue #143)

The front door's face: `pearl/` 302s to `/home/`, and that redirect is the
one bookmark the whole system needs — every module is a tile from there.
**Homestead** is the app's settled name (it was the epic's working title;
"Trailhead" was the working name it replaces).

`launchpad/` is a top-level peer of `mcc/` and `music/` and is deliberately
**not a Next app**: four hand-rolled files (`index.html`, `styles.css`,
`app.js`, `tiles.json`) plus two self-hosted font subsets, served by Caddy's
`file_server` straight from pearl's checkout. No framework, no build — which
means **a merge deploys it by `git pull` alone**; `merle-autodeploy` needs
no gate for it and its journal shows nothing, and that's correct, not
broken. **Adding a module is one `tiles.json` entry, not a code change**
(`{name, icon, color, url, description, status: live|soon}`; file order is
porch order; an unknown/omitted icon key falls back to the signpost glyph,
so config-alone stays true even for a tile with no bespoke icon; `soon`
tiles render dimmed and inert). The page fetches `tiles.json` with
`cache: no-store` so edit-and-refresh never serves a stale porch.

Design language is the MCC's "Ranger Station, Night Watch" one room over —
same tokens/topo background/type pairing, **copied, not abstracted** (the
music app's precedent). The tiles are trail placards, not a Card-Card-Card
grid: each wears a painted **blaze** in the module's own color (hue carries
identity, the narrators' voice-color idiom — Merle squirrel-orange, Weather
rain-blue, Music hi-res gold, Mole earth-brown, Hearth ember, and the
coming-soon pair dimmed to a whisper). Fonts are the same Fraunces/Sometype
Mono subsets next/font self-hosts for the MCC, with its fallback metrics
copied so the swap can't shift layout; the grid reserves two placard rows
before `tiles.json` lands (house rule #1).

**The lamps (epic #110 Phase 4 — issue #147).** Tiles whose config carries a
`presence: "<topic>"` wear a live status dot fed by the bus's retained
`"online"`/`"offline"` strings — the house Last-Will idiom generalized, not
health checks invented: a publisher dying (crash, Ctrl+C, SIGTERM) flips its
lamp within seconds with no polling and no cleanup code, verified live by
hard-killing a publisher. Three states, one class swap on a fixed 8px slot
every tile reserves (no layout shift): online pulses `--led`, offline is a
static faint dot, and *unknown* — no verdict yet, or the bus itself
unreachable — is fainter still, because a dead broker is not a verdict about
the services. The broker URL is the top-level `bus` key in `tiles.json`
(explicit, per the epic's note; the same-host fallback idea only works
because broker and Caddy share pearl, and explicit survives that changing).
The MQTT client is a **vendored** `mqtt.esm.js` (5.15.1, `launchpad/vendor/`
— the *browser* build, honoring the MCC's hard-won default-entry gotcha;
vendoring is not a build step, the no-build rule holds). Current wiring:
Merle → `services/merle-daemon/status` (the new house-wide namespace from
#147 — `bus.py` grew `service_status_topic()`, and the perception daemon
passes it as `status_topic`, so the lamp answers "is the driveway watch on
right now?"; dark is its *normal* state), Music → `music/status` (the
engine, not the GUI — pearl's always-on Next apps are boring, the engine is
the news), Weather Post → `weather/status`. Mole/Hearth/coming-soon carry no
topic and keep their slot invisible.

## Repo layout

**The Python is grouped into role packages** (issue #123). It was ~25 flat root scripts until then, and the guide used to defend that as *"deliberate for a single-machine project"* — a rationale that expired two machines ago. The repo is now three boxes plus two web apps, with an import boundary that must hold, so the shape follows the roles:

| Package | Holds | Runs on |
| --- | --- | --- |
| `vision/` | `perception.py` (the shared tracker/annotation brain), `merle_daemon.py` (FastAPI app), `frames.py` (frame sources), `storage.py` (SQLite), `label_utils.py` | bluejay |
| `narration/` | `narrator.py`, `personas/`, `character_bible.yaml` | pearl (Marlin), merle (Jim) |
| `weatherpost/` | `weather.py` (Willard), `weather_archive.py` (the seasonal archive) | pearl |
| `jukebox/` | `music_catalog.py` (the store), `music_index.py` (the read-only NAS indexer), `music_daemon.py` (playback: /stream + DLNA control + /queue, port 8090), `music_playlist.py` (the Phase 3 engine, pure), `music_analyze.py` (the offline analyzer), `music_import.py` (its importer) | pearl (analyzer: bluejay) |
| `tools/` | `replay_events.py`, `replay_fixture.py`, `record_fixture.py`, `extract_frames.py`, `dedup.py`, `prelabel.py`, `make_2class.py`, `live.py` | bluejay, by hand |
| *root* | `bus.py` (topics + publisher, the contract every bus process shares), `frame_archiver.py` (the still-shot archive on pearl) | everywhere |

**Packages are named for their role, not their contents, and that's load-bearing**: a package `X/` containing `X.py` breaks `import X` — the name resolves to the package, not the module — which is why it's `vision/` and not `perception/`, and `weatherpost/` and not `weather/`. `jukebox/` exists because `music/` is the Next app. Every file kept its own name through the move, so `git log --follow` still works. **All `__init__.py` files are empty on purpose** — a convenience re-export is exactly how merle's minimal venv dies (see below), and `test_import_boundary.py` fails if one appears.

`bus.py` stays at root because every box imports it and it belongs to none of them; `frame_archiver.py` stays because it's pearl-resident but needs no vision deps. The broker itself lives on pearl, and the repo-root `mosquitto.conf` is just a pointer to its config there.

**Tests stay at root** and import across (`from vision import perception`). `requirements.txt` pins the daemon/bus deps (plus `mutagen`, optional by design — the indexer's span parsing and hashing are pure stdlib, so a box without it still builds a correctly-identified catalog, just with null tags). **CI's Python job runs an explicit file list** (`.github/workflows/tests.yml`) — there is no `pytest.ini`/`testpaths` fallback, so a new `test_*.py` that isn't added to that line is silently never run. `test_bus.py` and `test_frame_archiver.py` proved that the hard way: they existed for months and never once ran, until #123 added them.

**`test_import_boundary.py` guards merle's venv.** The Pi runs `narrator-jim` and nothing else, on a deliberate two-package venv (`paho-mqtt` + `pyyaml`, never `requirements.txt` — the vision stack has no business on a Pi 5). Nothing enforced that; it held by luck. The test imports the narrator in a subprocess with `cv2`/`torch`/`ultralytics`/`fastapi`/`uvicorn`/`numpy` poisoned to raise, so a stray import fails in CI instead of taking merle down 60 seconds after a merge. It also asserts the poison bites and that the daemon *does* need the vision stack — otherwise the whole file could pass by doing nothing.
- `models/`: deployed-weights shelf — `current.pt` (what the app loads) plus versioned `merle-trainNN.pt` copies. Only its README is tracked; the `.pt` files are gitignored. See `models/README.md`.
- `mcc/`: Next.js 16 App Router, TypeScript, Tailwind 4, pnpm. Tests: Vitest (`pnpm test`), CI runs them on every PR (`.github/workflows/tests.yml`).
- `music/`: the music player UI (epic #115 / issues #116, #129) — same stack as `mcc/`, own lockfile and CI job (`web-music`), reading the real catalog and driving the playback daemon. See the "Music app" section above.
- `launchpad/`: Homestead, the house launchpad (issue #143) — deliberately static (no framework, no build, no lockfile, no CI job), served by Caddy from pearl's checkout. See the "Homestead" section above.
- Not in git: datasets (`training/`), weights (`*.pt`, including `models/`), captures (`hard_frames/`, `snapshots/`, `debug_frames/`, `frames/` — the still-shot archive on pearl), SQLite stores (`*.db` + WAL companions — `merle.db` on bluejay, `weather.db` and `music.db` on pearl), `.venv/`. Gitignored means *state*, not *disposable*: `weather.db` cannot be regenerated (see the weather post above), and neither can `music.db`'s `ratings` / `play_history` — the rest of that file rebuilds from the NAS in ~3 hours, those two tables don't rebuild at all.
